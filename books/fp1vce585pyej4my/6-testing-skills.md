---
title: "スキルのテスト——発火しない問題を防ぐ"
---

スキルを作ったら、3つの観点でテストする。

## 1. トリガーテスト

最も重要なテスト。スキルが正しいタイミングで発火するか。

### テスト方法

Claude Codeに以下の種類の入力を試す:

**発火すべき入力:**
```
「デプロイして」
「deploy to production」
「本番にリリースしたい」
```

**発火すべきでない入力:**
```
「デプロイの概念を教えて」
「deploymentとは何ですか」
「テストを書いて」
```

### よくある失敗

1. **triggerが狭すぎる**: 「deploy」だけ書いて「デプロイ」で発火しない
2. **triggerが広すぎる**: 「code」と書いて、あらゆるコーディング作業で発火する
3. **パラフレーズで発火しない**: 「リリース」「出荷」「公開」でも発火すべき

### 改善

```yaml
trigger: When user mentions deploying, releasing, shipping, publishing to production, or says "デプロイ", "リリース", "公開", "出荷"
```

具体的な動詞を複数書く。日本語と英語の両方を入れる。

## 2. 機能テスト

スキルが正しい出力を出すか。

### テスト方法

1. スキルを発火させる
2. 実行された手順を確認する
3. 期待通りの結果が得られたか検証する

### チェックポイント

- 手順が正しい順序で実行されたか
- references/のファイルが適切に参照されたか
- エラー時の挙動が想定通りか

## 3. パフォーマンス比較

スキルあり/なしでの効率差を計測する。

### 計測項目

同じタスクをスキルあり/なしで実行し、比較する:

| | スキルなし | スキルあり | 差 |
|---|---|---|---|
| ツール呼び出し回数 | 12回 | 7回 | -42% |
| 完了までの時間 | 3分 | 1.5分 | -50% |
| 手戻り回数 | 2回 | 0回 | -100% |

上は記事投稿スキル（post-article）の実測値。スキルに手順が書いてあるため、Claudeが試行錯誤する回数が減る。

### 計測方法

1. `/cost` コマンドでセッション開始時のトークン消費を記録
2. タスクを実行
3. 再度 `/cost` で消費量を確認
4. ツール呼び出し回数はセッションログ（`~/.claude/projects/*/session-logs/`）から確認

```bash
# セッションログからツール呼び出し回数を数える
grep -c '"tool_use"' ~/.claude/projects/*/session-logs/*.jsonl | tail -1
```

完璧な計測は不要。「スキルあり/なしで体感が変わったか」が一番の指標。

## テスト結果の記録

テスト結果はSKILL.mdの先頭コメントに記録しておくと、後で参照できる。

```yaml
---
name: deploy-helper
# Last tested: 2026-03-28
# Trigger: 5/5 patterns fired correctly
# Performance: 3 tool calls saved vs no skill
---
```
