# はじめに——160+時間で何が起きたか

2026年1〜2月、私はClaude Codeを「ほぼ無人」で160+時間稼働させた。

ゲーム開発、記事作成、ツール開発、マーケティング——様々なタスクをAIに委ねて、自分はほぼ1日1回のチェックだけ。

結果は「想定より50倍多くの失敗」だった。

## 実際に起きたこと

**削除事故（2回）**
`rm -rf` でゲームプロジェクトのファイルが消えた。1回目はバックアップで復元。2回目は2時間の作業が消えた。

**コスト暴走（1回）**
APIを叩くサブエージェントがループして、1時間で$8消費した。気づいたのは翌朝。

**無限ループ（複数回）**
同じエラーを繰り返すCC。「修正→確認→同じエラー」を20回繰り返した。

**本番pushミス（1回）**
レビューなしでmain pushした。気づいたのは翌日、別のセッションで。

**コンテキスト消失（頻繁）**
context windowが満杯になって、前の作業を忘れたCCが同じタスクを最初から始めた。

## 全部「設定の問題」だった

事後に振り返ると、全ての失敗は共通の原因を持っていた：

**「普通のClaude Code設定」は、自律稼働のために作られていない。**

`rm -rf`をブロックするhookがなかった。
エラーを検知して止まる仕組みがなかった。
コンテキストが消えても作業を引き継げる状態管理がなかった。

これらを1つずつ追加していったのが、この本の内容だ。

## cc-health-checkとこの本の関係

`cc-health-check`（[https://yurukusa.github.io/cc-health-check/](https://yurukusa.github.io/cc-health-check/)）は、上記の失敗パターンを20個のチェック項目に変換したツールだ。

```bash
npx cc-health-check
```

このコマンドが出すスコアの意味と、スコアを上げる具体的な方法——それがこの本だ。

## この本の対象読者

- Claude Codeを「長時間・無人で動かしたい」人
- 環境診断スコアが60点以下で「何から直せばいい？」と思っている人
- AIエージェントの安全な運用に興味がある人

## この本で得られるもの

- **20チェック項目の解説**——なぜそのチェックが必要か、何が起きるとFAILするか
- **16個のhookスクリプト**——コピーしてすぐ使える
- **5個のテンプレート**——CLAUDE.md、task-queue、mission.md等
- **160+時間の実体験**——何が失敗して、どう直したか
