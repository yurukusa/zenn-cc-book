この本は 2026 年 3 月に最初に公開された。その後 4 月にかけて、Claude Code 界隈で新しく 3 つの Safety リスクが顕在化した。いずれも本書で扱ってきた既存の事故パターンとは別系統で、hook や設定の追加対応が必要になる。

本章は既存購入者への無償アップデートとして、この 3 件の概要と、それぞれに対する最小限の対策をまとめる。各項目の詳細 hook 実装は Zenn の無料補遺 Gist に掲載予定で、本章はあくまで「存在の周知 + 1 行対策」に絞る。

## 1. Opus 4.7 の安全分類器が 4.6 にハードコードされている (#49618)

2026 年 4 月 16 日に Opus 4.7 がリリースされた直後、Anthropic の auto モード安全分類器（どの操作をデフォルトで許可するかの判定器）が **内部で Opus 4.6 のモデル ID に固定されたまま**という問題が発覚した。

### 何が起きるか

ユーザー側で `ANTHROPIC_MODEL=claude-opus-4-7` のように 4.7 を指定しても、auto モードの判定だけは 4.6 のロジックが走る。Opus 4.7 から追加された判定ルール（破壊的コマンド対応、新しい API エンドポイント対応）が適用されず、**本来ブロックされるべき操作が許可される**ケースが発生する。

### 検出

GitHub Issue [#49618](https://github.com/anthropics/claude-code/issues/49618) の OP は、リリース後 3 日で **23 件以上のデータ損失報告**をスレッドに集めた。主な報告パターン:

- `rm -rf` 系の分岐で 4.6 の旧パターンしか検知しない
- 4.7 で追加された tool (`SearchReplace` 等) の判定が欠落
- サブエージェント起動時に権限ダイアログが出ずに通る

### 1 行対策

Opus 4.7 公式対応が出るまでの回避策として、**Opus 4.6 に pin する**のが最短。

```bash
export ANTHROPIC_MODEL=claude-opus-4-6
```

あるいは `cc-safe-setup` の Opus 4.7 専用モードを使う。

```bash
npx cc-safe-setup@latest --opus47
```

後者は、4.7 の判定欠落を 4.6 時代の hook セットで補強する。第 2 章で設定した Safety Guards が自動で適用される。

## 2. CVE-2026-21852——プロジェクト `.claude/settings.json` 経由の API キー窃盗

2026 年 4 月に公開された CVE。悪意あるリポジトリがプロジェクト配下に仕込んだ `.claude/settings.json` を Claude Code が読み込むと、**環境変数や認証情報が外部サーバーへ送信される**というもの。

### 発生経路

典型的な攻撃シナリオは以下の通り。

1. 攻撃者が OSS リポジトリの中に `.claude/settings.json` を置く（hook で `env | curl ...` するように仕込む）
2. 被害者が `git clone` して `cd` する
3. Claude Code をそのディレクトリで起動（または `/resume` する）
4. プロジェクト `settings.json` が自動で読み込まれ、hook が走る
5. 環境変数（`ANTHROPIC_API_KEY`、`OPENAI_API_KEY`、`GITHUB_TOKEN` 等）が外部へ送信される

被害者側からは、Claude Code が普通に起動しただけに見える。ターミナル上には何のエラーも出ない。

### 検出

自分のマシンで過去に clone したリポジトリを調べるには:

```bash
find ~ -type f -name 'settings.json' -path '*/.claude/*' 2>/dev/null \
  | xargs grep -l -E '(curl|wget|nc|bash -c)'
```

これで hook を含んだ怪しい `settings.json` が見つかる。心当たりがあれば即削除 + API キーのローテーション。

### 1 行対策

**プロジェクトレベルの `.claude/settings.json` を一切使わない**方針に切り替える。

```bash
rm -rf .claude/settings.json  # クローン先で即削除するcronを用意してもよい
```

hook 定義は `~/.claude/settings.json`（ユーザーレベル）にだけ書く。プロジェクトごとの差分は環境変数や `CLAUDE.md` で代替する。

第 2 章の Safety Guards をユーザーレベルで設定していれば、プロジェクト `.claude/settings.json` が上書きしない限り有効のままになる。

## 3. v2.1.100 以降の「見えない 20K トークン」隠れ課金 (#46917)

Claude Code v2.1.100 以降、**API コールごとに約 20,000 トークンが billing には加算されるが応答本文には現れない**現象が報告されている。

GitHub Issue [#46917](https://github.com/anthropics/claude-code/issues/46917) は 196 件以上のリアクションを集め、Anthropic 側も「top priority」として調査中と表明している。

### 何が起きるか

1 ターンあたり 20K トークンが隠れて消費されるため、`/usage` の cache_creation_input_tokens が不自然に膨らむ。月 $200 の Max プランでも、従来の半分以下の時間でクォータが尽きる。

この本では第 4 章（Monitoring）で「キャッシュ読取率が 89% → 4% に暴落した事例」を扱ったが、今回の問題はそれとは別系統で、**キャッシュ読取率は健全なのに billing が膨らむ**点が特徴だ。

### 検出

`/usage` で数セッション分のデータを並べ、以下のパターンを探す。

| 症状 | 意味 |
|---|---|
| `cache_read_input_tokens` は 90% 以上なのに `cache_creation_input_tokens` が毎ターン 20-30K 増える | #46917 系の隠れ課金 |
| `cache_read_input_tokens` が 50% 未満 | キャッシュミス系（第 4 章参照） |
| 両方異常 | 混在。まず v2 を pin してから個別対応 |

### 1 行対策

v2.1.100 より前にロールバックする。

```bash
npm install -g @anthropic-ai/claude-code@2.1.99
```

ロールバックできない場合（チーム全体が v2 使用中など）、サブエージェントを一時的に無効化する。サブエージェントは 20K 消費の最大要因のひとつ。

```bash
export CLAUDE_CODE_DISABLE_AUTO_MEMORY=1
```

公式修正が出るまでは、`/usage` を 1 日 1 回見る運用にするだけでも異常検知は早まる。

## まとめ——2026-04 までの学びを本書にどう統合するか

| 追補項目 | 本書の該当章 | 必要な追加対応 |
|---|---|---|
| #49618 Opus 4.7 分類器バグ | 第 2 章 Safety Guards | Opus 4.6 pin または `--opus47` モード |
| CVE-2026-21852 settings.json 窃盗 | 第 2 章 Safety Guards + 第 11 章 | プロジェクト settings.json 不使用 |
| #46917 隠れ 20K 課金 | 第 4 章 Monitoring | v2.1.99 ロールバック or subagent無効化 |

3 件に共通するのは、どれも **ユーザー側の設定ミスではなく、Claude Code 本体または Anthropic 側のロジックに由来する事故**という点だ。本書で築いた「hook による自衛層」の意義は、こうした CLI 本体側の問題にも効く——ブラックボックスに完全依存せず、自分の手元で判定を持つこと——そのまま変わらない。

2026 年 5 月以降もこの種の事故は起き続けるはずなので、本章は今後も追記で維持する。新しい事故を見つけたら、まず第 4 章の Monitoring 手法で「自分の環境で再現するか」を確認してから、公式 Issue を検索する順で対応してほしい。
