---
title: "第2章 CLAUDE.mdの書き方でトークン消費が変わる"
---

CLAUDE.mdはClaude Codeへの指示ファイルだ。ここに書いた内容は、セッション中の**すべてのターンで**コンテキストに含まれる。つまり、CLAUDE.mdが1行増えるたびに、セッション全体のトークン消費が増える。

しかし「短く書けばいい」という単純な話ではない。短すぎる指示は曖昧になり、Claude Codeが意図を汲めずにリトライを繰り返す。そのリトライのほうがよほどトークンを食う。

大事なのは「短い」ではなく「効率的」。同じ意図を、より少ないトークンで、より正確に伝える書き方がある。

## 原則: CLAUDE.mdは100行が目安

800時間の運用から得た経験則として、CLAUDE.mdは100行以内が効率的だ。Anthropic公式でも「簡潔に」と推奨されているが、具体的な行数は示されていない。以下は運用実績に基づく分類だ。

- 50行以下: 基本的な指示しか入らない。自律運用には不足することが多い
- 50-100行: 主要なルールと判断基準が収まる。最もバランスが良い
- 100-200行: 細かいルールまでカバーできるが、固定トークンコストが増え始める
- 200行以上: 毎ターンのオーバーヘッドが体感できるレベル。分割を検討すべき

自分のCLAUDE.mdは最終的に141行に収まった。ここに至るまでに500行以上あった時期もある。

## パターン1: 禁止リストより許可リスト

### 非効率な書き方

```markdown
## 禁止事項
- rm -rfを使うな
- git reset --hardを使うな
- git push --forceを使うな
- .envファイルを削除するな
- node_modulesを削除するな
- /を含むパスでrmを使うな
- sudoを使うな
```

7行で7つの禁止事項。しかし「禁止されていない操作は許可」と解釈される余地がある。`find . -delete`は禁止されていないから使える、となる。禁止リストは常に不完全だ。

### 効率的な書き方

```markdown
## 許可されている操作
- ファイル読み取り: 常に可
- git commit: 可（mainへの直接pushは不可）
- npm install: 可（--save-devのみ）
- ファイル削除: 自分が作成したファイルのみ

上記にない破壊的操作は実行前に確認。
```

6行で同じ範囲をカバーしつつ、リストにない操作に対して慎重になる。ホワイトリスト方式はセキュリティの基本原則と同じだ。

**トークンへの影響**: 行数は同程度だが、リトライの削減効果が大きい。禁止リスト方式では「これは禁止されてるか？」の判断が毎回発生し、判断を間違えると「hookにブロックされる→別の方法を試す→またブロック」のループになる。許可リスト方式では「リストにないからやめよう」で1回で済む。

## パターン2: 具体例を1つ添える（few-shot）

### 非効率な書き方

```markdown
コミットメッセージは分かりやすく書くこと。
```

Claude Codeがこの指示から生成するコミットメッセージ: 「Update files」「Fix bug」「Add feature」。分かりやすいかどうかの基準がないからだ。

### 効率的な書き方

```markdown
コミットメッセージ: 「なぜ」を含める。
✓ fix: prevent duplicate API calls on rapid button clicks
✗ fix bug
```

3行追加しただけで、品質が劇的に変わる。LLMは具体例からパターンを抽出するのが得意だ（few-shot prompting）。1つの良い例と1つの悪い例があれば、その間の判断基準を自分で補完する。

**トークンへの影響**: 例を2行足すコスト（毎ターン+数十トークン）に対し、「コミットメッセージが不適切→書き直し指示→再生成」のリトライ（数百トークン）がなくなる。投資対効果が高い。

## パターン3: 理由を1行だけ添える

### 非効率な書き方

```markdown
テストファイルは `__tests__/` に配置する。
```

理由がないルールは、エッジケースで無視されやすい。「src/の隣に置いた方が分かりやすいな」とClaude Codeが判断して、ルールを破ることがある。

### 効率的な書き方

```markdown
テストファイルは `__tests__/` に配置する。
  理由: src/と同階層に置くとビルド出力に混入し、バンドルサイズが増加する
```

1行の理由を足すだけで、Claude Codeは「なぜダメなのか」を理解する。類似のケース（例: 設定ファイルの配置）にも自発的にルールを適用するようになる。

**注意**: 理由は1-2行で十分。3段落以上の説明は逆効果だ。運用データでは、長い理由説明がトークン消費を体感で1割前後増加させていた。1行の理由ならほぼ誤差の範囲。

## パターン4: テーブル形式で判断基準を書く

### 非効率な書き方

```markdown
技術選択で迷ったら、最もポピュラーなオプションを選んでください。
要件が不明確な場合は、最もシンプルな解釈で実装してください。
パフォーマンスと可読性が衝突したら、通常は可読性を優先してください。
ただし、プロファイリングでボトルネックが確認された場合はパフォーマンスを優先してください。
セキュリティに関わる判断は、常に安全な方を選んでください。
```

5行の散文。Claude Codeは毎回この文を全部読んで、該当する箇所を探す必要がある。

### 効率的な書き方

```markdown
| 状況 | 判断 |
|------|------|
| 技術選択 | 最もポピュラーなもの |
| 不明確な要件 | 最もシンプルな解釈 |
| パフォーマンスvs可読性 | 可読性優先（プロファイリング済みなら例外） |
| セキュリティ | 常に安全な方 |
```

同じ情報量で、構造が明確。LLMはテーブル形式の条件分岐を高精度で処理する。散文よりもパースが速く、誤読が少ない。

**トークンへの影響**: テーブル形式は散文より少ないトークンで同じ情報量を表現できる傾向がある（構造化によりClaude側の解釈コストも下がる）。加えて、判断ミスによるリトライが減る。

## パターン5: hookに任せるべきルールをCLAUDE.mdから外す

### 非効率な書き方

```markdown
## 安全ルール
- rm -rfは絶対に使わない。代わりにファイルを個別に削除する
- git reset --hardは使わない。間違えた場合はgit reflogで復旧する
- .envファイルは git add しない。.gitignoreに入っているか確認する
- mainブランチに直接pushしない。feature branchを作る
- sudoは使わない
- chmod 777は使わない
```

これらはCLAUDE.mdに書いても「お願い」に過ぎない。Claude Codeがルールを忘れる可能性はゼロではない。

### 効率的な方法

hookで強制し、CLAUDE.mdには書かない:

```markdown
## 安全ルール
hookが破壊的コマンド・branch保護・secret検知を自動ブロックする。
hookがブロックした操作はリトライしないこと。
```

2行で済む。hookが技術的に強制するルールをCLAUDE.mdにも書くのは二重管理だ。トークンの無駄であり、管理の手間でもある。

**トークンへの影響**: 上の例では6行→2行で4行分の固定トークンコスト削減。加えて、hookがブロックするのでリトライのサイクルが「試行→ブロック→諦める」の1回で済む。CLAUDE.mdだけの場合は「試行→成功してしまう→事故→ロールバック」で数倍のトークンを浪費する。

## まとめ: 5パターンの早見表

| パターン | 核心 | 節約メカニズム |
|---------|------|---------------|
| 許可リスト | 「やっていいこと」を書く | リトライ削減 |
| 具体例 | 良い例＋悪い例を1つずつ | 品質向上→やり直し減少 |
| 理由1行 | 「なぜ」を15-30語で | エッジケースの自律判断 |
| テーブル形式 | 条件→判断の表 | パース効率＋判断精度 |
| hook分離 | 強制ルールはhookへ | 二重管理の排除 |
