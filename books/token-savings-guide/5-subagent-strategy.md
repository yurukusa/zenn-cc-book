---
title: "第5章 サブエージェント戦略——メインのコンテキストを守る"
---

サブエージェントは、Claude Codeの中で別のClaude Codeを起動する機能だ。Agentツールで呼び出され、独自のコンテキストウィンドウで動く。

メインのコンテキストを汚さずに並列処理できる。ただし、使い方を間違えるとトークン消費が爆発する。

## サブエージェントの仕組み

メインのClaude Codeが`Agent`ツールを呼び出すと:

1. 新しいClaude Codeプロセスが起動する
2. プロンプト（指示）がそのプロセスに渡される
3. サブエージェントが独自に作業する（ファイル読み込み、検索、コマンド実行）
4. 結果がメインに返される

重要なのは、**サブエージェントの作業内容はメインのコンテキストに入らない**こと。メインには「サブエージェントを起動した」「結果はこうだった」だけが記録される。

つまり、サブエージェントが50ファイル読んでも、メインのコンテキストは50ファイル分膨張しない。結果の要約だけが入る。

## いつサブエージェントを使うべきか

### 使うべき場面

1. **広範な検索**: 「このエラーメッセージがどのファイルで発生するか調べて」
2. **並列調査**: 「AとBとCの3つのアプローチを調べて比較して」
3. **大量のファイル処理**: 「全テストファイルで非推奨APIの使用箇所を探して」
4. **独立したタスク**: メインの作業と関係ない確認作業

### 使うべきでない場面

1. **1ファイルの検索**: `Grep`や`Glob`で直接やった方が速くて安い
2. **単純なコマンド実行**: `Bash`で直接実行すべき
3. **メインの文脈が必要な作業**: サブエージェントはメインの会話履歴を知らない

## サブエージェントのトークンコスト

サブエージェントのコストを見積もる:

```
サブエージェントのコスト = システムプロンプト + プロンプト（指示）+ 作業中のツール呼び出し + 結果の生成
```

最低でも数千トークンはかかる。「このファイルの10行目を読んで」程度の作業にサブエージェントを使うと、直接`Read`するより10倍以上のトークンを消費する。

### コスト比較

| 操作 | 直接実行 | サブエージェント経由 |
|------|---------|-------------------|
| 1ファイルの検索 | ~100トークン | ~3,000トークン |
| 10ファイルの調査 | ~1,000トークン（コンテキストに残る） | ~5,000トークン（コンテキストに残らない） |
| 50ファイルの調査 | ~5,000トークン（コンテキストが膨張） | ~8,000トークン（コンテキストは清潔） |

※ 上記は800時間の運用中に観察したおおよその値。実際のトークン消費はファイルサイズ、プロンプトの長さ、モデルの応答量によって変動する。

### 隠れたspawnコスト: cache_controlの不在

2026年4月に判明した問題として、サブエージェントのシステムコンテキストには`cache_control`ヘッダーが設定されていない（[#50213](https://github.com/anthropics/claude-code/issues/50213)）。つまり、**毎回のspawnで約4,700トークンがcache_creation（通常の1.25倍コスト）として課金される**。メインエージェントではキャッシュが効いて再利用されるコンテキストが、サブエージェントでは毎回フルコストで再計算される。

並列10エージェントのワークフローだと、spawn時点で約59,000トークン（約5,900トークン×10）が「作業を始める前に」消えている計算だ。

この問題を踏まえると、上のコスト比較テーブルの「サブエージェント経由」の数字はさらに+4,700トークン/spawnが上乗せされる。分岐点が10-20ファイルではなく、やや上にずれることを意識しておくべきだ。

**分岐点は10-20ファイル**。それ以下なら直接やる方が安い。それ以上ならサブエージェントの方が、メインのコンテキストを守れる分だけ得になる。

## 5つの戦略

### 戦略1: 1サブエージェント = 1タスク

サブエージェントに複数のタスクを詰め込まない。

```
❌ 悪い例:
「ファイル構造を調べて、テストを実行して、ドキュメントを更新して」

✅ 良い例:
サブエージェント1:「src/以下のファイル構造を調べてレポートして」
サブエージェント2:「テストを実行して失敗箇所を報告して」
（ドキュメント更新はメインで実行）
```

1タスクに集中させると:
- プロンプトが明確になり、無駄な試行錯誤が減る
- 1つが失敗しても他に影響しない
- 結果が簡潔になり、メインのコンテキストを圧迫しない

### 戦略2: プロンプトに十分なコンテキストを渡す

サブエージェントはメインの会話履歴を知らない。必要な情報はプロンプトに含める。

```
❌ 悪い例:
「さっき話していたバグを修正して」
→ サブエージェントは「さっきの話」を知らない

✅ 良い例:
「src/api/handler.ts の getUserById 関数で、
  id が undefined のとき TypeError が出る。
  null チェックを追加して修正して」
→ サブエージェントが即座に作業できる
```

ただし、プロンプトが長すぎるのも問題だ。100文字未満のプロンプトは曖昧すぎ、1,000文字以上は冗長すぎる。200-500文字が目安。

cc-safe-setupの`subagent-context-size-guard`hookは、100文字未満のプロンプトを警告する。

### 戦略3: 並列起動を活用する

独立したタスクなら、サブエージェントを同時に複数起動できる。

```
メイン: 「AとBを同時に調べて」
  → サブエージェントA: Aを調査
  → サブエージェントB: Bを調査
  （並列実行）
メイン: 両方の結果を受け取って判断
```

直列だと「A調査 → 結果確認 → B調査 → 結果確認」で4ステップかかるが、並列なら2ステップで済む。トークン消費はほぼ同じだが、作業時間が半分になる。

### 戦略4: 探索はサブエージェント、実装はメイン

サブエージェントはファイルの編集もできるが、メインの文脈なしに編集すると意図しない変更が生まれやすい。

推奨パターン:

```
サブエージェント: 「この問題の原因を調査してレポートして」
  → 調査結果を返す

メイン: 調査結果を読んで、自分で修正を実装
```

調査でコンテキストが膨張するのを防ぎつつ、実装はメインの全体理解に基づいて行う。

### 戦略5: 同時起動数を制限する

5分間に5個以上のサブエージェントは多すぎる。

- サブエージェント1個 = 数千トークン
- 5個同時 = 1万トークン以上が一瞬で消費
- 10個同時 = quotaに直接的な打撃

`subagent-budget-guard`hookで上限を設定しておくと安全だ。

## よくある罠

### 罠1: 単純検索にサブエージェントを使う

```
❌ 「'handleError'が使われている場所を探して」
→ サブエージェント起動 → 内部でGrep実行 → 結果返却

✅ 直接Grepを実行
→ 結果がすぐ出る。コスト1/30
```

Grep、Glob、Read（limit付き）で済む作業にサブエージェントを使わない。

### 罠2: サブエージェントの結果を信じすぎる

サブエージェントはメインの文脈を知らない。「問題ありません」という報告が、メインの文脈では間違っていることがある。

サブエージェントの結果は「参考情報」として受け取り、最終判断はメインで行う。

### 罠3: サブエージェントが古いコードを見ている（stale worktree）

サブエージェント（特にExplore agent）は、隔離されたworktreeで動く。このworktreeは`origin/main`を基準に作成されるため、**ローカルの未コミット変更やフィーチャーブランチの変更が見えない**（[#49169](https://github.com/anthropics/claude-code/issues/49169)）。

結果として:
- 「このファイルにバグがある？」→ サブエージェントは**修正前のコード**を分析して「問題ありません」と答える
- 「リファクタリングの影響範囲を調べて」→ **古い構造**で依存関係を調査し、間違った結果を返す
- 警告は一切出ない。**間違いに気づく手がかりがない**

対策:
1. サブエージェントに調査を任せる前に`git push`しておく（worktreeが最新になる）
2. 絶対パスを指定する（例: `/home/user/project/src/file.ts`）。worktreeではなく実際の作業ディレクトリを直接読む
3. 重要な分析はメインエージェントで行う

### 罠4: CLAUDE.mdの内容が渡らない

Claude Codeのバージョンによっては、サブエージェントにCLAUDE.mdの内容が自動的に渡されないことがある。

重要なルール（コーディング規約、禁止操作など）はサブエージェントのプロンプトにも明記するか、`subagent-claudemd-inject`hookでCLAUDE.mdの内容を自動注入する。

### 罠5: 並列サブエージェントのファイル競合

2つ以上のサブエージェントが同じファイルを触ると、「File modified since read」エラーが発生する。

実例（GitHub Issue [#46968](https://github.com/anthropics/claude-code/issues/46968)）:
- 並列サブエージェントが同じファイルを読み書きしようとした
- 「File modified since read」エラーが連続発生
- 各サブエージェントがリトライを繰り返し、**101,000トークン以上**がretryループで消費された
- さらにセッション切り替えで150,000トークン以上が完全にロスした

対策:
- 並列サブエージェントには**重複しないファイル範囲**を指定する
- 「src/api/を調べて」「src/models/を調べて」のように、作業範囲を明確に分ける
- 同じファイルへの書き込みが必要なら、直列（1つずつ順番）にする

## 実践: サブエージェント使用の判断フロー

```
作業を依頼したい
  ├─ 1-2ファイルの検索/読み込み → 直接やる
  ├─ 3-10ファイルの調査 → 直接やる（コンテキストに余裕があれば）
  ├─ 10ファイル以上の広範な調査 → サブエージェント
  ├─ 独立した並列タスク → サブエージェント（複数同時）
  ├─ メインの文脈が必要な作業 → メインでやる
  └─ 編集・実装 → メインでやる（調査だけサブエージェントに）
```

この判断を毎回意識するだけで、サブエージェントの無駄遣いが大幅に減る。
