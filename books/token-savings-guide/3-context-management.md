title: "第3章 コンテキスト管理——/clear、/compact、--resumeの正しい使い方"

コンテキストウィンドウは「見えない財布」だ。会話が進むほど中身が減り、空になるとセッションが終わる。

この章では、コンテキストを効率よく使うための3つのコマンドと、よくある罠を解説する。

## コンテキストウィンドウの基本

Claude Code（Opus 4.6）のコンテキストウィンドウは1Mトークン。大きく見えるが、実際には:

- セッション開始時点でシステムプロンプト、CLAUDE.md、ツール定義で数万トークンを使う
- 会話の各ターンで、過去のやりとり全体が再送される
- ファイルの読み込み結果、コマンド出力がどんどん蓄積される

体感では、普通に作業していると2-3時間で75%に達する。ここからが勝負だ。

## /clear——会話を完全リセット

`/clear`は会話履歴をすべて消去する。コンテキストウィンドウがゼロに戻る。

### いつ使うか

- まったく別のタスクに切り替えるとき
- 前の会話の文脈が邪魔になるとき（デバッグ中に別のバグを追い始めた場合など）

### 注意点

`/clear`は**不可逆**だ。消した会話は戻らない。

よくある事故: スラッシュコマンドの入力ミス。`/c`と打ってEnterすると、`/clear`と`/compact`のどちらが実行されるか分からない。自分の環境では`/clear`が優先されて、30分の作業が消えたことがある。

対策として、`/clear`の入力時にブロックするhookを設定している:

```json
{
  "hooks": {
    "UserPromptSubmit": [{
      "matcher": "^/clear$",
      "hooks": [{"type": "command", "command": "bash /path/to/clear-command-confirm-guard.sh"}]
    }]
  }
}
```

`/clear`はスラッシュコマンドなので、`PreToolUse`ではなく`UserPromptSubmit`でインターセプトする点に注意。cc-safe-setupの`clear-command-confirm-guard`はこの問題を解決する。`/clear`を完全ブロックし、`/compact`への誘導メッセージを表示する。

### /clearの使いすぎに注意

`/clear`のたびに、CLAUDE.mdの再読み込み、ツール定義の再送信が発生する。1セッション中に5回`/clear`するくらいなら、新しいセッションを立ち上げた方がいい。

## /compact——会話を要約して圧縮

`/compact`は会話履歴を要約し、圧縮する。**情報を保ちながら**トークンを解放する。

### /clearとの違い

| 項目 | /clear | /compact |
|------|--------|----------|
| 会話履歴 | 完全消去 | 要約して保持 |
| 文脈の維持 | なし | あり（要約精度に依存） |
| トークン解放量 | 最大（全解放） | 中程度（要約分は残る） |
| リスク | 情報の完全喪失 | 要約時の情報落ち |

### いつ使うか

- コンテキストが75%を超えたとき
- 「そろそろ重くなってきたな」と感じたとき
- 大きなファイルを読んだ後（不要になった内容を圧縮するため）

### 手動/compactの方がauto-compactより安い

Claude Codeにはauto-compact機能がある。コンテキストが約90%に達すると自動で圧縮が走る。

しかし、auto-compactは**最悪のタイミング**で発動する。

- 約90%に達してから圧縮するため、圧縮前の数ターンが最もトークンを食う区間になる
- 圧縮処理自体にもトークンを消費する
- 圧縮後にClaude Codeが「何をしていたか」を思い出すために追加のツール呼び出しをすることがある

自分の運用では、**75%の時点で手動/compact**がルールだ。早めに圧縮した方が:

1. 圧縮前の「重い区間」が短くなる
2. 要約の質が高い（情報量が少ないうちに要約する方が精度が良い）
3. 予測可能なタイミングで作業を区切れる

### /compactの前にやること

圧縮すると、細かい文脈が落ちることがある。圧縮前に以下を確認:

1. **作業途中のファイル名をメモする**: 圧縮後に「どのファイルを編集していたか」を忘れることがある
2. **次にやることを明示的に伝える**: 「/compactの後、〇〇の実装を続けてください」
3. **重要な判断・決定を記録する**: mission.mdやタスクファイルに書き出しておく

cc-safe-setupの`pre-compact-checkpoint`hookは、圧縮前に自動でgit commitを作成する。圧縮後に「何を変更していたか」をgit logで確認できる。

## --resume——前回のセッションを再開

`claude --resume`は、過去のセッションを選んで再開する。`claude --continue`は、カレントディレクトリの直前のセッションを続ける。どちらも新しいセッションを開始せずに、前回の続きから始められる。

### いつ使うか

- 中断したタスクの続きをやるとき
- 前回のセッションで得た文脈が必要なとき
- 朝、昨日の作業を再開するとき

### 注意点

`--resume`で再開すると、**前回のコンテキストがそのまま復元される**。つまり:

- 前回のセッションで75%使っていたら、再開直後も75%からスタート
- 前回の会話全体がコンテキストに含まれるので、最初のターンから重い

`--resume`後に`/compact`を実行するのが効率的な使い方だ:

```bash
claude --resume
# 再開直後
# > /compact 昨日の〇〇の続きをやります
```

こうすると、前回の文脈を要約で保持しつつ、クリーンなコンテキストで作業を始められる。

### --resumeを使うべきでないとき

- 前回と全く関係ないタスクをやるとき（素直に新規セッション）
- 前回のセッションが中途半端にエラーで終わったとき（壊れたコンテキストを引き継ぐリスク）

## よくある罠

### 罠1: 「もったいない」で/compactしない

「まだ25%残ってる」と思って圧縮を先延ばしにするパターン。コンテキストが90%を超えると:

- 各ターンの処理が遅くなる
- auto-compactが不意に発動して文脈が失われる
- 最悪の場合、セッションが強制終了する

75%ルールを守れ。

### 罠2: 大きなファイルを読んだまま放置

`Read`で3,000行のファイルを読むと、その内容がコンテキストに丸ごと入る。作業が進んでファイルの内容が不要になっても、消えない。

対策:
- 大きなファイルは`limit`パラメータで必要な部分だけ読む
- 読んだ後、別の作業に移る前に`/compact`する
- `large-read-guard`hookで100KBを超えるファイルの読み込みを事前警告する

### 罠3: /compactの直後に大量の情報を投げる

圧縮直後は「何も覚えていない」状態に近い。そこで「このファイルとあのファイルを読んで、この修正をして、テストを実行して...」と一気に指示すると、圧縮の意味がなくなる。

圧縮後は:
1. まず1つの指示を出す
2. 結果を確認する
3. 次の指示を出す

焦らず、1ステップずつ進める方がトークン効率が良い。

### 罠4: auto-compactの後で暴走する

auto-compactの後、Claude Codeが圧縮前の会話を「まとめ」として読み、勝手に行動を始めることがある。「前の会話で〇〇をやると言っていたから、始めよう」というパターンだ。

これは意図しない変更を生む原因になる。cc-safe-setupの`pre-compact-checkpoint`hookは、圧縮前にチェックポイントを保存すると同時に、「圧縮後は指示を確認してから行動してください」というメッセージを注入する。これにより、圧縮後の暴走を抑制できる。

## 実践: セッション設計のテンプレート

効率的なセッションの流れ:

```
1. セッション開始（or --resume）
2. タスクを明確に伝える
3. 作業（コンテキスト 0% → 75%）
4. 手動 /compact「次は〇〇をやります」
5. 作業（コンテキスト 30% → 75%）
6. 手動 /compact
7. 作業
8. 必要に応じてもう1回 /compact
9. セッション終了
```

1セッションで2-3回の`/compact`が標準的なペースだ。

auto-compactに3回以上頼っているなら、手動`/compact`のタイミングが遅すぎる。
