「何もしていないのに減っている」——Claude Codeユーザーが最も戸惑う瞬間だ。

[GitHub Issue #46829](https://github.com/anthropics/claude-code/issues/46829)では、ある開発者が「92日で$1,582燃やした」と報告した。HNの[Pro plan除外騒動](https://news.ycombinator.com/item?id=47855832)（2026-04-21、400+コメント）では「Quota hit before lunch」というフレーズが繰り返された。$200/月のMax planを払っているのに、午前中で枯渇する人が後を絶たない。

「何もしていない」わけではない。Claude Codeはあなたが見ていないところでもトークンを使っている。問題は、それがどこで起きているかが見えにくいことだ。

この章では、トークンがどこで使われているかの全体像を整理する。全体像が見えれば「自分の場合はここが原因だ」と特定できるようになる。

## トークン消費の4つの層

Claude Codeのトークン消費は、大きく4つの層に分かれる。

### 層1: システムプロンプト（セッション開始時）

Claude Codeが起動するたびに、内部のシステムプロンプトが読み込まれる。これはユーザーには見えないが、毎回トークンを消費する。

加えて、以下がセッション開始時に読み込まれる:
- **CLAUDE.md**: グローバル（`~/.claude/CLAUDE.md`）とプロジェクトローカル（`./CLAUDE.md`）の両方
- **ツール定義**: 利用可能なツールの説明。Deferred Tool Loading（v2.1.89以降）で一部遅延されるが、基本ツール群は常に含まれる
- **MCP設定**: MCPサーバーのツール定義も追加される

つまり、**CLAUDE.mdが長いほど、セッション開始時のトークン消費が増える**。100行と500行では、毎ターンの基礎消費が変わる。

### 層2: コンテキストウィンドウ（会話の蓄積）

会話が進むにつれ、過去のやりとりがすべてコンテキストウィンドウに蓄積される。Claude Codeは1Mトークンのコンテキストを持つが、蓄積が増えるほど：

- 各ターンで送信されるトークン量が増える
- レスポンス生成に使われるトークンも増える
- **プロンプトキャッシュ**が効いていれば再利用されるが、キャッシュが壊れると全量がカウントされる

プロンプトキャッシュの破壊は、トークン急増の最大の原因の一つだ（第8章で詳述）。

### 層3: ツール呼び出し（読み書きの実行）

Claude Codeがファイルを読んだり、Bashコマンドを実行するたびにトークンが消費される:

- **Read**: ファイルの内容がコンテキストに追加される。大きなファイルほど消費が大きい
- **Glob/Grep**: 検索結果がコンテキストに入る
- **Bash**: コマンドの出力がコンテキストに入る。`ls -la`の出力と`cat large-file.txt`の出力では桁が違う
- **Edit/Write**: 変更内容がコンテキストに記録される

ツール呼び出しのたびに、入力（ツールのパラメータ）と出力（実行結果）の両方がトークンとしてカウントされる。

なお、ツール結果には合計200Kトークンの内部予算上限がある。これを超えるとツール結果が無断で切り捨てられ、不完全な情報をもとにClaudeが判断→リトライ→追加消費というパターンが起きる（第8章で詳述）。

### 層4: サブエージェント（並列処理のコスト）

`Agent`ツールでサブエージェントを起動すると、各サブエージェントが独自のコンテキストを持つ。メインのコンテキストからは消えないが、APIコール数は増える。

サブエージェントは便利だが、使いすぎるとトークン消費が急増する（第5章で詳述）。

## まず自分の消費量を確認する

対策の前に、現在の消費量を知ることが重要だ。Claude Codeには2つの確認方法がある:

- **`/cost`**: API課金ユーザー向け。セッション中のトークン消費量とコストを表示
- **`/stats`**: Max/Proプラン向け。使用パターンを表示（サブスクリプション課金のため金額は出ない）

ステータスラインに常時表示することもできる（`/config`→statusline設定）。対策の効果を測定するために、最適化の前後で`/cost`の値を比較する習慣をつけたい。

## 何が最もトークンを食うか

800時間の運用データから、トークン消費の大きさを順に並べると:

1. **大きなファイルの読み込み**: 数千行のファイルを`Read`で読むと一気に消費が増える
2. **コンテキストの肥大化**: `/compact`せずに長時間使い続けると、各ターンの基礎消費が膨張
3. **プロンプトキャッシュの破壊**: キャッシュが効かなくなると、同じプロンプトでも消費が数倍に
4. **不要なツール呼び出し**: ファイル一覧の取得、gitログの表示など、実際には不要な情報収集
5. **CLAUDE.mdの肥大化**: 500行を超えるCLAUDE.mdは、毎ターン数千トークンの固定コスト
6. **状態管理ファイルの頻繁な更新**: mission.mdやtask-queueを毎ツール呼び出しごとに更新するパターン。対策と実測データは第7章で詳述する
7. **アイドルセッションの放置**: 何もしていないのにセッションを開いたまま放置すると、バックグラウンドでトークンが消費される。2時間で18%のquotaが消えた報告がある

## この本で扱う対策の全体像

| 消費源 | 対策 | 解説章 |
|--------|------|--------|
| CLAUDE.mdの肥大化 | 100行以内に圧縮、5パターンの書き方 | 第2章 |
| コンテキスト蓄積 | /clear・/compact・--resumeの適切な使い分け | 第3章 |
| 不要なツール呼び出し | hookで検知・ブロック | 第4章 |
| サブエージェント乱用 | 適切な使い分けと上限設定 | 第5章 |
| 非効率なワークフロー | タスク分割と順序の最適化 | 第6章 |
| キャッシュ破壊 | 原因の理解と回避策 | 第8章 |

次の章から、具体的な対策に入る。

## 自分のケースで「どこに埋まっているか」を見る

構造がわかっても、自分のセッションでどれが犯人なのかは別問題だ。

確認には2段階ある。

**1段階目:** `/cost`（APIユーザー）か`/stats`（Max/Proプラン）でセッション全体の消費量を見る。

これだけでも「思ったより使っている／意外と大丈夫」の肌感がつく。ただし総量しか出ない。どの層で漏れているかまでは分からない。

**2段階目:** ローカルログから浪費パターンを検出する。Claude Codeは`~/.claude/projects/*/`に全会話ログを自動保存していて、そこから「どのセッションが$いくら使ったか」「キャッシュミスはどのくらい発生しているか」「100ターン超えのrunaway sessionは何本あるか」が事後的に分かる。

ログを手で読むのは辛いので、私は無料ツールを書いた。

```bash
npx github:yurukusa/cc-token-diet
```

（npm 公開準備中のため、現時点では GitHub 直接の形で起動する。npm 公開後は `npx cc-token-diet` 短縮形にも対応する予定）

実行すると次のような診断が出る（筆者の過去3日のデータ）:

```
Sessions analysed: 27   Assistant turns: 4,689
Estimated API-equivalent spend: $1,368.14 (Opus pricing)
Cache hit ratio: 99.0%

🔥 Waste patterns:
  ❗ 10 runaway session(s) (>100 assistant turns) (≈ $393.39 wasted)
     Fix:  Use /compact or restart for each logical task.
     See:  Token Book ch3 (context management)

🌡️  Hottest sessions:
  $431.08  cc-loop     1103 turns / 180min
  $192.83  cc-loop      664 turns / 143min
```

何がわかるか——「過去3日で$1,368相当のAPI換算spend」「そのうち$393は100ターン超えのセッションで溶かした」という**具体的な金額**だ。Proプランを$20/月で使っていても、Max $100に上げるべきか・自分の使い方を直すべきかを判断する材料になる。

本文中に出てくる対策は、この数字が動くかどうかで効果を判定できる。**対策の前後で同じコマンドを打つ**習慣をつけるとよい。

ツールは100%ローカル動作で外部送信しない（ソースは1ファイル）。詳細は[cc-token-diet](https://github.com/yurukusa/cc-token-diet)。

次の第2章では、最も即効性がある対策——CLAUDE.mdの最適化——に入る。100行のCLAUDE.mdを35行に圧縮しただけで、キャッシュヒット率が89%→95%に改善した実測データを示す。

「CLAUDE.mdの肥大化」が表で第5位だったが、実は**毎ターン**の固定コストとして効いてくるため、累積では他のどの対策よりも効果が出やすい。1日に何時間もClaude Codeを動かす人ほど、第2章の効果は大きい。

なお、本書はAnthropicの挙動変化（Opus 4.7のキャッシュTTL短縮、Pro plan仕様変更など）に追随して継続的にアップデートしている。最終更新日はZenn上で確認できる。
