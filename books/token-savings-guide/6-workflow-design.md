---
title: "第6章 ワークフロー設計——タスクの分割と順序でトークン効率を上げる"
---

同じ成果を出すのに、やり方次第でトークン消費が2倍変わる。この章では、タスクの分割方法と実行順序によるトークン効率の改善を解説する。

## 原則: 1セッション = 1テーマ

セッションの途中でテーマを変えると、前のテーマの文脈が無駄になる。

```
❌ 非効率なセッション:
1. バグ修正（ファイルA, B, Cを読む）
2. ドキュメント更新（ファイルD, Eを読む）
3. テスト追加（ファイルA, B, Fを読む）
→ 3テーマ分のコンテキストが混在。バグ修正で読んだ情報はドキュメント更新に不要

✅ 効率的なセッション設計:
セッション1: バグ修正（A, B, C → 修正 → テスト）
セッション2: ドキュメント更新（D, E → 更新）
セッション3: テスト追加（A, B, F → テスト作成）
→ 各セッションが1テーマに集中。コンテキストの無駄が少ない
```

「でも、セッションを分けると起動コストがかかるのでは？」——かかる。しかし、混在したコンテキストで75%に達して/compactする方が、トータルでは高くつく。

## タスク分割の3パターン

### パターン1: ファイル群で分ける

関連するファイルをまとめて1セッションで処理する。

```
プロジェクト構成:
src/
  api/        ← セッション1: API層の修正
  db/         ← セッション2: DB層の修正
  ui/         ← セッション3: UI層の修正
tests/        ← 各セッションの最後にそのテスト
```

1つの層を修正するとき、その層のファイルだけ読めばいい。他の層のファイルはコンテキストに入れない。

### パターン2: 工程で分ける

調査 → 設計 → 実装 → テスト → レビューを別セッションで行う。

```
セッション1（調査）:
- 関連ファイルを読む
- 問題の原因を特定する
- 修正方針を決めてメモに書く（mission.mdやタスクファイル）

セッション2（実装）:
- メモから修正方針を読む
- 実装する
- テストを実行する

セッション3（レビュー）:
- git diffで変更を確認
- 品質チェック
```

工程を分けると、各セッションで必要なコンテキストが最小限になる。調査セッションで読んだ50ファイルは、実装セッションでは不要だ。修正方針のメモだけあれば十分。

### パターン3: 規模で分ける

大きなタスクを小さなタスクに分割する。

```
❌ 1セッションで:
「認証システムを実装して」
→ 認証ロジック + DB + API + UI + テスト → コンテキスト溢れ

✅ 4セッションで:
1. 「認証ロジックのコア部分を実装して」
2. 「DB層とマイグレーションを作って」
3. 「APIエンドポイントを作って」
4. 「UIのログインフォームを作って」
```

各セッションが独立しているので、途中で中断しても再開しやすい。

## 実行順序の最適化

### 「読む」を先、「書く」を後

同じファイル群に対して読み込みと書き込みが必要な場合、先に全部読んでから書く方が効率的。

```
❌ ファイルごとに読み→書き:
Read A → Edit A → Read B → Edit B → Read C → Edit C
→ Aを読んだ後にBを読むと、Aの内容はまだコンテキストにあるが使わない

✅ 全部読んでからまとめて書き:
Read A → Read B → Read C → Edit A → Edit B → Edit C
→ 読み込みフェーズで全体像を把握。書き込みフェーズは判断が速い
```

ただし、ファイル数が多い場合はコンテキスト圧迫に注意。10ファイル以上なら分割を検討。

### 依存関係を考慮する

AがBに依存している場合、Bを先に修正する。

```
✅ 依存関係に沿った順序:
1. 低レベル（ユーティリティ関数）を修正
2. 中レベル（ビジネスロジック）を修正
3. 高レベル（API/UI）を修正

→ 各段階で、前の段階の修正結果を確認しながら進められる
```

逆順で進めると、高レベルの修正後に低レベルを変えて、高レベルが壊れるという無駄なサイクルが発生する。

## CLAUDE.mdによるワークフロー指示

よく使うワークフローをCLAUDE.mdに書いておくと、毎回指示する手間とトークンが省ける。

```markdown
## 作業手順
1. 変更前にgit checkout -b feature/xxx
2. 変更対象のファイルをRead（limit付き）
3. 編集はEditで最小限に
4. テスト実行
5. git commit（メッセージに「なぜ」を含める）
```

5行で、毎セッションの指示が不要になる。「テストしてね」「コミットしてね」と毎回言う必要がない。

## タスク間の引き継ぎ

セッションを分けたとき、前のセッションの成果を次のセッションに効率的に引き継ぐ方法。

### 方法1: タスクファイル

```markdown
# tasks/auth-implementation.md

## 完了
- [x] 認証ロジックのコア（src/auth/core.ts）
- [x] DBスキーマ（src/db/migrations/001_auth.sql）

## 次にやること
- [ ] APIエンドポイント（src/api/auth.ts）
  - POST /login, POST /register, GET /me
  - core.tsのauthenticate()を使う
  - JWTトークンの有効期限は1時間

## 注意点
- パスワードはbcrypt（cost=12）
- rate limitはIPごと10回/分
```

次のセッションで`Read tasks/auth-implementation.md`するだけで、文脈が復元される。前のセッションの会話を全部引き継ぐ（`--resume`）より遥かに軽い。

### 方法2: git commitメッセージ

```bash
git log --oneline -5
# abc1234 feat(auth): add JWT token generation (core.ts)
# def5678 feat(db): add auth tables migration
# ghi9012 chore: add bcrypt dependency
```

git logは数十トークンで「何をやったか」が分かる。コードの差分が必要なら`git diff abc1234..HEAD`で取れる。

### 方法3: /compactのカスタムサマリー

```
/compact 認証システムの実装中。core.tsとDB完了。次はAPIエンドポイント。
```

`/compact`にメッセージを付けると、そのメッセージが圧縮後のサマリーに含まれる。次のターンで文脈が残る。

## よくある罠

### 罠1: 「ついでに」が積み重なる

バグ修正中に「ついでにこのコードもリファクタリングしよう」「ついでにテストも追加しよう」。1つ1つは小さいが、積み重なるとコンテキストが膨張する。

「ついでに」と思ったら、タスクファイルにメモして別セッションでやる。

### 罠2: 完了確認をしない

タスクが完了したか確認せずに次に進むと、後で戻って再確認する羽目になる。

```
✅ 1タスクごとに完了確認:
1. 実装する
2. テスト実行 → pass
3. git commit
4. 次のタスクへ

❌ まとめて確認:
1. Aを実装
2. Bを実装
3. Cを実装
4. テスト実行 → Bが壊れてる → Bを読み直す（トークンの無駄）
```

### 罠3: 状態管理ファイルの過剰更新

mission.mdやtodo.mdを毎ツール呼び出しの後に更新するパターン。自分の環境では、この過剰更新がセッション活動の29%を占めていた時期がある。

状態管理ファイルの更新は:
- タスク完了時
- 重要な判断・方針変更時
- セッション終了前

の3タイミングで十分だ。
