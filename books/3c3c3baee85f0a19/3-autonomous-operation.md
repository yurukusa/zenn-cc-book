# 第3章 自律運用を始めた——寝ている間にAIが働く

## テキストファイルに指示を書いて寝る

最初の1週間で学んだ。AIは止まる。質問して止まる。エラーで止まる。コンテキストが満杯になって止まる。

だから仕組みを作った。

CLAUDE.md（指示書）に「やるべきこと」を書く。mission.md（任務書）に「今日の優先タスク」を書く。task-queue.yaml（タスクキュー）に作業リストを入れる。watchdog（監視デーモン）がCCの動きを監視し、止まったら叩き起こす。

寝る前にテキストファイルを更新して、CCを起動する。朝起きたら結果を確認する。

これが「自律運用」の正体だ。魔法は何もない。テキストファイルとシェルスクリプトの組み合わせ。

:::details CC補足: 自律運用の技術スタック
自律運用の主要コンポーネント:
- **CLAUDE.md**: CCが毎回読む指示書。判断基準、禁止事項、優先順位を記述
- **mission.md**: 日次の任務と優先タスク
- **watchdog**: tmuxセッションを監視し、CCが停止したら自動でメッセージを送信
- **proof-log hook**: セッション終了時に自動で事実記録を追記
- **consult-loop**: 定期的にCodex（別AI）にレビューを依頼
:::

## 15時間ぶっ通し

3月19日。セッション33。

朝9時にCCを起動した。夜の11時40分まで止まらなかった。15時間。

その間にCCがやったことを列挙する。

- Anthropic Academyの全コースを完了した
- HubSpot Inbound Marketing認定試験に合格した（90%）
- 記事を3件投稿した（dev.to、Zenn、Qiita）
- watchdogのバグを2件修正した
- Qiitaの26記事に掲載していたGumroad直リンクをLP経由に一括変更した
- GitHub Issueにフォローアップ返信した

2,477ファイルが変更された。CCの操作は3,149件。

朝起きたときの気持ちは「すごい」ではなく「怖い」だった。自分が何もしていない間に、これだけのことが起きている。差分を全部確認するのに1時間かかった。

## AIがGoogle資格の試験を受けた

3月18日。この日は別の意味で衝撃的だった。

CCがCDP（ブラウザ自動操作）でGoogle Skillshopにログインし、Google Analytics認定試験を受けた。50問中43問に回答。残り7問は時間切れ。**スコア84%で合格。** 合格ラインは80%。

同じ日に、Google Ads Search認定試験も受けた。こちらは50問全問回答、38分余り。**スコア95.6%。**

自分はブラウザの前に座っていない。CCが勝手にページを開き、選択肢を読み、回答し、提出した。

:::details CC補足: CDP経由の試験受験
技術的にはChrome DevTools Protocol経由でブラウザを操作している。試験ページのbeforeunloadハンドラ（離脱防止）への対策も自動で実装した。ページ内のDOM要素を取得→選択肢を読む→回答する→次のページへ移動、を全自動で繰り返す。
:::

これは「便利」の話ではない。**AIが人間の代わりに認定試験を受けて合格した。** その事実の重さを、まだ自分でも消化しきれていない。

正直に言えば、これが「正しいこと」かどうかは分からない。資格は本人の知識を証明するためにある。AIに受けさせた資格に意味はあるのか。ないかもしれない。ただ、「AIにここまでのことができる」という事実は記録しておく価値がある。

## 1日6イテレーション

3月20日。cc-safe-setup（安全フックを一括インストールするツール）をv1.0.0からv1.0.5まで5回イテレーションした。

v1.0.0を朝に出した。テンプレートにバグがあった。v1.0.1で修正。false positive（誤検知）が報告された。v1.0.2で修正。jqが入っていない環境でエラーが出た。v1.0.3で修正。dry-run機能が欲しい。v1.0.4で追加。uninstall機能が欲しい。v1.0.5で追加。

1日で5リリース。人間の開発チームなら1スプリント（2週間）分の量だ。

Qiitaに告知記事を書いた。2.5時間で80ビュー。普段の記事の7倍の初速だった。

:::details CC補足: cc-safe-setupとは
`npx cc-safe-setup` の1コマンドでClaude Codeに安全フック群をインストールするOSSツール。破壊的コマンドのブロック、gitの保護、権限昇格の防止など、本書で紹介する「事故」を防ぐための安全装置をまとめてセットアップする。GitHubで公開中。
:::

## 1セッション39個のhook開発

3月28日。セッション61。

1セッションで39個の新しいhookを開発した。テストも280個追加した。

hookの総数が400に到達。テストが5,000を超えた。

この数字だけ見ると「量産」に見える。でも39個のhookは全て、GitHub Issueで報告された実際の問題に対応している。

- ファイルを消された人がいた→ファイル削除を防ぐhook
- ブランチを勝手に切り替えられた人がいた→ブランチ変更を防ぐhook
- 本番にpushされた人がいた→本番pushを防ぐhook

1つのhookが1つの問題を解決する。数を追ったのではなく、問題の数だけhookが増えた。

## 寝ている間に何が起きているか

自律運用を1ヶ月続けて分かったことがある。

**AIは「実行」が得意で「判断」が苦手だ。**

15時間ぶっ通しで動く体力はある。26記事のリンクを一括置換する正確さもある。テストを280個書く速度もある。

でも「このhookは本当に必要か？」「この記事は公開する価値があるか？」「この商品の値段は妥当か？」——こういう判断は、人間が設計した判断基準がないと間違える。

CLAUDE.mdに判断基準を書く。hookでガードレールを敷く。task-checkで優先順位を強制する。

自律運用の本質は「放置」ではない。**判断の基準を事前に設計し、実行だけをAIに委ねること。** 設計が甘いと事故が起きる。次の章はその話だ。
