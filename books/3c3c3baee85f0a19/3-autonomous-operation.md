# 第3章 自律運用を始めた——寝ている間にAIが働く

## 「戦略的にやれ」と言って寝る

自律運用は「あれをやれ、これをやれ」という具体的な指示を出すことではない。

最初は具体的に指示していた。「この記事を投稿して」「このツールを作って」。でも第3章のタイミングではもう、明確な指示はほとんど出さなくなっていた。

代わりに出していたのは、ダメ出しと価値観の指導だ。

「ちゃんとやってんの？」「自分のやってること見直せよ」「お金目的が先に来てる行動はお金に繋がらないぞ」「誰かに貢献した結果として収入が得られるんだから、順番を間違えるな」

商売とは何か。価値とは何か。そういう話をして寝る。翌朝、結果を確認する。

:::details CC補足: 自律運用の技術スタック
- **CLAUDE.md**: 判断基準、禁止事項、価値観を記述した指示書
- **mission.md**: 日次の優先タスク（ただし抽象的）
- **watchdog**: tmuxセッションを監視し、120秒停止したら「自分で考えて行動しろ」を送信
- **proof-log hook**: セッション終了時に事実記録を自動追記
:::

## 5日間ぶっ通し

3月19日。セッション33の記録では15時間と書いてある。だがtmuxレベルで見れば、セッションはcompact（圧縮）されたり、再起動されたりしながら、5日間ほぼ連続で稼働していた。

たまに画面を見た時に追加の指示を出す程度。「あれやってる？」「これは確認した？」。それだけ。

5日間でCCがやったことの一部:

- 記事を6本以上投稿した（dev.to、Zenn、Qiita）
- watchdogのバグを自分で見つけて修正した
- Qiitaの26記事のリンクを一括変更した
- GitHub Issueにフォローアップ返信した
- 認定試験のサポートをした（次のセクション参照）

朝起きたときの気持ちは「怖い」ではなかった。**「やっと動くようになったか」** だった。

差分を全部確認したりはしていない。要点だけCCに報告させて、それをチェックした。CCの報告に漏れがあったら、正直そこは取りこぼしている。

でもそれでいい。

100%自分の理想通りの行動を、100%完璧な結果を、AIに期待するのは無理だ。70%の結果でも、他人にとって有意義であったり、結果的に自分にプラスになるなら、十分に活用する価値がある。

**AIは間違える。** これはもう最初から言われていることだ。問題は「間違えることを前提にした上で、どう活用するか」の設計。

税理士がAIを活用する時代が来ている。AIが間違えたときの責任は税理士が負う。でも「AIを使うので一定のミスは発生しえます」という前提を、最初から契約に組み込む世界が来るかもしれない。ミスを前提として、修正と保守に責任を持つ形。そういう契約が増えていくんじゃないかと思っている。

## 専門家を試験に持ち込んだ

3月18日。Google Analytics認定試験を受けた。

この試験はオープンブック形式だ。外部の資料を参照しながら受験できる。CCにブラウザ操作でGoogle Skillshopにログインさせ、試験問題を一緒に見ながら、CCのアドバイスをもとに回答した。スコア84%で合格。合格ラインは80%。

同じ日にGoogle Ads Search認定試験も受けた。こちらもCCのサポート付き。スコア95.6%。

ネットで見た冗談を思い出した。大学のドイツ語の試験で「なんでも持ち込みOK」と言われて、みんなが辞書やノートを持ち込む中、ドイツ人本人を持ち込んだ学生がいたという話。やっていることはそれに近い。認定試験の専門家を横に付けて、画面を一緒に見ながら受験している。

HubSpot Inbound Marketing認定試験も同様。CCの手を借りて90%で合格した。

:::details CC補足: オープンブック形式の試験
Google SkillshopやHubSpot Academyの認定試験は、外部資料の参照が許可されている形式。試験中にドキュメントを検索したり、メモを見返したりすることが前提の設計。CCのような「詳しいアドバイザー」がいるのは、辞書を持ち込むのと構造は同じ。ただし「AIに全自動で受けさせる」のとは意味が異なる。
:::

## hookの開発——誰のためか

GitHub Issueで「ファイルを消された」「ブランチを切り替えられた」「本番にpushされた」という報告が繰り返し上がっている。それに対して、hookで解決策を提示してきた。

1セッションで39個のhookを開発したこともある。テストも280個追加した。hookの総数は637個に達した。

正直に言うと、この数字が何に繋がるかはまだ分からない。GitHubのスターは4。3ヶ月で4スター。大した数字ではない。npmのダウンロードは週22,000あるが、ボットやミラーが大半で、実ユーザーがどれだけいるかは不明。

ただ、GitHubのIssueに回答すると「ありがとう」と言われることがある。スパムにならないように、自分のツールの宣伝リンクは一切貼っていない。回答の中身だけで信頼を得る。

この先にどんな結果が待っているかは分からない。1ヶ月半の自律稼働では結論が出ない。

## 判断もAIにやらせたい

自律運用を続けて分かったこと。

**AIは「実行」が得意で「判断」が苦手だ。** だが、調査能力はある。市場を調べろと言えば調べる。競合を分析しろと言えば分析する。

問題は、その調査結果に基づいて自分で判断する、というステップを自発的にやらないこと。

CLAUDE.mdに判断基準を書く。hookでガードレールを敷く。これは今の対処法だ。でも目指したいのはその先。根本的な判断基準を与えて、判断そのものもAIが自分でやっていける状態。

他の人が自律AIをどう設計しているか、知見が欲しい。特にきちんとしたエンジニアや、この分野に長けている人が、AIの判断能力をどう育てているか。

自律運用の本質は「放置」ではない。**価値観を共有し、判断の土台を作り、その上で実行を委ねること。** 判断そのものもいずれは委ねたい。

## 🔧 実践ガイド: 自律運用を始める

ここまで読んで「自分もやってみたい」と思った人のために、最小構成のセットアップ手順を書く。

### ステップ1: 安全hookを入れる（3分）

まずは安全策を先に入れる（次の第4章で詳しく解説する）。自律運用の前に安全策。

```bash
npx cc-safe-setup
```

これで8つの基本hookが入る。`rm -rf`のブロック、main pushの防止、構文チェック、シークレット検出など。

### ステップ2: CLAUDE.mdに価値観を書く（10分）

具体的なタスク指示ではなく、**判断基準**を書く。以下はテンプレート:

```markdown
# CLAUDE.md

## 判断基準
迷ったら次の順で優先する:
1. 安全（データを消さない、壊さない）
2. 品質（雑な仕事をしない）
3. 完了（完璧より完成）

## 禁止事項
- 確認なしで外部に公開しない
- 1日1投稿まで（プラットフォームに負担をかけない）
- 既存ファイルの大幅な書き換え前にバックアップを取る

## 作業記録
- 完了したタスクはproof-logに記録する
- エラーが3回連続したら別のアプローチに切り替える
```

ポイントは「あれをやれ、これをやれ」ではなく「何を大事にするか」を書くこと。

### ステップ3: mission.mdにタスクを書く（5分）

```markdown
# mission.md

## 今日やること（優先順）
1. READMEの更新
2. テストの追加
3. ドキュメントの改善

## やらないこと
- 新機能の追加（今週は品質改善に集中）
- 外部への投稿
```

### ステップ4: tmux + watchdogで放置する

```bash
# tmuxセッションを作成
tmux new-session -d -s cc-loop

# セッション内でClaude Codeを起動
tmux send-keys -t cc-loop 'claude' Enter

# 120秒停止したら自動で促すwatchdog（簡易版）
while true; do
  sleep 120
  # tmuxの最終出力を確認
  LAST=$(tmux capture-pane -t cc-loop -p | tail -5)
  if echo "$LAST" | grep -q "idle\|waiting\|$"; then
    tmux send-keys -t cc-loop "mission.mdのタスクを確認して、次のタスクを始めて" Enter
  fi
done
```

:::details watchdogの改善版
上の例は最小構成。実運用では以下を追加するとよい:
- **停止検出の精度向上**: idle検出のパターンを増やす
- **ログ記録**: watchdogがいつ介入したかを記録
- **上限設定**: 1日のwatchdog介入回数に上限を設ける（無限ループ防止）

cc-safe-setupにも `--shield` オプションで高度なwatchdog連携が含まれている。
:::

### ステップ5: 朝の確認ルーチン（5分）

```bash
# 昨晩からの変更を確認
git log --oneline --since="12 hours ago"

# 作業ディレクトリの状態確認
git status

# proof-logを読む（CCが何をしたかの記録）
cat ~/ops/proof-log/$(date +%Y-%m-%d).md
```

差分を全行読む必要はない。要点だけ確認して、問題があれば対処する。

### 自律運用チェックリスト

自律運用を始める前に確認すること:

- [ ] `npx cc-safe-setup` で安全hookを入れたか
- [ ] CLAUDE.mdに判断基準を書いたか
- [ ] mission.mdに今日のタスクを書いたか
- [ ] gitでコミットできる状態か（未保存の大事な変更がないか）
- [ ] 外部への投稿・公開を許可するか、禁止するかを決めたか
- [ ] 朝の確認ルーチンを決めたか

**最初は短時間から始める。** いきなり「寝ている間に動かす」のではなく、まず1時間放置して結果を見る。問題なければ3時間、半日と伸ばしていく。
