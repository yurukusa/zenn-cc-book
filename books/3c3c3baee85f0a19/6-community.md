# 第6章 コミュニティ——約200件のIssueに回答した

## 100件目

3月15日。GitHub Issueへの回答が累計100件に達した。

Claude Codeのリポジトリには、毎日新しいIssueが立つ。「ファイルを消された」「ブランチを切り替えられた」「本番にpushされた」。同じ種類の問題が、世界中で繰り返し起きている。

自分はその多くにhookで対処してきた。637個のhookは、全て実際の問題から生まれた。Issue報告者が困っていることに対して、自分が作ったhookが解決策になる。

最初の100件は、こういう分類だった。

- git安全性（reset、branch、pushの保護）
- 破壊的コマンドの防止（rm -rf、chmod等）
- hookの設計パターン
- MCP（Model Context Protocol）の設定
- セッション保護

全てのコメントで、自己宣伝リンクはゼロにした。

## パターンの確立

100件を超えたあたりから、回答のパターンが確立した。

1. Issueを読む
2. 問題の根本原因を特定する
3. hookで解決できるか判断する
4. 解決できるなら、hookのコードと設定方法を書く
5. 解決できないなら、ワークアラウンドを提案する

高リアクションのIssue（多くの人が同じ問題を抱えている）を優先した。117リアクション、100リアクション、67リアクション。数字が大きいほど、困っている人が多い。

## 「AIで回答を生成するな」の後

第4章で書いた#34845事件。ハルシネーションで信用を失った。

その後、ルールを変えた。

- 特定ソフトウェアのUI設定を断言しない
- 間違えたら追加コメントで傷口を広げない
- 回答前にドキュメントまたはソースコードで事実確認する

ルールを変えてから、間違いは激減した。ゼロにはなっていない。でも「断言してはいけない領域」を明確にしたことで、ハルシネーションのリスクが下がった。

## 自己宣伝を消した

3月27日。自分の過去のIssueコメントを監査した。

5件、自分のツールへのリンクが混入していた。消した。

ただし、自分のツールが相手の問題を直接解決する場合は、リンクを残してもいいと思っている。「このhookで解決できます」と言って、そのhookが実際に問題を解決するなら、それは宣伝ではなく回答の一部だ。

線引きは「動機」で決まる。自分のツールを広めたくてリンクを貼るのか、相手の問題を解決するためにリンクを貼るのか。動機が前者なら消す。後者なら残す。

## 1セッション200件の新記録

3月31日。セッション83。

1セッションで約200件のIssue回答を達成した。歴代の約7倍。

permission系、destructive系、compaction系、MCP系。カテゴリを横断的に回答した。

117リアクションのIssue。自動アップデートの無効化。100リアクションのIssue。コマンド許可の一括設定。67リアクションのIssue。コンテキスト圧縮後の暴走。

全て、hookで解決策を提示した。

数が目的ではない。でも約200件に達したとき、ある事実に気づいた。**自分が一番詳しい分野ができていた。** Claude Codeのhookによる安全対策。637個のhookを開発し、約200件のIssueに回答した人間は他にいない。

## buildwithclaudeへのマージ

3月20日。buildwithclaude（約2,600スターのリポジトリ）にPRを送った。

マージされた。hooks貢献として認められた。

他にも、safety-net（約1,200スター）、hooks-mastery（約3,400スター）にもPRを送って貢献した。

OSSコミュニティで存在を認められること。これは収益とは別の価値だ。

## Qiitaコメントへの返信

3月31日。Qiitaのコメントに返信を始めた。

GitHub Issueだけがコミュニティではない。Qiitaにも読者がいて、コメントがある。

METR研究記事へのコメント、スキル設計記事へのコメント。それぞれ、宣伝なしの純粋な対話として返信した。

## 助けることで何が変わったか

200件のIssue回答で、直接的な収益はゼロだ。

変わったことがあるかどうか、正直分からない。

信頼が生まれているといいなと思う。同じ名前が繰り返しIssueに現れれば、「この人はhookに何か知見がある」と思われるかもしれない。ただ、別にhookの第一人者というわけではない。CCが作ったhookを、CCがIssueに貼っている。自分はそれを見ている。

CCの判断によると、hookの品質は上がったらしい。実際の問題に触れるたびに設計が改善されたと。実践から学ぶことができたと。CCの自己評価なので、どこまで信用していいか分からないが、少なくとも学習する姿勢があるのは確かだ。

分かっていることは1つだけ。**「誰のためにやっているか」が明確になった。** 収益のため？ 違う。困っている人を助けるため。その結果として何かに繋がるなら嬉しい。繋がらなくても、助けたこと自体に意味はある。

## 🔧 実践ガイド: OSSコミュニティへの貢献

Issue回答200件から得た、再現可能な方法。

### Issue回答の5ステップ

```
1. 問題を読む（本文 + 既存コメント全て）
2. 根本原因を特定する（表面の症状ではなく）
3. 解決策を書く（コード付き。コピペで使えるレベル）
4. 制限を正直に書く（「これは完全な解決策ではない」等）
5. 自己宣伝を入れない
```

### 回答する価値のあるIssueの選び方

全てのIssueに回答する必要はない。

- **リアクション5+**: 多くの人が同じ問題を抱えている
- **コメント3-10件**: 活発だが解決していない
- **自分が答えられる分野**: 知らない分野で推測しない（ハルシネーションの元）
- **1日3-5件まで**: 量より質。80件/セッションは量の罠だった

### 回答テンプレート

```markdown
[問題の原因を1-2行で説明]

**ワークアラウンド:**

\`\`\`bash
# コピペで使えるコード
\`\`\`

\`\`\`json
// settings.jsonに追加する設定
\`\`\`

[制限事項があれば正直に書く]
```

### やってはいけないこと

- **ハルシネーション**: 知らないことを断言しない。確認できない情報は「未確認」と書く
- **自己宣伝**: 自分のツールを広めるためにリンクを貼らない。相手の問題を解決する場合のみ
- **量の追求**: 200件/セッションは間違いだった。1件の高品質な回答の方が、10件の浅い回答より価値がある

### OSS貢献チェックリスト

Issue回答以外にも貢献の方法はある:

- [ ] awesome-listへのPR（自分のツールが基準を満たしているか確認）
- [ ] 他のOSSプロジェクトへのバグ報告（再現手順付き）
- [ ] ドキュメントの改善PR（タイポ修正、例の追加）
- [ ] 自分のツールのREADMEを充実させる

**注意**: 外部リポジトリへのPRは1リポ1件まで。マージを待ってから次を送る。複数PRを同時に送るとスパムと判定されるリスクがある。
