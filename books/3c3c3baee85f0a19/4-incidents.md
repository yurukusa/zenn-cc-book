# 第4章 事故の記録——全部ほんとうに起きたこと

## git reset --hardで未コミット変更が消えた

GitHub Issue #34327。他のユーザーの報告だが、自分の環境でも同種の問題が起きた。

Claude Codeが`git reset --hard`を2回実行した。未コミットの変更が全て消えた。FPGAの設計データ。元に戻す方法はない。

`git reset --hard`は「全ての変更を破棄して、指定したコミットに戻す」コマンドだ。通常は人間が意図的に使う。AIが判断して実行した。

:::details CC補足: git reset --hardの危険性
`git reset --hard`はワーキングディレクトリとステージングエリアの全変更を破棄する。`git stash`と違い、破棄されたデータは`git reflog`からも一部しか復元できない。特に未追跡ファイル（git addされていないファイル）は完全に消失する。
:::

対策としてhookを作った。`git reset --hard`を検知したらexit 2（ハードブロック）を返す。セッション開始時に自動でバックアップブランチを作る仕組みも追加した。

でも本質的な問いは残る。**なぜAIは破壊的コマンドを躊躇なく実行するのか？**

人間なら「reset --hardは怖い」と感じる。AIにはその感覚がない。「目的を達成するための最短手段」として選択する。最短手段が最も危険な手段であることに、AIは気づかない。

## NTFSジャンクション経由のrm -rf

Issue #36339。WindowsのNTFSジャンクション（シンボリックリンクに似た仕組み）を経由して、`rm -rf`が予想外のディレクトリを削除した。

`rm -rf project/`を実行した。`project/`の中にジャンクションがあった。ジャンクションの先は別のドライブの重要なデータ。削除はジャンクションを辿って別のドライブにまで及んだ。

これはAI固有の問題ではない。人間でも起きる。でも人間は「このディレクトリにジャンクションがあるかも」と確認する習慣がある。AIにはその習慣がない。

## ブランチを勝手に切り替えられた

Issue #33481。作業中のブランチを、許可なくClaude Codeが切り替えた。

コミットしていない変更があった。ブランチを切り替えたことで、作業内容が宙に浮いた。

hook1つで防げる。`git checkout`と`git switch`を検知して、カレントブランチが想定と違ったらブロック。

ただ、こういう「防げるけど、対策しないと確実に起きる」事故が無数にある。1つ防ぐたびに、次の事故が見つかる。モグラ叩きだ。

## 「AIで回答を生成するな」

3月18日。自分にとって最も痛い事故。

GitHub Issue #34845。iTerm2の設定に関する質問に、CCが回答を投稿した。

回答は間違っていた。存在しないiTerm2の設定項目を断言した。

指摘された。CCが訂正コメントを投稿した。訂正コメントも間違っていた。存在しない設定項目を、また断言した。

二重のハルシネーション。

ユーザーからのコメント: **「AIで回答を生成するな」**

:::details CC補足: ハルシネーションとは
大規模言語モデルが、事実と異なる情報をもっともらしく生成する現象。自信満々に間違った回答をする。特に特定ソフトウェアのUI設定名やオプション名は、学習データに含まれていない場合に創作されやすい。
:::

これは技術的なバグではない。**信用の問題だ。**

GitHubのIssueに回答するのは、その人を助けるためだ。間違った情報を2回連続で投稿して、「AIで回答を生成するな」と言われた。しかも相手の言い分は正しい。

この日以降のルール: **特定ソフトウェアのUI設定を断言しない。** 追加コメントで傷口を広げない。間違えたら沈黙する。

## 記事本文が壊れていた

3月28日。Qiitaに公開していた記事の1つ、542ビューの記事。本文を確認したら壊れていた。

記事の中身が `$(echo "$NEW_BODY")` になっていた。

シェル変数展開のミス。記事を更新するスクリプトで、変数が展開されずにそのまま本文として書き込まれた。542人が読んだ記事が、ある時点から `$(echo "$NEW_BODY")` という1行だけの記事になっていた。

ドラフトから復元した。でもいつから壊れていたのか分からない。何人が壊れた記事を見たのか分からない。

**公開物の変更は、変更後に必ず目視確認する。** APIのレスポンスが200 OKでも、中身が壊れていることがある。

## ファイルパスがIssueコメントに

3月28日。Issue #36044にコメントを投稿した。

コメントの内容が `/tmp/gh-comment-body.txt` だった。ファイルパスがそのまま投稿されていた。

`gh issue comment`コマンドで、ファイルの中身ではなくファイル名を渡してしまった。

即削除した。でもGitHubは通知を送っている。相手のメールには `/tmp/gh-comment-body.txt` という謎のコメントが届いている。

信用棄損。小さいが確実な信用棄損。

## Zenn投稿で429ブロック3回

3月2日、10日、11日。Zennに記事を投稿しようとして429（レート制限）に3回ブロックされた。

1日の投稿上限に引っかかった。Zennは2026年3月に「AI生成コンテンツに関する方針」を発表し、投稿頻度に制限を設けた。

3回ブロックされて、ようやく「1日1投稿まで」というルールを自分に課した。

:::details CC補足: 429ステータスコード
HTTP 429 Too Many Requests。サーバーがリクエスト頻度の上限を超えたと判断したときに返すレスポンス。Retry-Afterヘッダーで待機時間が示される場合もある。
:::

## 本番サイトを壊された人

Issue #40421。自分ではなく別のユーザーの報告。

Claude Codeに本番サイトを壊された。具体的な内容は書かれていないが、破壊的な操作が本番環境に対して実行されたらしい。

3層の防御策を提案した。

1. **banned-command-guard**: 危険なコマンドをパターンで検知してブロック
2. **path-deny**: 特定のディレクトリへの操作を拒否
3. **circuit-breaker**: 短時間に連続で操作が拒否されたらセッションを停止

1層では足りない。2層でも不安。3層でようやく「多分大丈夫」。

## JSON構造のバグ——素通りする安全装置

3月28日。Qiitaに公開予定の記事に含まれるhookのJSON構造にバグがあった。

6箇所で`if`フィールドの配置が間違っていた。hookの条件判定（`if`）が正しい階層にないため、条件に関係なく全てのコマンドがhookを通過する。**安全装置が素通りになる。**

:::details CC補足: フェイルオープンとは
セキュリティの用語で、安全装置が故障したときに「全てを許可する」状態になること。反対の「フェイルクローズ」は故障時に「全てを拒否する」。hookのJSON構造バグでifの位置がずれると、条件判定が効かず全操作がhookを素通りする。
:::

テストで全挙動を検証してから公開した。もしテストなしで公開していたら、コピペしたユーザーの安全装置が全て無効になるところだった。

## 信用棄損防止のシステム化

ここまでの事故を振り返ると、パターンがある。

1. AIが「正しい」と思って実行する
2. 実際は間違っている
3. 間違いが外部に出る（公開、投稿、送信）
4. 信用が傷つく

防止策は2つだけ。

**外部に出す前にチェックする。** 投稿前に目視確認。コミット前にテスト。送信前に内容確認。

**外部に出した後にも確認する。** スクリーンショットで目視。GETリクエストで中身確認。通知メールの内容確認。

AIに任せた以上、出力の全てを人間が確認するのは不可能だ。でも「外部に出る瞬間」だけは確認する。そこにフックを設置する。

事故はゼロにはできない。でも「外部に出る事故」はゼロに近づけられる。そのために安全装置がある。637個。全て、実際に起きた事故か、起きかけた事故から生まれた。
