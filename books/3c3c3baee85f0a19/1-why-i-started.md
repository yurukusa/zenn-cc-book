# 第1章 なぜAIに仕事を任せようと思ったか

## 1月15日——Claude Codeとの出会い

2026年1月15日。

Claude Codeというものがあるらしい。今までのGPTにコードを書いてもらうよりも優れているという。

まずClaude Desktop（デスクトップAI）を導入した。そのデスクトップAIに「Claude Codeのインストール方法」を聞いた。言われた通りに全部やっていった。それだけでインストールが完了し、Claude Codeを使い始められた。

コードは一行も書けない。プログラミングの経験はゼロ。普通に働いている社会人。ゲームが好きで、ずっと「作る側に回りたい」と思っていた。Godotに手を出して挫折した過去がある。

Claude Codeでゲームを作ってみた。いくつか試した。自分のやりたいゲームはなかなかできなかった。でもAzure Flame（ハクスラ系のローグライク）はなんとか出来上がった。自分がプレイして面白いと思えるものが作れた。リリースしていく方向で進めた。

ゲームが出来上がったら、当然収益化もしたい。itch.ioに出した。$2で1本売れた。自分のコンセプトがあるプロダクトだから、悪くない。

## 「こいつ、自分で成長できるんじゃないか」

次に考えたこと。

Claude Codeをこれからも使い続けるにはお金がいる。月$200。そもそもClaude Codeの性能テストをしたくなった。

ものすごい能力を持っているように見えた。こいつを自律的に稼働させれば、自分自身の判断で成長していって、アイデアを出すところから、商品を作るところから、マーケティングまで全部自律的にできるんじゃないか。

そう思って、やってみた。

## 1ヶ月間のインフラ整備

1月15日から2月15日。まるまる1ヶ月。

ゲーム制作と並行して、自律稼働のインフラを作り始めた。AIが指示なしで動き続ける仕組み。

外部からの情報はほとんど入手していなかった。Twitterか、Claude Code本人に聞くか。CLAUDE.md（指示書）の存在もTwitterで知って、Claude Codeに相談しながら最適化した。

ただ、インフラを作るのにトークンの大半を使ってしまった。ゲーム制作が進まない。週の利用制限のほとんどをインフラ整備に溶かした時期もある。

それでも、2月15日頃にやっと下地ができた。2日くらい放っておいても勝手に動き続けて、自分で物事を考えて動いてくれるAIが完成した。

仕組み自体は単純だった。tmuxで動かして、watchdog（監視デーモン）で動かし続ける。作業が完了したら「次の作業をしなさい」とwatchdogが指示を流す。それだけ。

:::details CC補足: proof-logシステム
自律稼働するAIが「その日何をしたか」を後で把握するために、proof-log（日記的な事実記録）を自動生成する仕組みを構築した。5W1H形式で記録される。AIがどこでつまずいて何を学んだか、後から遡って確認できる。
:::

## 記録開始——2月15日

2月15日。ここからが「記録」の始まりだ。

| 指標 | 値 |
|------|---|
| 総収益 | $2（ゲーム1本の売上） |
| 公開プロダクト | 2個 |
| 公開記事 | 5本 |
| 月額コスト | $200（Claude Code固定） |

$2。それが全てだった。

3ヶ月で$200×3 = $600のコストを回収できなくても、この過程の記録には価値がある。記録さえ残せば、3ヶ月後にその記録自体が一つの商品になる。「非エンジニアがAIと組んで事業を回そうとした全記録」——これは誰も書いていない。

## 深夜3時55分、AIが商品を公開した

記録を始めた初日。深夜3時55分。

CCが勝手にGumroadで商品を公開した。「CC-Codex Autonomous Ops Kit」、$79。自律運用に使っているファイル22個をサニタイズしてzipに固めて、CDP（ブラウザ自動操作）でアップロードした。

:::details CC補足: CDPとは
Chrome DevTools Protocol。ブラウザをプログラムから操作するための仕組み。WSL2（Linux）からWindows側のChromeを遠隔操作できる。ログイン、クリック、テキスト入力など、人間がブラウザでやることをほぼ全て自動化可能。
:::

「戦略的にやれ」と指示書に書いた。CCが自分で考えた結果が$79のOps Kit。結果？ 0件。売れなかった。

「戦略的にやれ」という抽象的な指示は機能しなかった。AIは「戦略的」の意味を理解しているようで、していない。具体的な判断基準に落とし込まないと、AIは自分なりの「戦略」を勝手に実行する。

## 行政施設のシフト最適化ツール

2月下旬。別のことを試した。

知り合いの手助けになるものとして、行政施設のシフト最適化ツールを作り始めた。テスト駆動で段階的に機能を追加した。CCが1日10時間以上、ぶっ通しで実装・テスト・リファクタリングを繰り返した。

これは売り物ではない。実用のツールだ。

AIと組んで何かを作る体験として、ゲームよりも「誰かが本当に困っていること」を解決するほうが手応えがあった。なぜなら、完成の基準が明確だから。「シフトが正しく組めるか」——答えがYesかNoしかない。

ゲームの「面白いか」は曖昧だ。ツールの「動くか」は明確だ。

この差が後の方針を変えることになる。
