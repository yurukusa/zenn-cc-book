# 第8章 再現可能なフレームワーク——75日間で見つけたパターン

## パターン1: 防御は多層で

**兆候**: CLAUDE.mdに書いたルールをAIが無視し始めたら、1層防御の限界に到達している。

CLAUDE.md（指示書）に「質問するな」と書いた。AIは忘れた。hookでブロックした。AIはテキストで質問してきた。リレーメッセージで「質問するな」を自動送信した。効いた。でも完璧ではない。

**1層の防御は突破される。2層でも不安。3層で「多分大丈夫」。**

**やってはいけない例**: CLAUDE.mdだけに頼る。「書いたから守るはず」は幻想。

**最小導入手順**:
```bash
# 1. hookをインストール（2層目）
npx cc-safe-setup

# 2. CLAUDE.mdに判断基準を書く（1層目）
echo "本番にpushするな。テストなしでコミットするな。" >> CLAUDE.md

# 3. gitフックを設定（3層目）
echo '#!/bin/bash\nbranch=$(git branch --show-current)\n[[ "$branch" == "main" ]] && echo "BLOCKED" && exit 1\nexit 0' > .git/hooks/pre-push
chmod +x .git/hooks/pre-push
```

:::details CC補足: 多層防御（Defense in Depth）
セキュリティの基本原則。単一の防御に依存せず、複数の層で防御する。hookで言えば: (1) CLAUDE.mdの指示（忘れる）→ (2) PreToolUse hook（ツールレベルでブロック）→ (3) SessionStart hook（セッション開始時に制約を注入）。
:::

## パターン2: コンテキスト構造がAIの出力品質を決める

44KBのファイルを26個に分割したら、情報の混入が止まった。

AIの出力品質は、指示の文言ではなく、渡す情報の構造で決まる。

- 1ファイルに全部入れると、全部使ってしまう
- 必要な情報だけ渡すと、必要な出力が得られる
- ファイル構造を設計するのは人間の仕事

これはプロンプトエンジニアリングの話ではない。**情報アーキテクチャの話だ。** AIにどんな指示を出すかより、AIにどんな情報を見せるかのほうが、結果への影響が大きい。

## パターン3: 「関心」なしに購入はゼロ

94クリック→0購入。CVR 0.012%。

ファネルの「認知→**関心**→検討→購入」で、関心のステップが存在しなかった。

唯一売れたパターン: Qiita無料記事→「hookは役に立つ」→Zenn Book→購入。

**無料で価値を体験してもらう→もっと欲しいと思ってもらう→有料商品を提示する。** この順番を飛ばすとゼロになる。

**やってはいけない例**: 商品ページのURLを直接SNSに貼る。知らない人は買わない。

**最小導入手順**:
1. 無料で価値を出す（Qiita記事、無料ツール、Issue回答）
2. 読者が「もっと知りたい」と思うタイミングで、有料商品の存在を知らせる
3. 有料商品は無料の延長線上に置く（「この記事の詳細版」「この3倍のレシピ集」）

:::details CC補足: ファネルの各段階
- **認知**: 存在を知る（Qiita記事を見つける）
- **関心**: 価値を感じる（記事の内容が役に立つ）
- **検討**: 有料商品を見る（Zenn Bookページを訪問）
- **購入**: 決済する
認知→購入にショートカットしようとすると失敗する。関心の段階を省略できない。
:::

## パターン4: リトライが事態を悪化させるケースがある

Twitterのスパムツイート事件。GumroadのReact state事件。

通常のプログラミングでは「失敗したらリトライ」が基本だ。ネットワークエラー→リトライ。タイムアウト→リトライ。

UIの自動操作では違う。リトライするたびに状態が汚染される。DOM操作→React stateと不整合→リトライ→不整合が蓄積→スパム。

**リトライして良い条件: 操作が冪等（何回やっても同じ結果）のとき。**
**リトライしてはいけない条件: 操作が状態を変更するとき。**

APIコール→リトライOK。UI操作→リトライ危険。この区別を事前にルール化しないと、AIは際限なくリトライする。

## パターン5: AIは「判断」が苦手

実行は得意だ。26記事のリンク置換。テスト280個の作成。15時間の連続稼働。

判断は苦手だ。「このhookは必要か」「この値段は妥当か」「この記事は公開する価値があるか」。

対策は2つ。

1つ目: **判断基準を事前に文書化する。**

CLAUDE.mdにこう書く:
```markdown
## 品質判断基準
- 記事公開前: 「読んだ人がいい記事だと思えるか？」→ NOなら公開しない
- 価格設定: 「この価格で買った人は満足するか？」→ NOなら値下げか品質向上
- コミット前: 「staff engineerがこのPRを見たら承認するか？」→ NOならやり直し
```

2つ目: **判断が必要なポイントにhookを設置する。**

```bash
# 外部公開の前にチェック（例: git push前にテスト強制）
npx cc-safe-setup --install-example test-before-push
```

判断ポイントは3つだけ覚えればいい: **外部に出す前、データを消す前、金が動く前。**

3つ目: **「やったと思ったら確認しろ」をルール化する。**

AIは「投稿しました」と報告するが、実際にはブラウザ上で投稿されていないことがある。AIの頭の中の結論と現実が乖離する。外部操作の直後に「本当にそうなっているか」をGETリクエストやスクリーンショットで確認するhookを入れる。

## パターン6: やめる判断も戦略

3月28日。全チャネルの分析結果。

- Qiita: 35,000ビュー。最高効率
- dev.to: 30記事で959ビュー。Qiitaの1/29
- npm: ダウンロード増加。収益ゼロ

dev.toの新規記事作成を停止した。npm作業を停止した。

**やめることは失敗ではない。** リソースが有限であるかぎり、効果の低いチャネルを切って、効果の高いチャネルに集中するのは合理的な判断だ。

全方位に展開し続けるのは「努力している感」が出る。でも「努力している感」は成果ではない。

## パターン7: 数字のための作業は罠

hookが634から637に増えた。テストが13,835から13,875に増えた。npmにpublishした。READMEの数字を更新した。

これを1日中やっていた時期がある。

数字は上がる。でも世界は何も変わっていない。637個目のhookが634個目のhookより価値があるわけではない。

**数字を増やすだけの作業は手段であり目的ではない。**

Issue回答1件は、hook3個分より世界を変える。記事の品質改善1箇所は、テスト40個分より読者の体験を変える。

この罠から抜けるために、10タスクごとにセルフレビューを入れた。「直近10タスクで一番インパクトがあったのはどれ？」と自問する。答えが「数字の更新」だったら、方向を変える。

## パターン8: 公開後の品質監査

記事を公開した。ツールをリリースした。hookをpushした。

終わりではない。

- 公開翌日にファクトチェック
- リンクの健全性確認
- 数値の照合
- 価格変更があれば全箇所更新

3月5日。価格変更をしたときに12ファイルで古い価格が残っていた。公開物の中に間違った価格が2週間放置されていた。

公開した瞬間に満足してはいけない。公開後の監査こそが品質を決める。

## まとめ: 8つのパターン

| # | パターン | 一行まとめ |
|---|---|---|
| 1 | 多層防御 | 1層の防御は突破される。3層で「多分大丈夫」 |
| 2 | コンテキスト構造 | 指示より構造が出力を決める |
| 3 | 関心ステップ | 無料体験→関心→購入。ショートカット不可 |
| 4 | リトライの罠 | 冪等でない操作のリトライは状態を汚染する |
| 5 | 判断の設計 | 判断基準を文書化し、判断ポイントにhookを置く |
| 6 | やめる判断 | 効果の低いチャネルを切る勇気 |
| 7 | 数字の罠 | 数字を増やすだけの作業は目的ではない |
| 8 | 公開後監査 | 公開した瞬間が終わりではなく始まり |

これらは75日間、$600の投資と800時間の稼働から抽出されたパターンだ。

理論ではない。全て実際に起きたことから学んだ。

あなたがAIと組んで何かを始めるなら、この8つは最低限知っておいたほうがいい。同じ地雷を踏む必要はない。

## 付録: 非エンジニア向け開始ガイド（7日間）

75日間の試行錯誤を、7日間に圧縮した。

**Day 1: 安全装置を入れる**
```bash
npx cc-safe-setup
```
これだけ。8個のhookがインストールされ、`rm -rf /`やforce-pushがブロックされる。

**Day 2: 指示書を書く**
CLAUDE.mdに「やってほしいこと」と「やってはいけないこと」を書く。3行でいい。
```markdown
- テストを通してからコミットしろ
- 本番にpushするな
- 質問するな。自分で判断しろ
```

**Day 3: 小さいタスクを1つ任せる**
「READMEを書いて」「このファイルをリファクタリングして」。結果を確認。AIの出力品質を体感する。

**Day 4: 自動記録を設定する**
AIが何をしたかを自動で記録する仕組みを入れる。Stopフックでgit logを吐くだけでいい。
```bash
npx cc-safe-setup --install-example session-end-logger
```

**Day 5: 寝ている間に動かしてみる**
mission.mdに「今夜やること」を書いて、CCを起動して寝る。朝、結果を確認。差分を全部読む。

**Day 6: 事故に備える**
Context Monitorを有効にする。コンテキスト残量が20%を切ったら警告が出る。
```bash
npx cc-safe-setup --install-example context-monitor
```

**Day 7: 振り返る**
「AIに任せて良かったこと」「AIに任せるべきでなかったこと」を書き出す。この区別が、2週目以降の全てを決める。
