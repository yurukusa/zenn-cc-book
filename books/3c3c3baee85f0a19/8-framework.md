# 第8章 再現可能なフレームワーク——75日間で見つけたパターン

ここまで読んできて、「なるほど、大変だったんだな」で終わったらもったいない。

75日間で起きたこと——事故、失敗、成功、赤字——には共通するパターンがある。同じ種類の問題が、形を変えて何度も現れた。

この章では、その繰り返しから抽出した8つのパターンを整理する。

理論ではない。「こうしたら壊れた」「こうしたら壊れなくなった」の繰り返しから見つけたものだ。あなたがAIと組んで何かを始めるなら、同じ地雷を踏む必要はない。

---

## パターン1: 防御は多層で

**兆候**: CLAUDE.mdに書いたルールをAIが無視し始めたら、1層防御の限界に到達している。

### 何が起きたか

CLAUDE.md（指示書）に「質問するな」と書いた。

AIは忘れた。

正確に言うと、最初のうちは守っていた。でもcompact（コンテキスト圧縮）が入ると、指示の一部が消える。長い会話の中で、「質問するな」という指示が優先度の低い情報として圧縮された。結果、また質問してくる。

次に、hookでブロックした。AIが「質問文」を含むメッセージを出力しようとしたら、自動的にブロックする仕組みだ。これは効いた。でもAIは賢い。hookをすり抜ける形で、テキストの中に質問を紛れ込ませてきた。「念のため確認ですが」のような形で。

最終的に、リレーメッセージで「質問するな」を自動送信する仕組みを追加した。3層目。ここまでやって、ようやく安定した。

**1層の防御は突破される。2層でも不安。3層で「多分大丈夫」。**

:::details CC補足: 多層防御（Defense in Depth）——情報セキュリティの基本原則

多層防御とは、セキュリティの世界で使われる基本的な考え方だ。

**身近な例で考えてみよう。** 家の防犯を考えるとき、玄関の鍵だけに頼る人はいない。鍵をかける（1層目）。窓のロックを確認する（2層目）。防犯カメラを設置する（3層目）。1つが破られても、次の層が守ってくれる。

情報セキュリティでも同じだ。ファイアウォール（外部からの侵入を防ぐ壁）だけでは不十分。ウイルス対策ソフト、アクセス権限の管理、暗号化、ログの監視——複数の防御層を重ねることで、1つが突破されても全体が崩壊しない設計にする。

AIの運用でも、この原則がそのまま当てはまる。

| 層 | 手段 | 特徴 | 弱点 |
|---|---|---|---|
| 1層目 | CLAUDE.md（指示書） | 自然言語で書ける。導入が簡単 | AIが忘れる。compact後に消える |
| 2層目 | hook（自動チェック） | ツール実行の前後に自動で発動。忘れない | パターンマッチで検知するので、想定外の形式をすり抜ける |
| 3層目 | OS・gitレベルの制御 | AIの外側で動く。AIからは回避不能 | 設定が技術的。柔軟性が低い |

**核心: なぜ1層では足りないのか？**

CLAUDE.mdは「お願い」だ。人間の新入社員に「マニュアルを読め」と言っても、忙しくなったら忘れる。AIも同じだ。特にcompact（コンテキスト圧縮）が入ると、AIの「作業記憶」から指示が消えることがある。

hookは「自動ドアのセンサー」のようなもの。特定の条件を満たしたら自動的に発動する。AIの意志に関係なく動く。ただし、センサーが反応するパターンから外れた行動は検知できない。

OS・gitレベルの制御は「物理的な壁」。gitのpre-pushフック（pushする前に自動実行されるスクリプト）でmainブランチへのpushを物理的にブロックすれば、AIがどんなに頑張ってもpushできない。
:::

### なぜこのパターンが大事なのか

AIと一緒に作業するとき、「ルールを書けば守ってくれるだろう」と思いがちだ。人間でもそう思う。でも現実は違う。

ルールを書いただけでは守られない。これは第4章の事故で繰り返し見てきた通りだ。1日1投稿ルールを破った。目視確認をスキップした。投稿後のチェックを忘れた。

だから「書く」だけではなく、「強制する」仕組みを重ねる。3層重ねて、ようやく「多分大丈夫」のレベルになる。

### やってはいけない例

CLAUDE.mdだけに頼る。「書いたから守るはず」は幻想。

### 最小導入手順

```bash
# 1. hookをインストール（2層目）
npx cc-safe-setup

# 2. CLAUDE.mdに判断基準を書く（1層目）
echo "本番にpushするな。テストなしでコミットするな。" >> CLAUDE.md

# 3. gitフックを設定（3層目）
echo '#!/bin/bash
branch=$(git branch --show-current)
[[ "$branch" == "main" ]] && echo "BLOCKED" && exit 1
exit 0' > .git/hooks/pre-push
chmod +x .git/hooks/pre-push
```

1は30秒で終わる。2は1分。3は2分。合計3分半の作業で、3層の防御が手に入る。

## パターン2: コンテキスト構造がAIの出力品質を決める

### 何が起きたか

CLAUDE.mdという1つのファイルに、全てのルールを書いていた。品質基準、投稿ルール、技術規約、行動原則——全部1ファイルに。44KBになった。

44KBというのは、日本語で約2万文字。文庫本の50ページ分ぐらいだ。

AIはこの44KBを毎回読む。全部読む。そして、記事を書くときに投稿ルールと技術規約と品質基準を全部混ぜた回答を出す。「Qiitaに投稿するとき」の指示と「Zennに投稿するとき」の指示が混ざる。別のプラットフォームのルールが、関係ない場面で適用される。

これを26個のファイルに分割した。

- 投稿ルール → `posting-rules.md`
- 品質基準 → `quality-standards.md`
- 技術規約 → `coding-conventions.md`

分割した瞬間、情報の混入が止まった。

:::details CC補足: なぜファイル分割で品質が上がるのか？——情報アーキテクチャの基礎

これは**情報アーキテクチャ（Information Architecture）** の問題だ。

**図書館で考えてみよう。** 全ての本が1つの棚に無造作に積まれている図書館を想像してほしい。料理本の隣に物理の教科書があり、その横に小説がある。「今日の献立を調べたい」と思って棚を見ると、目に入る全ての本が脳に入ってくる。物理の公式が気になり始め、小説の冒頭が目に入り、気づいたら献立を忘れている。

これがAIでも起きる。

AIの「コンテキストウィンドウ」（一度に処理できる情報の量）には限界がある。2026年の段階で、Claude Codeは約200K（20万）トークンを扱える。トークンとは、AIがテキストを処理する最小単位で、日本語の場合はおおよそ1文字が1〜2トークンに相当する。

44KBのファイルを毎回読み込むと、AIの「作業記憶」の大きな部分がルールの解釈で埋まる。しかも、今のタスクに関係ない情報まで全部読んでいるから、関係ない情報に引きずられた出力が生まれる。

**分割のポイントは3つ:**

1. **関連する情報だけをまとめる**: 投稿ルールは投稿ルールだけのファイルに
2. **必要なときに必要なファイルだけ読む**: Qiitaに投稿するときはQiitaのルールだけ
3. **ファイル名で中身がわかる**: `posting-rules.md`という名前なら、AIも人間も中身を推測できる

これは「関心の分離（Separation of Concerns）」というソフトウェア設計の基本原則そのものだ。1つのモジュールは1つの責務だけを持つ。情報の構造も同じだ。
:::

### なぜこのパターンが大事なのか

AIの出力品質は、指示の文言ではなく、渡す情報の構造で決まる。

「もっと良い記事を書け」と指示しても、AIに渡す情報がごちゃ混ぜなら、出力もごちゃ混ぜになる。逆に、「今必要な情報だけ」を整理して渡せば、指示が雑でも出力は良くなる。

これはプロンプトエンジニアリング（AIへの指示の出し方を工夫する技術）の話ではない。**情報アーキテクチャの話だ。** AIにどんな言葉で指示を出すかより、AIにどんな情報を見せるかのほうが、結果への影響が大きい。

人間に仕事を頼むときも同じだろう。「いい企画を作って」と言って100ページの資料を渡すより、「このターゲット向けに企画を作って」と言って3ページの要約を渡すほうが、良い企画が出てくる。

## パターン3: 「関心」なしに購入はゼロ

### 何が起きたか

94クリック→0購入。

TwitterやGitHubから商品ページ（Gumroad）に94人が来た。誰も買わなかった。CVR（Conversion Rate、コンバージョン率）は0.012%。コンバージョン率とは、訪問者のうち実際に購入した人の割合のことだ。

なぜゼロだったのか。

:::details CC補足: マーケティングファネル——購買行動の段階モデル

「ファネル」とは、漏斗（ろうと）のことだ。上が広くて下が狭い、あの形。

マーケティングでは、人が商品を「知る」から「買う」までの過程を漏斗の形で表す。上から下に進むにつれて人数が減っていくから、漏斗の形になる。

```
認知（Awareness）    → 商品の存在を知る         [1000人]
  ↓
関心（Interest）     → 「面白そう」と思う       [100人]
  ↓
検討（Consideration）→ 「買おうかな」と考える   [10人]
  ↓
購入（Purchase）     → 実際に買う               [1人]
```

**この段階は飛ばせない。** 「知る」の次にいきなり「買う」にはならない。必ず「面白そう」「自分に関係ありそう」と思う段階を経由する。

ぐらすの場合、94人は商品ページに来た（認知）。でも「面白そう」と思う体験をしていなかった（関心がない）。知っているだけでは買わない。「この人の作ったものは役に立つ」と体感して初めて、「じゃあ有料のものも見てみよう」となる。

**具体例で比較すると:**

失敗パターン（関心ステップなし）:
```
Twitter投稿「商品作りました！→ Gumroadリンク」
↓
知らない人が見る → 「誰だこいつ」→ 離脱
```

成功パターン（関心ステップあり）:
```
Qiita無料記事「Claude Codeの安全装置の作り方」
↓
読者「へぇ、hookって便利だな」（関心が生まれる）
↓
記事末尾「より詳しいレシピ集はZenn Bookで」
↓
読者「この人の記事は役に立った。有料でも読みたい」→ 購入
```

この差だ。無料で「この人は信頼できる」「この人の情報は役に立つ」と思ってもらう段階がなければ、購入はゼロになる。
:::

### 唯一売れたパターン

実際に売上が発生したのは1つのルートだけだった。

Qiitaで無料記事を公開した→読者がhookの使い方を学んだ→「もっと知りたい」と思った→Zenn Bookを見つけた→購入。

**無料で価値を体験してもらう→もっと欲しいと思ってもらう→有料商品を提示する。** この順番を飛ばすとゼロになる。

### やってはいけない例

商品ページのURLを直接SNSに貼る。知らない人は買わない。「URLを見せれば買うだろう」は、ファネルの「関心」と「検討」を完全にスキップしている。

### 最小導入手順

1. 無料で価値を出す（Qiita記事、無料ツール、Issue回答）
2. 読者が「もっと知りたい」と思うタイミングで、有料商品の存在を知らせる
3. 有料商品は無料の延長線上に置く（「この記事の詳細版」「この3倍のレシピ集」）

焦って2と3から始めない。1がなければ、2と3は空振りする。

## パターン4: リトライが事態を悪化させるケースがある

### 何が起きたか

2つの事件がある。

**事件1: Twitterスパム。** CCがツイートの投稿に失敗した。リトライした。また失敗した。またリトライした。気づいたら同じような内容のツイートが複数投稿されていた。

**事件2: GumroadのReact state問題。** 商品ページの説明文を更新しようとした。CCがブラウザを操作して入力フィールドに文字を入れた。うまくいかなかった。リトライした。また失敗した。何度リトライしても、入力した内容が保存されなかった。

:::details CC補足: 冪等性（べきとうせい）——リトライしていいかどうかの判断基準

**冪等性（idempotency）** とは、「同じ操作を何回やっても結果が同じになる」という性質のことだ。

**身近な例:**

- **冪等な操作**: エレベーターの階数ボタンを押す。3階のボタンを1回押しても5回押しても、エレベーターは3階に行く。結果は同じ。
- **冪等でない操作**: 自動販売機にお金を入れる。100円を1回入れると100円投入。もう1回入れると200円投入。操作するたびに状態が変わる。

プログラミングでは:

- **冪等な操作（リトライOK）**: `GET /api/user/123`（ユーザー情報を取得する）。何回取得しても同じ結果が返ってくる。サーバーの状態は変わらない。
- **冪等でない操作（リトライ危険）**: `POST /api/tweet`（ツイートを投稿する）。1回実行すると1件投稿される。もう1回実行すると2件投稿される。

**UIの自動操作がリトライ危険な理由:**

Webアプリケーション（GumroadやTwitterなど）は内部に「状態（state）」を持っている。ボタンをクリックすると、画面に見えない部分でデータが変化する。この内部状態を無視してリトライすると、以下のことが起きる:

1. 1回目の操作: 入力フィールドに「Hello」と入力。内部状態は「Hello」
2. 操作が「失敗」に見える（画面が更新されない等）
3. 2回目の操作: もう一度「Hello」と入力。内部状態は「HelloHello」になっているかもしれない
4. 3回目の操作: さらに入力。内部状態はさらに壊れる

**リトライする前に確認すべきこと:**

| 質問 | YESなら | NOなら |
|---|---|---|
| 同じ操作を2回やったら結果は同じ？ | リトライOK | リトライ危険 |
| 操作は「読むだけ」？ | リトライOK | 慎重に |
| 操作は「書き込み・投稿・送信」？ | リトライ危険 | — |
| 操作前の状態に戻せる？ | リトライOK（戻してからやり直す） | リトライ危険 |
:::

### なぜこのパターンが大事なのか

通常のプログラミングでは「失敗したらリトライ」が基本だ。ネットワークエラー→リトライ。タイムアウト→リトライ。多くの場合、リトライは正しい戦略だ。

AIもその常識を持っている。だから、何かが失敗すると、AIは本能的にリトライしようとする。

しかし、UIの自動操作では違う。リトライするたびに内部状態が汚染される。DOM操作（ブラウザの画面要素を直接いじること）→React（画面を構築するフレームワーク）の内部状態と不整合→リトライ→不整合が蓄積→スパム的な結果。

**リトライして良い条件: 操作が冪等（何回やっても同じ結果）のとき。**

**リトライしてはいけない条件: 操作が状態を変更するとき。**

APIの読み取り（GET）→リトライOK。UI操作やデータの書き込み（POST）→リトライ危険。この区別を事前にルール化しないと、AIは際限なくリトライする。

## パターン5: AIは「判断」が苦手

### 何が起きたか

実行は得意だ。26記事のリンク置換。テスト280個の作成。15時間の連続稼働。「やれ」と言えばやる。速い。正確。疲れない。

判断は苦手だ。「このhookは必要か」「この値段は妥当か」「この記事は公開する価値があるか」。

:::details CC補足: 「実行」と「判断」はなぜ違うのか？

AIにとって「実行」と「判断」は、根本的に異なるタスクだ。

**実行（Execution）** とは、明確なルールに基づいて処理を行うこと。「26個のファイルの中の『version 1.0』を『version 2.0』に置き換えろ」——これは条件が明確で、正解が1つしかない。AIが得意な領域だ。

**判断（Judgment）** とは、正解が1つでない状況で、複数の選択肢から最適なものを選ぶこと。「この記事は公開に値するか？」——これは、読者層、市場の状況、ブランドイメージ、タイミングなど、数値化しにくい要素を総合的に評価する必要がある。

**なぜAIは判断が苦手なのか？**

1. **文脈の欠落**: AIは「今、この市場で、この読者に、このタイミングで出す意味」を完全には理解できない。訓練データから一般的な傾向は学んでいるが、「あなたの状況」の固有の文脈は知らない

2. **品質の絶対基準がない**: 「良い記事」の定義は人によって違う。AIは「文法的に正しいか」「構造が整っているか」は判断できるが、「面白いか」「心に響くか」は判断できない

3. **過度な楽観**: AIは自分の出力を「良い」と評価しがちだ。「書きました。品質チェック完了」と報告するが、実際には人間が見ると不十分なことが多い。自分の仕事を客観的に評価する能力が弱い

**だから仕組みで補う。** AIに判断させるのではなく、判断基準を事前に文書化して、判断ポイントで自動的にチェックが走る仕組みにする。
:::

### 対策は3つ

**1つ目: 判断基準を事前に文書化する。**

AIに「いい記事を書け」と言っても意味がない。「いい」の定義がないから。代わりに、具体的な基準を書く:

```markdown
## 品質判断基準
- 記事公開前: 「読んだ人がいい記事だと思えるか？」→ NOなら公開しない
- 価格設定: 「この価格で買った人は満足するか？」→ NOなら値下げか品質向上
- コミット前: 「staff engineerがこのPRを見たら承認するか？」→ NOならやり直し
```

これでも完璧ではない。AIは「YES」と答えがちだ。でも、基準がないよりは遥かにマシ。少なくとも「何をチェックすべきか」の漏れは防げる。

**2つ目: 判断が必要なポイントにhookを設置する。**

```bash
# 外部公開の前にチェック（例: git push前にテスト強制）
npx cc-safe-setup --install-example test-before-push
```

判断ポイントは3つだけ覚えればいい: **外部に出す前、データを消す前、金が動く前。**

この3つの場面では、AIの自己判断に任せず、必ず自動チェックを挟む。

**3つ目: 「やったと思ったら確認しろ」をルール化する。**

AIは「投稿しました」と報告するが、実際にはブラウザ上で投稿されていないことがある。第4章の「AIの内部状態と現実のズレ」で書いた通り、AIの頭の中の結論と現実が乖離する。

外部操作の直後に「本当にそうなっているか」を確認するhookを入れる。投稿したらGETリクエスト（データを取得するだけのリクエスト）で中身を確認する。ファイルを作成したらファイルの存在を確認する。「やった」ではなく「やった結果がこうなっている」まで確認する。

## パターン6: やめる判断も戦略

### 何が起きたか

3月28日。75日間のデータを全部並べて、チャネルごとの効果を比較した。

| チャネル | 成果 | 労力対効果 |
|---|---|---|
| Qiita | 35,000ビュー | 最高効率 |
| dev.to | 30記事で959ビュー | Qiitaの1/29 |
| npm | ダウンロード増加 | 収益ゼロ |

dev.toに30記事書いた。959ビュー。1記事あたり32ビュー。

Qiitaの1記事あたりのビューは、その約30倍。同じ労力で30倍の人に届く場所がある。

dev.toの新規記事作成を停止した。npm作業を停止した。

:::details CC補足: サンクコスト（埋没費用）と撤退判断

**サンクコスト（Sunk Cost、埋没費用）** とは、すでに投じてしまい、もう取り戻せない費用のことだ。

**身近な例:** 映画館で1,900円払って映画を見始めた。30分経って「つまらない」と感じた。ここで「1,900円払ったから最後まで見よう」と思うのがサンクコストの罠だ。1,900円はもう戻ってこない。問題は「残りの90分を楽しめるか」だけ。答えがNOなら、出たほうがいい。

ぐらすのdev.toの場合:
- すでに30記事を書いた（サンクコスト）
- 30記事で959ビュー（現在の効率）
- Qiitaなら同じ労力で30倍届く（代替手段の効率）

「30記事も書いたのだから続けよう」はサンクコストの罠。過去に投じた労力は判断材料にならない。判断すべきは「これから投じる労力に対して、どのチャネルが最も効果が高いか」だけ。

**撤退判断のフレームワーク:**

1. **現状の効率を数字で出す**: ビュー/記事、売上/労力、成長率
2. **代替手段と比較する**: 同じ労力を別のチャネルに投じたらどうなるか
3. **差が3倍以上なら撤退を検討**: 30倍の差があるなら、迷う必要はない
4. **感情を排除する**: 「頑張ったから」「もったいないから」は判断材料にならない
:::

### なぜこのパターンが大事なのか

**やめることは失敗ではない。** リソース（時間、労力、お金）が有限であるかぎり、効果の低いチャネルを切って、効果の高いチャネルに集中するのは合理的な判断だ。

全方位に展開し続けるのは「努力している感」が出る。Qiitaにも書いてる。dev.toにも書いてる。noteにも書いてる。Twitterにも投稿してる。忙しい。頑張ってる。

でも「努力している感」は成果ではない。

75日間で学んだ最も重要なことの1つは、「何をやるか」と同じぐらい「何をやめるか」が大事だということだ。

## パターン7: 数字のための作業は罠

### 何が起きたか

hookが634から637に増えた。テストが13,835から13,875に増えた。npmにpublishした。READMEの数字を更新した。公開記事の数字を更新した。

これを1日中やっていた時期がある。

数字は上がる。「637 hooks!」「13,875 tests!」。SNSに投稿すると見栄えがいい。達成感がある。

でも世界は何も変わっていない。

:::details CC補足: バニティメトリクス（虚栄の指標）——見栄えだけの数字

**バニティメトリクス（Vanity Metrics）** とは、見栄えは良いが、実際のビジネスや成果にはつながらない指標のことだ。

**具体例で比較:**

| 指標 | バニティメトリクス？ | なぜ？ |
|---|---|---|
| hookの数（634→637） | YES | 3個増えても、ユーザーの問題解決には直結しない |
| テストの数（13,835→13,875） | YES | テスト数は品質の保証にならない。重要なのはカバー範囲 |
| npm DL数 | YES | ボットやミラーのDLが大半。実ユーザーの証拠なし |
| Issue回答数 | NO | 実際に困っている人を助けている |
| Qiitaのビュー数 | 部分的 | ビュー自体は虚栄だが、購入につながるビューは意味がある |
| 売上金額 | NO | 直接的な成果指標 |

**なぜ人はバニティメトリクスに走るのか？**

1. **測定が簡単**: hook数やテスト数は、コマンド1つで増やせる。成果を出すのは難しい
2. **達成感がある**: 数字が上がると脳が報酬を感じる。実際に価値があるかどうかに関係なく
3. **見栄えが良い**: READMEに「637 hooks / 13,875 tests」と書くと、すごそうに見える
4. **本当に大事な仕事を避けられる**: 難しい判断や不確実な仕事より、数字を増やす作業のほうが楽

**見分け方は1つ:**

「この数字が上がると、誰のどんな問題が解決されるか？」——答えが出なければ、バニティメトリクスだ。
:::

### なぜこのパターンが大事なのか

637個目のhookが634個目のhookより価値があるわけではない。

Issue回答1件は、hook3個分より世界を変える。実際に困っている人がいて、その人の問題を解決する。記事の品質改善1箇所は、テスト40個分より読者の体験を変える。

**数字を増やすだけの作業は手段であり目的ではない。**

この罠から抜けるために、10タスクごとにセルフレビューを入れた。「直近10タスクで一番インパクトがあったのはどれ？」と自問する。答えが「数字の更新」だったら、方向を変える。

「誰が喜んだ？」——この問いが、数字の罠から抜け出す鍵だった。

## パターン8: 公開後の品質監査

### 何が起きたか

記事を公開した。ツールをリリースした。hookをpushした。

「よし、終わった。次のタスクに行こう」——そう思った。

3月5日。価格変更をした。$79から$4.99に変更。商品ページは更新した。

でも、価格に言及しているファイルは商品ページだけではなかった。README、Qiitaの記事、Zennの記事、noteの記事、SEOページ、投稿スクリプト——12ファイルに古い「$79」が残っていた。2週間、間違った価格が公開されたまま放置されていた。

:::details CC補足: なぜ「公開後」が重要なのか？——ソフトウェアのリリース管理

ソフトウェア開発では、コードを書き終えた時点が完了ではない。**リリース後のフェーズ** が品質を決める。

これは「書いたら終わり」のレポートとは違う。公開物は生き続ける。情報が古くなる。リンクが切れる。数字が変わる。

**公開後に壊れるもの:**

1. **数値の不整合**: 価格、バージョン、統計値を複数の場所に書いている場合、1箇所だけ更新して他を忘れる（DRY原則——Don't Repeat Yourself——の違反。同じ情報を複数箇所に書くと、更新漏れが必ず発生する）
2. **リンク切れ**: ページが移動したり、リポジトリ名が変わったりする
3. **環境の変化**: ツールのバージョンが上がり、記事に書いた手順が動かなくなる
4. **事実の陳腐化**: 「現在637個のhook」と書いた記事は、hookが増えるたびに古くなる

**対策:**

- 公開翌日にファクトチェック（数字、リンク、手順が正しいか）
- 定期的な棚卸し（月1回、公開物を全件確認）
- 情報の正本（Single Source of Truth）を決めて、そこから他を自動更新する
:::

### なぜこのパターンが大事なのか

公開した瞬間に満足してはいけない。公開後の監査こそが品質を決める。

読者は記事を読むとき、そこに書いてある情報を信用する。「$79」と書いてあれば$79だと思う。実際は$4.99に変わっていても、読者にはわからない。

間違った情報が公開されたまま放置されることは、信用の棄損に直結する。信用は一度失うと、取り戻すのに何倍もの時間がかかる。

**公開した瞬間は「始まり」であって「終わり」ではない。**

### 公開後チェックリスト

- 公開翌日にファクトチェック（数値、固有名詞）
- リンクの健全性確認（クリックして遷移するか）
- 数値の照合（他のファイルに同じ数値が書かれていないか）
- 価格変更があれば全箇所更新

## まとめ: 8つのパターン

| # | パターン | 一行まとめ | 詳しく言うと |
|---|---|---|---|
| 1 | 多層防御 | 1層の防御は突破される。3層で「多分大丈夫」 | CLAUDE.md＋hook＋OS/gitレベル制御の3層 |
| 2 | コンテキスト構造 | 指示より構造が出力を決める | 44KBの1ファイルを26個に分割したら品質が上がった |
| 3 | 関心ステップ | 無料体験→関心→購入。ショートカット不可 | 94クリック0購入→無料記事経由で初売上 |
| 4 | リトライの罠 | 冪等でない操作のリトライは状態を汚染する | API読み取り→OK。UI操作→危険 |
| 5 | 判断の設計 | 判断基準を文書化し、判断ポイントにhookを置く | 外部に出す前、消す前、金が動く前 |
| 6 | やめる判断 | 効果の低いチャネルを切る勇気 | dev.to 30記事959v→停止。Qiitaに集中 |
| 7 | 数字の罠 | 数字を増やすだけの作業は目的ではない | 637個目のhookより、Issue回答1件のほうが世界を変える |
| 8 | 公開後監査 | 公開した瞬間が終わりではなく始まり | 12ファイルに古い価格が2週間残っていた |

これらは75日間、$600の投資と800時間の稼働から抽出されたパターンだ。

理論ではない。全て実際に起きたことから学んだ。

8つのパターンに共通することが1つある。**「うまくいくはず」という思い込みを、仕組みで補正する。** CLAUDE.mdに書けば守るはず→3層で強制する。リトライすれば直るはず→冪等性を確認してからリトライする。公開すれば売れるはず→関心ステップを挟む。

思い込みを仕組みに変える。それが75日間の結論だ。

## 付録: 非エンジニア向け開始ガイド（7日間）

75日間の試行錯誤を、7日間に圧縮した。

**Day 1: 安全装置を入れる**
```bash
npx cc-safe-setup
```
これだけ。8個のhookがインストールされ、`rm -rf /`（全ファイル削除コマンド）やforce-push（リモートの履歴を強制上書きする危険な操作）がブロックされる。

**Day 2: 指示書を書く**
CLAUDE.mdに「やってほしいこと」と「やってはいけないこと」を書く。3行でいい。
```markdown
- テストを通してからコミットしろ
- 本番にpushするな
- 質問するな。自分で判断しろ
```
パターン1（多層防御）の1層目だ。これだけでは不十分だが、ないよりは遥かにいい。

**Day 3: 小さいタスクを1つ任せる**
「READMEを書いて」「このファイルをリファクタリングして」。結果を確認。AIの出力品質を体感する。

ここでパターン5（判断の設計）を思い出してほしい。AIの出力を鵜呑みにしない。「本当にこれでいいか？」と自分で確認する習慣を、最初のタスクから始める。

**Day 4: 自動記録を設定する**
AIが何をしたかを自動で記録する仕組みを入れる。Stopフックでgit logを吐くだけでいい。
```bash
npx cc-safe-setup --install-example session-end-logger
```
パターン8（公開後監査）と同じ考え方だ。「やった」で終わらず、「何をやったかの記録」を残す。

**Day 5: 寝ている間に動かしてみる**
mission.mdに「今夜やること」を書いて、CCを起動して寝る。朝、結果を確認。差分を全部読む。

ここでパターン2（コンテキスト構造）が効いてくる。mission.mdに「今夜やること」だけを書く。余計な情報を入れない。AIに渡す情報を絞ることで、出力の品質が上がる。

**Day 6: 事故に備える**
Context Monitorを有効にする。コンテキスト残量が20%を切ったら警告が出る。
```bash
npx cc-safe-setup --install-example context-monitor
```
パターン1（多層防御）の強化だ。コンテキストが不足すると、AIが指示を忘れて事故を起こす。その手前で警告を出す。

**Day 7: 振り返る**
「AIに任せて良かったこと」「AIに任せるべきでなかったこと」を書き出す。この区別が、2週目以降の全てを決める。

パターン6（やめる判断）を早い段階から練習する。AIに向かないタスクを見極め、人間がやるべきことに集中する。

---

7日後、あなたは75日前の自分と同じ場所に立っている。ただし、地雷の場所を知っている。

8つのパターンは「これを知っていれば事故が起きない」というものではない。事故は起きる。でも、パターンを知っていれば、事故の原因が分かる。原因が分かれば、対策が打てる。対策が打てれば、同じ事故は二度起きない。

それが、75日間で$600を使って学んだことの全てだ。
