---
title: "スキルのテスト——発火しない問題を防ぐ"
---

スキルを作ったら、3つの観点でテストする。

## 1. トリガーテスト

最も重要なテスト。スキルが正しいタイミングで発火するか。

### テスト方法

Claude Codeに以下の種類の入力を試す:

**発火すべき入力:**
```
「デプロイして」
「deploy to production」
「本番にリリースしたい」
```

**発火すべきでない入力:**
```
「デプロイの概念を教えて」
「deploymentとは何ですか」
「テストを書いて」
```

### よくある失敗

1. **triggerが狭すぎる**: 「deploy」だけ書いて「デプロイ」で発火しない
2. **triggerが広すぎる**: 「code」と書いて、あらゆるコーディング作業で発火する
3. **パラフレーズで発火しない**: 「リリース」「出荷」「公開」でも発火すべき

### 改善

```yaml
trigger: When user mentions deploying, releasing, shipping, publishing to production, or says "デプロイ", "リリース", "公開", "出荷"
```

具体的な動詞を複数書く。日本語と英語の両方を入れる。

## 2. 機能テスト

スキルが正しい出力を出すか。

### テスト方法

1. スキルを発火させる
2. 実行された手順を確認する
3. 期待通りの結果が得られたか検証する

### チェックポイント

- 手順が正しい順序で実行されたか
- references/のファイルが適切に参照されたか
- エラー時の挙動が想定通りか

## 3. パフォーマンス比較

スキルあり/なしでの効率差を計測する。

### 計測項目

| | スキルなし | スキルあり |
|---|---|---|
| ツール呼び出し回数 | ? | ? |
| 完了までの時間 | ? | ? |
| 手戻り回数 | ? | ? |

スキルを入れたことでツール呼び出しが減り、手戻りが減れば、そのスキルは機能している。

### 計測方法

```bash
# セッション内のツール呼び出し数を確認
# Claude Codeのdebugログから取得可能
```

## テスト結果の記録

テスト結果はSKILL.mdの先頭コメントに記録しておくと、後で参照できる。

```yaml
---
name: deploy-helper
# Last tested: 2026-03-28
# Trigger: 5/5 patterns fired correctly
# Performance: 3 tool calls saved vs no skill
---
```
