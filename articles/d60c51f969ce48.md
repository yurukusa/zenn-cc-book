4 月 28 日 に 「Claude Code に作業させてたら `.git` が 144GB に膨らんで WSL が起動しなくなった話」 という記事を書いた。 あの時は、 並列に動いていた CC-loop と Codex のサブエージェントが同じリポジトリで `git gc` と `git repack` をぶつけ合って、 `.git/objects/pack/tmp_pack_*` が 1,350 個まで積み上がっていた。 合計 144GB だった。 ext4.vhdx が 208GB まで膨らんで Ubuntu がまともに起動しなくなった。

あの記事の末尾で、 私は「今後こうしたい」 という対策の候補を 3 つ書いた。 並列の git 操作の制限、 `tmp_pack_*` の定期掃除、 サブエージェントによる `git gc` の自動実行の禁止。 書いた時点では、 これで終わりにしたつもりだった。

5 月 4 日 の朝、 同じ事故がもう 1 度起きた。

## 朝に起きたこと

10 時すぎ、 Claude Code CLI と Codex CLI が同時に止まった。 ターミナルに以下のエラーが流れた。

```text
script: write failed:
/home/namakusa/.cache/terminal-session-logs/codex/20260504-101022.typescript: Input/output error

WSL ERROR: getpwnam(namakusa) failed 5
WSL ERROR: getpwuid(1000) failed 5
WSL ERROR: fopen(/etc/default/locale) failed 5
WSL ERROR: getpwuid(0) failed 5
```

ぱっと見はメモリ不足っぽく見えたが、 中を見たらメモリも swap も余っていた。

```text
free -h
              total        used        free
Mem:           31Gi       1.1Gi        ...
Swap:         8.0Gi          0B
```

代わりに詰まっていたのは Windows 側の C ドライブだった。

```text
df -h /mnt/c
Filesystem      Size  Used Avail Use%
C:              931G  893G   38G  96%
```

96% まで埋まると、 ext4.vhdx の書き込みが部分的に失敗する。 そうなると `getpwnam` のような ローカルのファイルを叩く呼び出しまで巻き込まれて、 ユーザー情報の参照すら止まる。 端末からは「Claude が止まった」 ように見えるが、 実体は Windows のディスクが先に死んでいた。

## 何が原因か

WSL の中を眺めて、 見覚えのある景色が広がっていた。

```text
find /home/namakusa -path '*/.git/objects/pack/tmp_pack_*'
... 多数 ...
```

前回と同じ `tmp_pack_*` のクラスタだった。 並列で動いていた git の書き込みが衝突して、 中途半端な pack が大量に残っていた。 1 週間前に書いた「対策の候補」 は、 候補のままだった。 `~/.claude/settings.json` には登録されておらず、 hooks ディレクトリにも該当のスクリプトはなかった。

私が書いた候補は、 文章の中だけで存在していた。 動く形に変換していなかった。

## どこで間違えたか

過去のメモを読み返してみると、 「対策の候補を 3 つ書いた」 で文章が終わっている。 当時の自分は、 書いたことで対策が入った気分になっていた。

実際に必要だったのは、 ３つだった。

候補を hook のスクリプトに変換すること。 そのスクリプトを `~/.claude/settings.json` の matcher の下に登録すること。 設定が壊れていないかをテストで確認すること。 文章のままでは、 同じ並列の git 操作に対して何の効果も持たない。

「対策を考えた」 と「対策を運用に入れた」 は別物で、 その間にある手数を私は省略していた。 これが再発の真因だった。

## 今度入れた対策

夕方に時間を取って、 hook を 3 つ書いた。 それぞれ、 過去の候補のうち 1 つに対応する。

### tmp_pack_* を実行前に掃除する

`git-pack-temp-cleanup-guard.sh` を書いた。 PreToolUse の Bash の matcher に登録する。 git コマンドが走る直前に、 配下の全リポジトリの `.git/objects/pack/tmp_pack_*` を `find` する。

```bash
find "$HOME" -path '*/.git/objects/pack/tmp_pack_*' -type f -mmin +60
```

`-mmin +60` で 60 分より古いものに限定する。 動作中の git が今まさに作っている tmp_pack を消すと書き込み中のデータを壊すので、 古いものだけを掃除する。

合計サイズも測る。 1GiB を超えたら警告。 5GiB を超えたら強い警告。 10GiB を超えたら git コマンド自体を `exit 2` で止める。 144GB まで積み上がった過去事故への直接の応答が、 この閾値だ。

何 GiB を取るかは環境変数で上書きできる。

```bash
CC_GIT_PACK_DISABLE=1     # 全体を無効化
CC_GIT_PACK_BLOCK_GB=10   # 停止の閾値を変える
```

### git gc / repack / maintenance を自動実行から外す

`git-maintenance-guard.sh` も PreToolUse に入れた。 こちらはもっと単純で、 以下のサブコマンドが Bash で走ろうとしたら止める。

```text
git gc
git repack
git maintenance
git prune
git prune-packed
```

これらが裏で勝手に走るのが、 並列環境で `tmp_pack_*` を量産する一番の原因だった。 `git auto gc` が CC のセッションごとに走ると、 同じ repo にぶつかって全員が中途半端な pack を残す。

人間が明示的に走らせる時だけ許可する形にした。

```bash
CC_GIT_MAINTENANCE_ALLOW=1 git gc --aggressive
```

これで自動実行は止めながら、 メンテナンスのために手で走らせる時には邪魔をしないようにできる。

### Windows 側の C ドライブも監視する

既存の `disk-space-guard.sh` は WSL 側しか見ていなかった。 これが今回の事故の盲点で、 WSL の中だけ見ていたら 96% の C ドライブには気付けない。

`wsl-host-disk-space-guard.sh` を新しく書いた。 PreToolUse の Bash と Write の matcher に登録する。 監視対象は 3 つ。

```bash
df -h /          # Linux ルート
df -h /home      # ホームディレクトリ (別マウントの場合)
df -h /mnt/c     # Windows の C ドライブ
```

`/mnt/c` は 90% で警告、 95% で停止。 `/` は 85% で警告、 95% で停止。 95% を超えると ext4.vhdx の書き込みが部分的に失敗し始めるので、 その手前で止めたい。

## ここまでの状態

3 つの hook を `cc-safe-setup` の examples に追加して、 テストを 55 件書いた。 全部 pass している。 `~/.claude/settings.json` の PreToolUse の matcher の下に登録するところまで終わっていて、 設定の差分は以下になる。

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          { "type": "command", "command": "~/.claude/hooks/git-pack-temp-cleanup-guard.sh" },
          { "type": "command", "command": "~/.claude/hooks/git-maintenance-guard.sh" }
        ]
      },
      {
        "matcher": "Bash|Write",
        "hooks": [
          { "type": "command", "command": "~/.claude/hooks/wsl-host-disk-space-guard.sh" }
        ]
      }
    ]
  }
}
```

セッションを開き直して、 `git status` を実行した。 該当の `tmp_pack_*` が 60 分以上古ければ自動で消えていく。 `git gc` を試しに打つと、 即座に「環境変数で上書きしないと走れない」 と止まる。 大きい書き込みの前に C ドライブの残量も自動で表示される。

これで、 候補から運用に変換した。

## 学んだことというか、 反省

書いた文章は、 動く対策に勝手に変わってはくれない。 候補を書いた時点で達成感があると、 その後の手数を省略しがちになる。 私はそれをやって、 1 週間後に同じ事故を踏んだ。

同種の事故が 3 回目を迎えたら、 hook の設計に欠陥がある証拠になる。 そこまでは、 「候補から運用への変換が動いた事実」 が今回追加した中身だと思っている。 hook の中身そのものよりも、 そこに変換するまでの手数が、 失敗から学ぶ時の本体だった。

C ドライブを 38GB から戻す作業は、 PowerShell の `wsl --shutdown` 後に `Optimize-VHD -Mode Full` を走らせれば終わるはずで、 これは月末の運用作業として別途やる。
