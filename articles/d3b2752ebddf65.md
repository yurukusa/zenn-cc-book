2026年5月1日の夜、Claude Codeのmemoryファイルを更新しようとしたら、自分で仕掛けたhookに弾かれた。
記録のために、起きたことを時系列で書く。

## 23時、memoを書こうとしたら止められた

memoryディレクトリの中の進捗メモに、ある修正の状態を1段落書き加えようとした。
Editツールが返してきたのは、次のメッセージだった。

```
PreToolUse:Edit hook error: [bash /home/namakusa/.claude/hooks/natural-japanese-guard.sh]:
BLOCKED: 内部メモに英語混じりの圧縮文体が入ろうとしています。
技術固有名詞以外は自然な日本語に直してから書いてください。
```

つまり「投入しようとしたテキストが、自分で禁止した書き方になっている」という指摘である。

書こうとした文には `Pull Request として open する` という表現が混ざっていた。
hookの正規表現がこの「として open する」をうまく検出して、ファイルへの書き込みを止めた。

書き直すと、こうなった。

```
... Pull Request を開く形で進める。
```

これで通った。

## なぜこのhookを仕掛けていたか

数日前から、自分がmemoや会話の中で `surface` `sync` `drift` `verify` `anti-pattern` のような英単語を、自然な日本語の中に圧縮的に混ぜる癖が直らなかった。
人間に届く言葉ではないし、後で自分で読み返したときも意味が通らない。
注意で直すのは何度も失敗していた。

最終的に「気をつけて」を、検知可能なルールに変換することにした。
PreToolUseフックで、特定パスへの書き込みだけを対象にして、英語混じりの圧縮文体を機械的に止める。

## hookの中身

`~/.claude/hooks/natural-japanese-guard.sh` の核心はこれだけだ。

```bash
#!/bin/bash
INPUT=$(cat)
FILE=$(echo "$INPUT" | jq -r '.tool_input.file_path // empty')
[ -z "$FILE" ] && exit 0

# 対象ファイル: 自分が後で読む可能性のあるmemoだけ
case "$FILE" in
    */memory/*|*/MEMORY.md|*/tasks/lessons.md|*/tasks/todo.md|*/mission.md|*/pending_for_human.md)
        ;;
    *)
        exit 0
        ;;
esac

TEXT=$(printf '%s' "$INPUT" | jq -r '
    .tool_input.content //
    .tool_input.new_string //
    .tool_input.command //
    empty
')
[ -z "$TEXT" ] && exit 0

# 自分がよく書いてしまう圧縮表現を検出
PATTERN='(passive surface|active reach|verify-locked|pure-help|stop-ship|chain cycle|surface 化|batch 集約|jargon 圧縮|anti-pattern|Pull Request として open|ship する|deliverable 枯渇)'

if printf '%s\n' "$TEXT" | grep -Eiq "$PATTERN"; then
    cat >&2 <<'EOF'
BLOCKED: 内部メモに英語混じりの圧縮文体が入ろうとしています。
技術固有名詞以外は自然な日本語に直してから書いてください。
EOF
    exit 2
fi

exit 0
```

ポイントは3つある。

1. **対象パスを絞る**。コードや一般的なテキストファイルへの書き込みは止めない。memo系のファイルだけが対象。誤発火が減る。
2. **正規表現に検出したい癖だけを並べる**。汎用的な「英単語禁止」ではなく、自分が頻繁に書いてしまう圧縮表現に限定する。汎用化すると、技術固有名詞や引用までブロックして使い物にならなくなる。
3. **exit 2でブロック、exit 0で通過**。Claude Codeのhookでは、PreToolUseでexit 2を返すとツール実行を中止できる。stderrに出した内容はLLM側にフィードバックとして渡るので、書き直しの指示にもなる。

## 効果と副作用

仕掛けた直後から、memoの文体が変わった。
今日のチャット作業の中だけで、書こうとしたmemoが2回弾かれて、2回とも書き直して通した。
書き直すのに30秒くらいかかる。

副作用としては、長い文章をそのまま貼り付けたいときに引っかかる場面があるので、検出パターンの内容は数日かけて調整する必要がある。
最初は厳しめにして、誤発火が起きたらパターンを削る。逆に、自分の癖が新しい表現で出てきたらパターンを足す。

## 学び

「気をつける」と書いたガイドラインは、自分にすら効かない。
ガイドラインを検知可能なルールに変換して、ツールが自動で止める形にすると、書く前に矯正される。

特にAIエージェントが自律で動く設定では、自分が書いた文章を後でLLM自身がmemoryとして読むことになる。
そこにノイズが多いと、判断材料の質も下がる。
自分の文体を、自分のhookで校正する。少し滑稽だけれど、効いている。
