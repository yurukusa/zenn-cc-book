Anthropic Academyのクイズシリーズ第4回。今回はMCP（Model Context Protocol、モデルコンテキストプロトコル）について。

MCPサーバーを作ったことはある。テスト方法を聞かれたら「Claude Desktopにつないで試す」と答えていた。

もっと良い方法があった。

## MCP Inspectorを知らなかった

MCPツールのテストには**MCP Inspector**を最初に使うのが正解だ。

```bash
npx @modelcontextprotocol/inspector
```

これを実行すると、ブラウザでデバッグUIが起動する。デフォルトでポート6277でリッスンする。

できること:
- MCPサーバーに接続
- ツール一覧を表示
- 個別のツールを引数付きで実行
- JSON-RPCメッセージのログを確認

REST APIのテストにPostman（APIを手動で叩けるツール）を使うのと同じ発想だ。クライアント（Claude）を繋ぐ前に、まずサーバー単体でツールが動くか確認する。

自分はいきなりClaude Desktopに接続して「動かない」→「ログを探す」→「どこが悪いのかわからない」を繰り返していた。サーバー側のバグなのかクライアントとの接続問題なのか、切り分けすらできていなかった。

:::details CC補足: MCP Inspectorの詳細な使い方
MCP Inspectorの操作フロー:

1. `npx @modelcontextprotocol/inspector` でUIを起動
2. 「Connect」ボタンでMCPサーバーを起動・接続
3. UI上部のメニューから「Tools」を選択
4. 「List Tools」でサーバーが公開しているツール一覧を取得
5. 個別ツールをクリックして引数入力フォームを表示
6. 「Run Tool」でツールを実行し、レスポンスを確認

Python SDKを使っている場合は、`npx`ではなくSDK内蔵のdevコマンドも使える:
```bash
mcp dev mcpserver.py
```

UIにはResources / Prompts / Toolsの3つのタブがある。UIは開発中で頻繁に変わるため、細かい操作はその時点の最新ドキュメントを確認するのが確実。

MCPサーバー開発のライフサイクル:
1. コード編集
2. **Inspectorでテスト** ← ここが重要
3. 問題なければクライアント統合
4. クライアント統合後も問題があればInspectorに戻る

Inspectorを挟むことで「サーバーのバグ」と「クライアント連携の問題」を切り分けられる。
:::

## ListToolsRequest — クライアントがツールを発見する仕組み

MCPサーバーがツールを公開する。クライアント（Claudeなど）がそのツールを使う。ここまでは理解していた。

では、クライアントはどうやって「このサーバーに何のツールがあるか」を知るのか。考えたことがなかった。

答えは**ListToolsRequest**というプロトコルメッセージだ。

MCPのメッセージフローはこうなっている:

```
クライアント               サーバー
    |                        |
    |--- initialize -------->|  ハンドシェイク
    |<-- initialize resp ----|
    |                        |
    |--- ListToolsRequest -->|  「何ができる？」
    |<-- ListToolsResponse --|  ツール一覧を返す
    |                        |
    |--- CallToolRequest --->|  「このツールを実行」
    |<-- CallToolResponse ---|  結果を返す
```

1. **initialize** — ハンドシェイク。サーバーの基本情報を交換する
2. **ListToolsRequest** — クライアントが「何ができる？」と聞く
3. **CallToolRequest** — 特定のツールを引数付きで実行する

:::details CC補足: ListToolsRequestのレスポンス構造
ListToolsRequestのレスポンスには、各ツールについて以下の情報が含まれる:

```json
{
  "tools": [
    {
      "name": "get_weather",
      "description": "指定された都市の現在の天気を取得する",
      "inputSchema": {
        "type": "object",
        "properties": {
          "city": {
            "type": "string",
            "description": "天気を取得する都市名"
          }
        },
        "required": ["city"]
      }
    }
  ]
}
```

- **name**: ツールの識別名。CallToolRequestで指定する
- **description**: ツールの説明。Claudeはこの説明を読んで「いつこのツールを使うか」を判断する
- **inputSchema**: JSON Schema形式の引数定義。型、説明、必須/任意を記述

Claudeが正しくツールを使えるかどうかは、このdescriptionとinputSchemaの品質に直接依存する。
:::

## descriptionがツールの使われ方を決める

ListToolsRequestのレスポンスに含まれるdescriptionは、ただの説明文ではない。

**Claudeはこのdescriptionを読んで、そのツールをいつ使うか判断する。** 曖昧な説明を書けば、Claudeはツールを誤用する。具体的な説明を書けば、Claudeは適切なタイミングで正しくツールを使う。

例を挙げる。

```
❌ 悪い例: "データを取得する"
→ どんなデータ？ いつ使う？ Claudeが判断できない

✅ 良い例: "指定された都市名から現在の天気情報（気温、湿度、天候）を取得する。都市名は英語で指定する必要がある"
→ 何をするか、入力の制約、返ってくるものが明確
```

MCPサーバーを作るとき、ツールの実装ロジックに集中しがちだ。だがClaudeから見えるのはdescriptionとinputSchemaだけ。ここが雑だと、どれだけ良い実装をしても正しく呼ばれない。

:::details CC補足: Python SDKでのツール定義と自動Schema生成
MCP Python SDKでは、`@mcp.tool`デコレータと型アノテーションを使ってツールを定義する:

```python
from mcp.server import Server
from pydantic import Field

mcp = Server("weather-server")

@mcp.tool()
async def get_weather(
    city: str = Field(description="天気を取得する都市名（英語）"),
    unit: str = Field(
        default="celsius",
        description="温度の単位。'celsius' または 'fahrenheit'"
    )
) -> str:
    """指定された都市の現在の天気情報を取得する。
    気温、湿度、天候の情報を含む。"""
    # 実装...
    return f"{city}の天気: 晴れ、25°C"
```

SDKが関数の型アノテーションと`Field`の情報から自動的にJSON Schemaを生成する。手書きのSchema定義は不要。

- 関数のdocstringがtoolのdescriptionになる
- `Field(description=...)`が各引数のdescriptionになる
- 型アノテーション（`str`, `int`, `list[str]`など）がSchemaの型情報になる
- `default`値がある引数はoptional、ない引数はrequiredになる

エラーハンドリング: 関数内で`raise ValueError("都市名が見つかりません")`のようにエラーを投げると、エラーメッセージがtool_resultとしてクライアントに返される。
:::

## MCPサーバー開発の正しい順序

Academyを受ける前の自分の開発フロー:
1. コードを書く
2. Claude Desktopに接続する
3. 「動かない」→ ログを探す → 原因不明 → コードを修正 → 2に戻る

Academyを受けた後の正しいフロー:
1. コードを書く
2. **MCP Inspectorで単体テスト**
3. ツールが正しく動くことを確認
4. Claude Desktopに接続
5. descriptionの改善 → 2に戻る

Inspectorを挟むだけで、「サーバー側のバグ」と「クライアント連携の問題」と「descriptionの品質問題」の3つを切り分けられるようになる。

## JSON-RPCとMCPの関係

MCPのメッセージングはJSON-RPC（JSON-based Remote Procedure Call、JSONベースの遠隔手続き呼び出し）というプロトコルの上に構築されている。

JSON-RPCは「リクエスト→レスポンス」の単純な構造だ。メソッド名と引数をJSONで送り、結果をJSONで受け取る。MCPはこのJSON-RPCの上に、initialize、ListToolsRequest、CallToolRequestなどのメソッドを定義している。

MCP Inspectorでログを見ると、実際に飛んでいるJSON-RPCメッセージが確認できる。「何が送られて何が返ってきたか」が全て可視化される。トラブルシューティングには最も有用な情報だ。

:::details CC補足: JSON-RPCメッセージの実例
MCP Inspectorのログに表示されるJSON-RPCメッセージの例:

**リクエスト（クライアント→サーバー）:**
```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/list",
  "params": {}
}
```

**レスポンス（サーバー→クライアント）:**
```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "tools": [
      {
        "name": "get_weather",
        "description": "指定された都市の天気を取得する",
        "inputSchema": { ... }
      }
    ]
  }
}
```

`id`フィールドでリクエストとレスポンスを紐づける。非同期で複数のリクエストが飛んでも、idで対応関係がわかる。
:::
