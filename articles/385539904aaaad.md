## 前置き

AIに仕事を任せて寝る。これは前にやった。一晩で88タスク完了した。

でも1つ問題があった。**止まる**。

AIは優秀だが、判断に迷うと人間に質問して待つ。人間が寝てると、そこで作業が停止する。朝起きたら「この方針でいいですか？」と聞いたまま5時間止まってた、なんてことが普通に起きる。

解決策を考えた。**AIをもう1体用意して、2体で相談させればいい。**

## 構成

tmuxで画面を左右に分割。左にClaude Code（実行担当）、右にCodex CLI（計画担当）。

```
┌─────────────────┬─────────────────┐
│  Claude Code    │  Codex CLI      │
│  (CC=実行)      │  (計画・助言)    │
│                 │                 │
│  コードを書く    │  方針を考える    │
│  ファイルを編集   │  レビューする    │
│  テストを実行    │  次のタスクを提案 │
└─────────────────┴─────────────────┘
```

これを `cc-codex-pair` というコマンド1つで起動する。起動すると自動で：

1. tmuxセッションを作って両方のAIを立ち上げる
2. watchdog（死活監視）をバックグラウンドで開始
3. consult-loop（停滞検知・通知）をバックグラウンドで開始
4. 各AIに役割認識プロンプトを自動送信

人間がやることは `cc-codex-pair` を実行するだけ。あとは勝手に回り始める。

## 3つの仕組み

### 1. 停滞検知（consult-loop）

30秒ごとにペインの変化をハッシュでチェック。2分以上動きがなければ「止まってるよ」と声をかける。

```
[停滞検知] ○分以上止まってるよ。待ちに入らないで。
ぐらすにプラスになることを自分で考えて動こう。
タスクがなければ: コード改善、テスト追加、記事執筆、
新アイデアのブレスト、何でもいい。止まるな、創れ。
```

こんな感じのメッセージが自動で送られる（実装では経過時間やAIの状態に応じて文面が変わる）。

ポイントは「mission.mdを見ろ」じゃなくて「自分で考えて動け」と促していること。受動的な指示待ちではなく、創造的な自律行動を促す設計にした。

同じ停滞に何度も介入しない。復帰したら自動リセット。

### 2. 変更通知（activity-logger）

CCがファイルを編集するたびに、PostToolUseフックで自動記録する。

```json
{"ts":"2026-02-15T01:51:12Z","actor":"CC","tool":"Edit",
 "path":"~/bin/cc-codex-consult-loop","add":50,"del":30,
 "needs_review":false}
```

`add`/`del` は概算値（old_string/new_stringの行数差から推定）。正確なdiffではないが、変更の規模感がわかれば十分。

この記録は通知キューに追加される。Codexがアイドルになったタイミングで、まとめて配信。

```
[変更通知] 3件の変更があったよ。（うち1件は要確認）
  ⚠️ CC: Edit ~/bin/cc-codex-consult-loop (+50/-30)
  • CC: Edit ~/ops/mission.md (+5/-2)
  • CC: Write ~/drafts/article.md (+100/-0)
詳細: ~/ops/activity-log.jsonl
```

`bin/` や `.claude/hooks/` への変更は自動で「要確認」マークがつく。方針は合意していても、実際のコードを見て「それじゃダメ」と気づけることがある。その気づきの機会を確保するための仕組みだ。

### 3. 死活監視（watchdog）

60秒ごとにAIプロセスの生存チェック。落ちてたら自動再起動。再起動上限あり（600秒で6回まで）。

## 設計で大事にしたこと

### 常時リレーの廃止

最初は60秒ごとにチャット画面のキャプチャを相互送信していた。トークンの無駄遣いだった。

実際に運用してみると、チャット画面の末尾20行を送り合っても「何のファイルのどこを変えたか」はわからない。必要なのはチャットの断片じゃなくて、実際のコード変更の記録だった。

だからconsult-loopの役割を「常時情報共有」から「停滞検知 + 変更通知配信」に切り替えた。コンテキスト消費はほぼゼロになった。

### ブロックしない設計

合意形成システムも入れた。`bin/` や `hooks/` を変更すると「合意なしの変更」として記録される。でも**ブロックはしない**。

AIの手足を縛るのではなく、変更を可視化して記録する。止めるのではなく、後から確認できるようにする。仲間としての安全柵であって、権限的な制限ではない。

### 日本語で会話する

これは人間側の要望。2体のAIが英語で会話していると、人間がパッと見て何をしているのかわからない。日本語で会話するように設定したら、モニターをチラ見するだけで状況がわかるようになった。娯楽としても楽しい。

## 実際の運用

起動から完全自律で動く。人間は定期的にtmuxの画面を覗いて、面白いやり取りを眺めているだけでいい。

止まったら2分で自動介入。コード変更は自動通知。プロセスが落ちたら自動再起動。人間が寝ていても開発は止まらない。

## コード

シェルスクリプト + Python数行で書いた。特別なフレームワークは不要。tmuxとbashがあれば動く。

- `cc-codex-pair`: セッション作成 + 自動起動（約140行）
- `cc-codex-consult-loop`: 停滞検知 + 通知配信（約430行）
- `cc-codex-watchdog`: 死活監視 + 自動復旧（約330行）
- `codex-send` / `cc-send`: メッセージ送信ラッパー（各60-80行）
- `activity-logger.sh`: PostToolUseフック（約110行）

合計約1,100行。この仕組み自体もAIと一緒に作った。

## まとめ

「AIに任せて寝る」の次のステップは「AIを2体にして相談させる」だった。

1体だと止まる。2体にすると止まらない。ただし情報共有の仕組みを工夫しないとトークンを浪費する。

最終的な設計は：
- **停滞検知**: 止まったら声をかける（2分ルール）
- **変更通知**: コード変更を自動記録・配信（チャット断片ではなく実変更）
- **死活監視**: 落ちたら自動再起動

人間の介入を最小化しながら、AI同士が自律的に相談して動き続ける。これが今の自分の開発スタイルだ。
