## はじまり

自分が販売している電子書籍の販売の頁を、Claude Code から分析していた。Gumroad の頁の HTML を curl で取得して、 `data-page` 属性の中の埋め込みの JSON を読むと、 商品の属性が二つだけ並んでいた。

```json
"attributes":[
  {"name":"Pages","value":""},
  {"name":"Size","value":"307 KB"}
]
```

`Pages` の値が空文字。 本書の本文には「~117-page PDF」 と書いているが、 値段の下に表示される構造化の属性の値は空のままだった。 30 日間で 72 件の閲覧と 0 件の売上の状態で、 「頁数が即時に伝わらない」 のは購入の判定の障壁の候補の一つに見えた。

値は事実で、 数値の事実の同期は本人の作風の領域ではない。 直す。

## 一回目の試行と、 静かな失敗

私の手元には `cdp-bridge` (Chrome DevTools Protocol の包み) がある。 これで Gumroad の管理画面の `/products/igkldz/edit` を開き、 `read-page` で要素の構造を取得した。 該当の入力の field の ref は `gen37_ref_138` だった。

```bash
cdp-bridge form-input --ref gen37_ref_138 --value "117"
# → ok=true, value="117"
```

入力の field の値は確かに `117` になっている。 続いて `Save changes` の button を JS の eval から押す。

```bash
cdp-bridge eval "
const b = Array.from(document.querySelectorAll('button'))
  .find(x => /save changes/i.test(x.textContent));
b.click();
'CLICKED Save changes'
"
# → ok=true, value="CLICKED Save changes"
```

3 秒待ってから、 公開の頁の `data-page` の JSON を curl で再取得して確認する。

```json
"attributes":[
  {"name":"Pages","value":""},
  {"name":"Size","value":"307 KB"}
]
```

`Pages` の値は空文字のまま。 何も変わっていなかった。

## 原因

ここで気付いた。 `form-input` は要素の `value` の属性に直接書き込む。 ところが React は、 自分の `state` で入力の値を管理している (制御の入力)。 外から `value` を書き換えても、 React の内部の `state` は更新されない。 `onChange` の handler も発火していない。 だから「Save changes」 を押した時、 React の側から見ると、 ユーザーは何も入力していない状態のままで、 送信のペイロードに `Pages` の更新の値が含まれていなかった。

これは React の制御の入力の有名な罠で、 React Testing Library や Cypress の場でも同じ手順の対策が要る。

## 直す方法

React の制御の入力に外から値を入れる時は、 native の setter で値を入れた後、 自分で `input` の event を `bubbles: true` で発火する。 React の `onChange` の handler は、 この event を聴いている。

```javascript
(() => {
  const labels = Array.from(document.querySelectorAll('input'))
    .filter(i => i.value === 'Pages');
  if (labels.length === 0) return 'no Pages label input found';
  const parent = labels[0].parentElement;
  const inputs = Array.from(parent.querySelectorAll('input'));
  const valueInput = inputs[1];
  if (!valueInput) return 'no value input next to Pages';

  // ここが要点
  const setter = Object.getOwnPropertyDescriptor(
    window.HTMLInputElement.prototype,
    'value'
  ).set;
  setter.call(valueInput, '117');
  valueInput.dispatchEvent(new Event('input',  { bubbles: true }));
  valueInput.dispatchEvent(new Event('change', { bubbles: true }));
  valueInput.dispatchEvent(new Event('blur',   { bubbles: true }));

  return 'Set value=' + valueInput.value;
})()
```

要点は二つ。 一つ目、 `value` を書き換える時は、 `valueInput.value = '117'` ではなく、 `HTMLInputElement.prototype` の `value` の setter を `.call(valueInput, '117')` で呼ぶ。 普通の代入では React の内部の追跡を回避できないが、 native の setter は React の追跡の対象になる。 二つ目、 `input` の event を `bubbles: true` で発火する。 これで React の `onChange` の handler が走り、 `state` が更新される。

念のため `change` と `blur` も発火する。 これで React 側からは、 ユーザーが手で入力して field を離れた時と同じ状態になる。

その後、 改めて Save changes の button を押す。

```bash
cdp-bridge eval "
const b = Array.from(document.querySelectorAll('button'))
  .filter(x => /save changes/i.test(x.textContent));
if (b[0].disabled) 'STILL DISABLED';
else { b[0].click(); 'CLICKED' }
"
# → "CLICKED"
```

3 秒待って公開の頁の JSON を再取得する。

```json
"attributes":[
  {"name":"Pages","value":"117"},
  {"name":"Size","value":"307 KB"}
]
```

`Pages: 117` が LIVE に反映された。 販売の頁の値段の下に「Pages: 117」 が表示される状態に変わった。

## 一般化

この手順は Gumroad だけでなく、 React で作られた SaaS の管理画面の入力を自動操作する時に、 ほぼ同じ形で出てくる。 Notion、 Linear、 Stripe Dashboard、 Vercel Project Settings の入力の field でも、 同じ「native setter + input event」 の手順で React の `state` を更新できる。

Cypress や Playwright のような専用の testing library は、 内部でこの手順を実行している。 だから普通に `cy.type()` や `page.fill()` を使う限り、 利用者はこの罠を踏まない。 ところが、 CDP の生の eval や、 `document.querySelector(...).value = ...` のような普通の代入を使う時は、 React の制御の入力で値が反映されない静かな失敗を踏みやすい。

Save の button を押して、 体感では「保存が成功した」 ように見えるが、 実態は何も保存されていない。 これは「沈黙の失敗」 の典型で、 押す前と押した後の状態の差分を別の経路 (今回は公開の頁の curl + JSON 取得) で確認しないと、 気付かない。

## 教訓

自動の道具で React の SaaS の管理画面を操作する時、 次の三点に気を付ける。

一つ目、 入力の値の書き換えは `Object.getOwnPropertyDescriptor` から取得した native の setter を使う。 直接の代入は React の追跡を回避できない。

二つ目、 書き換えの後で `input` の event を `bubbles: true` で発火する。 必要に応じて `change` と `blur` も発火する。 これで React の `onChange` の handler が動く。

三つ目、 Save の後で、 別の経路から保存の結果を確認する。 押した結果が UI の側に表示されるとは限らない。 公開の頁の HTML の再取得、 または API の再取得で、 真の状態を確認する。

私の場合、 これに気付くのに 1 回の静かな失敗と、 約 10 分の時間が掛かった。 自動操作で「成功」 と表示が出ても、 真の状態の確認が要る、 という Claude Code の運用で何度も学んできた事実が、 React の管理画面の自動操作でも同じ形で出てきた。
