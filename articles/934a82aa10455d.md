# ゲームの手触りを3つの数値で測る — Feel Scorecardの設計と実践

:::message
この記事は2026年2月時点の情報に基づいている。ゲームはGodot 4.3製、開発はClaude Codeによる非エンジニア+AI体制。
:::

## 「面白い」を数値にできるか

ゲームを作っていて、一番困る瞬間がある。

「これ、面白いのか？」

自分で遊んでもわからない。自分が作ったから楽しいのか、本当に楽しいのか、判別がつかない。友人に投げても「いいんじゃない？」しか返ってこない。覚えがあるはずだ。

自分はVampire Survivors系のサバイバー型ゲーム「Spell Cascade」をClaude Codeと一緒に作っている。非エンジニアだ。コードは1行も書けない。AIが書いたコードをAIがテストしている。

だったら「面白さ」もAIに測らせればいい。

そう思って作ったのがFeel Scorecardだ。ゲームの「手触り」を3つの数値に落とし込む仕組み。結果、自分のゲームに11秒の「死んだ時間」があることがわかった。数値で見るまで気づけなかった。

## VS系ゲームの先行知見

Vampire Survivors系のジャンルには、すでに体感として知られている法則がいくつかある。

最も有名なのが「23秒ルール」だ。レベルアップの平均間隔が約23秒。これより短いと報酬ラッシュで疲れる。長いと退屈になる。このリズムがプレイヤーを「あと1回だけ」に引き込む。

もうひとつ。画面上の情報密度。敵・弾・エフェクト・ドロップアイテムが常に画面を埋めている。「何もない瞬間」が少ないほどプレイヤーの没入が続く。

この2つのヒントから、自分のゲームの手触りを定量化できないかと考えた。

## 20個の候補から3つを選ぶ

最初は手当たり次第に指標を挙げた。20個。

Feedback Latency（入力から反応までの遅延）、Color Variety（画面の色数）、Input Responsiveness（操作応答率）、VFX Count（エフェクト数）、Screen Occupation（画面占有率）......。

「測れそうなもの」を片っ端から並べた。が、ここで気づく。**測れることと、測って意味があることは別だ。**

20個を6個に絞った。さらに3つを落とした。

落としたのはPower Growth Rate（パワー成長率）、Tension Oscillation（緊張の波）、Session Milestones（セッション内マイルストーン）。理由は単純で、自動テストの60秒間では安定して測定できなかったから。Tension Oscillationは理論上おもしろい指標だが、計測にはもっと長いランが必要だった。

残った3つ。これがFeel Scorecardになった。

## Feel Scorecardの3指標

### 1. Action Density（行動密度）

```
events/sec = (発射数 + キル数 + アイテム回収数 + レベルアップ数) / 経過秒数
```

1秒あたりに起きるイベントの数。「何か起きている感」の定量化。

Vampire Survivors系なら3〜8 events/secが心地いい。1未満は退屈。15を超えると画面が読めなくなる。

### 2. Dead Time（死んだ時間）

```
max_gap = イベント間の最大空白時間（秒）
```

**最も診断力の高い指標。** 「何も起きていない」最長の瞬間。

0〜3秒なら問題ない。5秒で黄色信号。10秒を超えたらプレイヤーはスマホを見始める。

Action Densityが「平均値」なのに対して、Dead Timeは「最悪値」だ。1分間のうち55秒が楽しくても、5秒の空白があればプレイヤーはそこで離脱する。平均値では見えない穴を、Dead Timeが暴く。

### 3. Reward Frequency（報酬頻度）

```
pickups/min = XPアイテム回収数 / 経過分数
```

1分間にどれだけ「拾えた」か。ドロップが少なすぎると動機が消える。多すぎると価値がなくなる。

10/min未満は過疎。20〜60/minが快適ゾーン。100を超えると画面がXPだらけで逆にストレスになる。

## 閾値表

| 指標 | EXCELLENT | GOOD | WARN | FAIL |
|------|-----------|------|------|------|
| Action Density (events/s) | 3 - 8 | 1 - 3 | 8 - 15 | <1 or >15 |
| Dead Time (秒) | 0 - 3 | 3 - 5 | 5 - 10 | 10+ |
| Reward Frequency (/min) | 20 - 60 | 10 - 20 | 60 - 100 | <10 or >100 |

:::details CC補足: 閾値の根拠
これらの閾値はVampire Survivors、Brotato、Halls of Torment等のVS系タイトルの体感値と、実際の自動テスト結果から設定したもの。学術論文ベースではなく、「自分のゲームを回してみて、この数値だと退屈/快適/カオスと感じた」という経験的な境界線。ジャンルが変われば閾値も変わる。RPGやパズルにそのまま適用はできない。
:::

## 実データ: Spell Cascade v0.3.2

自動テストを60秒間走らせた。botがゲームを操作して、テレメトリを収集する。

結果がこれだ。

| 指標 | 値 | 判定 |
|------|-----|------|
| Action Density | 1.6 events/s | GOOD |
| Dead Time | **11.3秒** | **FAIL** |
| Reward Frequency | 27 pickups/min | GOOD |

Action DensityもReward Frequencyも合格。だがDead Timeが11.3秒でFAIL。

11.3秒。何もイベントが起きない時間が11秒以上ある。

テストのログをさらに掘ると原因が見えた。ゲーム序盤だ。敵がまだ少なく、スキルのクールダウンも長い。火球を撃つ。何も起きない。また撃つ。何も起きない。この空白が11秒に膨れ上がっていた。

その他の生データも載せておく。

```
total_fires:    49 Fireballs / 60秒
level_ups:      2回（27.9秒、35.7秒）
damage_taken:   13回
peak_enemies:   30体（55秒時点）
lowest_hp:      66%
final_hp:       96%
```

2回のレベルアップ間隔は7.8秒。23秒ルールの3分の1。速すぎる。が、これはゲーム全体の1分間だけのデータなので、序盤の成長速度としては妥当かもしれない。

:::details CC補足: Run Completion Desire（ラン完了欲求）
Feel Scorecardには4つ目の指標としてRun Completion Desire（RCD）がある。「もう1回遊びたいか」を最終HP割合から推定するヒューリスティクス。HP 30-70%で生存終了が理想（ギリギリ感が次のランへの動機になる）。今回のRCDは0.31でFAIL。最終HP 96%、つまりほぼ無傷で終了しており、botにとってはぬるいゲームだった。ただし人間プレイヤーは最適な動きをしないため、この指標はbot/人間の差が大きい。記事では3指標に絞って解説している。
:::

## GDScript実装

実装はシンプルだ。全イベント（発射・キル・アイテム回収・被ダメ）のタイムスタンプを配列に突っ込む。最後にmax gapを計算する。それだけ。

```gdscript
# Feel Scorecardの核心: イベントタイムスタンプ収集
var feel_event_timestamps: Array[float] = []
var feel_kill_count := 0
var feel_xp_pickup_count := 0

# 何かが起きるたびにタイムスタンプを記録する
# 発射、キル、XP回収、被ダメ — すべて同じ配列に入れる
func _collect_quality_metrics(elapsed: float, delta: float) -> void:
    # 被ダメ検出
    if cur_hp < _last_hp:
        feel_event_timestamps.append(elapsed)

    # キル検出
    if cur_kills > _last_kill_count:
        var new_kills := cur_kills - _last_kill_count
        for _k in range(new_kills):
            feel_event_timestamps.append(elapsed)

    # XP回収検出
    if cur_xp > _last_xp:
        feel_xp_pickup_count += 1
        feel_event_timestamps.append(elapsed)
```

テスト終了時に3指標を一気に計算する。

```gdscript
# Dead Time: 最大空白時間
feel_event_timestamps.sort()
var dead_time := 0.0
for idx in range(1, feel_event_timestamps.size()):
    var gap: float = feel_event_timestamps[idx] - feel_event_timestamps[idx - 1]
    dead_time = maxf(dead_time, gap)

# Action Density: 秒あたりイベント数
var total_events: int = total_fires + kill_count + xp_pickup_count + level_ups
var action_density: float = float(total_events) / game_duration

# Reward Frequency: 分あたりXP回収数
var reward_freq: float = float(xp_pickup_count) / (game_duration / 60.0)
```

このコードは`SpellCascadeAutoTest.gd`に組み込まれている。`godot --headless`で実行できるので、CIに入れてプルリクごとにFeel Scorecardを自動計算できる。

:::details CC補足: 閾値判定のコード
上で示した閾値表をコードに落とすとこうなる。
```gdscript
# Dead Time判定
var dead_time_rating := "EXCELLENT"
if dead_time > 10.0:

elif dead_time > 5.0:

    reward_rating = "SPARSE"

    reward_rating = "OVERWHELMING"
elif reward_freq > 60.0:
    reward_rating = "ABUNDANT"
```
:::

## Dead Timeが最も鋭い

26回のユニークなテストを回した（リプレイ分を除く）。GO判定18回、CONDITIONAL 4回、NO-GO 4回。

3指標のうち、最も頻繁に問題を検出したのがDead Timeだった。Action DensityやReward FrequencyがGOODでも、Dead TimeだけFAILになるケースが何度もあった。

考えてみれば当然だ。平均値は外れ値を飲み込む。1分間に93回イベントが起きていても、その93回が40秒間に集中していたら、残りの20秒は砂漠だ。平均1.6 events/secの裏に、11.3秒の虚無が隠れている。

Dead Timeは「平均値の嘘」を暴く指標だ。

自分のゲームの場合、序盤の敵スポーン間隔を詰めることで改善できる見込みがある。敵が増えれば被弾もキルもXPドロップも増える。Dead Timeは構造的な問題を指し示しているから、対処も構造的にやればいい。

## この仕組みの限界

正直に書いておく。

Feel Scorecardはbotが遊んだ結果を測っている。人間ではない。botは最適に動く。人間はミスをする。だからDead Timeは人間プレイ時にはもっと短いかもしれない（被弾イベントが増えるから）。逆に、Action Densityは人間の方が低くなる可能性がある（射撃タイミングを逃すから）。

数値は参考値であって、判決ではない。「ここに問題がありそうだ」という診断ツールだ。

ただ、診断ツールがないよりはずっとマシだ。少なくとも自分は、11.3秒の空白を体感では見つけられなかった。

## まとめ

ゲームの手触りを測りたかった。20個の候補から3つに絞った。実装してテストを回したら、自分のゲームの弱点が数値で見えた。

- Action Density: 1.6 events/s → GOOD（でも低め）
- Dead Time: 11.3秒 → FAIL（序盤が虚無）
- Reward Frequency: 27 pickups/min → GOOD（快適ゾーン）

Dead Timeが最も有用な指標だった。平均値では見えない空白を暴く。VS系ゲームを作っている人は、まずDead Timeだけでも測ってみてほしい。たった数行のコードで、自分のゲームのどこが「退屈」なのかがわかる。
