## 起きたこと

利用者の `settings.json` に次の禁止の規則を書いた状態で、

```json
{
  "permissions": {
    "deny": ["Bash(git commit *)"]
  }
}
```

Claude Codeは別の作業の場所の作業で次の命令を実行する。

```bash
git -C /path/to/repo commit -m "test"
```

結果は予想に反して、 確認の窓も拒否の警告もなく、 commitが成功する。 `Bash(git commit *)` の禁止の規則は、 静かに素通りされた。

## 公式の起票の継続

2026年5月14日、 利用者の `tnt108` がGitHub Issue [#59006](https://github.com/anthropics/claude-code/issues/59006) でこの事例を報告した。 根本原因の素案つきで、 詳細な再現の手順も含まれていた。

自動の重複の判定の段で、 GitHub-Actionsの応答は次の3件の起票を引用した。

- [#25270](https://github.com/anthropics/claude-code/issues/25270): Deny rules are bypassable via flag reordering (e.g. `git -C`) - 2026年2月12日に閉鎖済
- [#18613](https://github.com/anthropics/claude-code/issues/18613): Security: Deny rules can be bypassed by flag reordering/insertion - 2026年5月14日の時点で**開いたまま**
- [#52409](https://github.com/anthropics/claude-code/issues/52409): Permissions: command-level matching instead of string prefix matching - 2026年4月27日に閉鎖済

つまりこの問題は数ヶ月にわたって複数の利用者が独立に報告し続けており、 基幹の起票 #18613 が今も開いたまま、 利用者の側の予防の手段の必要性が高い状態。

## 根本原因

#59006 の起票者の根本原因の素案を引用する。

> The pattern matcher for `Bash(git commit *)` likely checks whether the command string starts with `git commit`, but `git -C <path> commit` places `-C <path>` between `git` and `commit`, causing the pattern match to fail silently.

要するに、 禁止の規則の照合は命令の文字列の頭の部分の一致のみを見ている。 `git -C /repo commit` の頭の部分は `git -C` であり、 `git commit` ではないため、 規則の照合が失敗する。 同じ系統の素通りは次の形でも発生する。

- `git --git-dir=.git commit` (`=`形の旗)
- `git --git-dir .git commit` (別の素の形の旗)
- `git --work-tree=. commit`
- `git -c user.name=foo commit`
- `git --no-pager commit`
- `git --namespace=ns commit`

## 利用者の側の予防の手段

公式の修正を待つ間、 PreToolUseの安全装置で同等の防衛を実装できる。 命令を素に分け、 旗を飛ばし、 実効の小命令を識別、 禁止の一覧と照合する経路。

```bash
#!/bin/bash
# git-flag-bypass-guard.sh
# TRIGGER: PreToolUse  MATCHER: "Bash"

set -euo pipefail

INPUT=$(cat)
CMD=$(echo "$INPUT" | jq -r '.tool_input.command // empty' 2>/dev/null)
[ -z "$CMD" ] && exit 0

# 禁止の一覧は環境変数で指定 (既定値はcommit、空で無効化)
DENY="${CC_GIT_FLAG_DENY-commit}"
[ -z "$DENY" ] && exit 0

# shellcheck disable=SC2206
read -r -a TOKENS <<< "$CMD"
[ "${#TOKENS[@]}" -eq 0 ] && exit 0

# gitの素を探す (環境変数の前置きや絶対の経路にも対応)
git_idx=-1
for i in "${!TOKENS[@]}"; do
    t="${TOKENS[$i]}"
    case "$t" in
        *=*)
            [[ "$t" == -* ]] && break
            continue ;;
        git|*/git)
            git_idx=$i; break ;;
        *)
            break ;;
    esac
done
[ "$git_idx" -lt 0 ] && exit 0

# gitの後の旗を飛ばし、 実効の小命令を識別
i=$((git_idx + 1))
SUBCMD=""
while [ "$i" -lt "${#TOKENS[@]}" ]; do
    t="${TOKENS[$i]}"
    case "$t" in
        # 値を別の素として取る旗
        -C|--git-dir|--work-tree|-c|--namespace)
            i=$((i + 2)); continue ;;
        # =形の旗、 または値なしの旗
        -*)
            i=$((i + 1)); continue ;;
        *)
            SUBCMD="$t"; break ;;
    esac
done

[ -z "$SUBCMD" ] && exit 0

# 禁止の一覧と照合
IFS=',' read -r -a DENY_LIST <<< "$DENY"
for denied in "${DENY_LIST[@]}"; do
    [ -z "$denied" ] && continue
    if [ "$SUBCMD" = "$denied" ]; then
        echo "BLOCKED: git deny rule for '$denied' (form: $CMD)" >&2
        echo "  Reason: prefix matcher misses 'git $denied' when prefixed with flags like -C, --git-dir, -c, --no-pager." >&2
        echo "  See #18613 for the broader fix." >&2
        exit 2
    fi
done

exit 0
```

`~/.claude/hooks/git-flag-bypass-guard.sh` の場所に保存して実行の許可を与え、 `~/.claude/settings.json` で次の段の構造で起動する。

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "~/.claude/hooks/git-flag-bypass-guard.sh"
          }
        ]
      }
    ]
  }
}
```

既定値は `commit` の小命令の禁止のみ。 利用者の側で `reset` や `push` も禁止したい場合、 環境変数 `CC_GIT_FLAG_DENY="commit,reset,push"` で一覧を指定できる。 空の値で無効化。

## 試験の結果

35件の試験の項目で全件通過。 対応の事例は次の通り。

- 中核の素通りの事例 6件 (`-C`、 `--git-dir=`、 `-c`、 `--no-pager`、 別の素の形の `--git-dir`、 plain commit)
- 正常な命令の素通り 3件 (`git log`、 `git -C /repo status`、 非gitの命令)
- 環境変数の前置きの検出 1件 (`GIT_AUTHOR_NAME=x git -C /repo commit`)
- 絶対の経路の `/usr/bin/git` の検出 1件
- 設定の検証 3件 (1件の値、 comma区切りの一覧、 空で無効化)
- 境界の事例 2件 (空の命令、 bare git)

実装の中身と試験の場の中身は GitHub の [yurukusa/cc-safe-setup](https://github.com/yurukusa/cc-safe-setup/blob/main/examples/git-flag-bypass-guard.sh) の `examples/git-flag-bypass-guard.sh` に置いた。 MIT の許諾、 自由に再配布と改変が可能。

## 限界

この安全装置は単一の命令の段のみを見る。 次の3件の素通りの経路は対応外。

- `bash -c "cd /repo && git commit"` の包みの形
- `sh -c '...'` や `eval` の包みの形
- `&&`、 `;`、 `||` の連結の形

これらは別の安全装置で被覆する必要がある。 cc-safe-setupでは `deny-bypass-detector.sh` が包みの形の検出を、 `compound-inject-guard.sh` が連結の形の検出を担う。 3件の安全装置を全件で起動すると、 禁止の規則の素通りの主要な3つの経路 (旗の位置、 包み、 連結) を全件で予防できる。

加えて、 PreToolUseの安全装置は子の作業者 (`Agent` の道具で起動される) の中の命令には届かない。 子の作業者の中で `git -C /repo commit` を実行された場合、 親の側の禁止の規則も親の側の安全装置も発火しない。 この経路は別の系統の問題で、 詳しくは #55488 の系統の起票群を参照。

## まとめ

公式の禁止の規則の照合は、 命令の文字列の頭の部分の一致のみを見ている。 `git -C <path> commit` のように旗が頭に挿入された形では照合が失敗し、 利用者の意図した禁止が静かに素通りする。

基幹の起票 #18613 は今も開いたままで、 公式の修正を待つ間、 利用者の側で予防の安全装置を起動する経路がある。 中身は素直な素の段の解析で、 35件の試験で確認した動作の段。 仕組みの段の理解と単純な実装で、 安全装置の機能の段の信用を取り戻せる。

似た系統の素通りは、 包みの形と連結の形にも存在する。 3件の安全装置を組み合わせると、 主要な3つの経路を全件で予防できる。 子の作業者の中の命令の経路の問題は、 別の系統で対応の段を進める必要がある。
