Claude Codeの中で `grep -E '(^|[-_.])(dev)([-_.]|$)'` と書いたシェルの命令が、 マッチするはずの入力で `NO MATCH` を返した時、 利用者が最初に疑うのは正規表現の書き間違いです。 中身を見直して、 そのまま外のシェルで叩くと普通に動く。 おかしいと思って色々調べた末に、 Claude Codeの中の `grep` が、 利用者の指定した `-E` を黙って `-G` に置き換えていた、 という話です。

## 何が起きるか

Claude Codeのシェル環境は、 `grep` をbashの関数で上書きしています。 `declare -f grep` を実行すると、 関数の中身が読めます。 関数の本体は `exec -a ugrep "$_cc_bin" -G --ignore-files ... "$@"` のような形で、 Claude Code本体に経路を変えて、 `ugrep` の名前で起動しています。

問題は、 `-G` が無条件で渡されている点です。 `-G` は基本の正規表現(BRE)で動作する旗で、 利用者が `-E` で指定した拡張の正規表現(ERE)が、 `-G` の上書きで黙って別の解釈になります。 誤りの合図は何も出ません。 結果だけが沈黙で違います。

## 1分の再現

Claude Codeのセッションの中で、 次のひとつめの命令を実行してみてください。

```bash
declare -f grep | grep '\-G'
# 出力: 関数の本体に -G が含まれる行が表示される
```

次にふたつめの命令で、 EREの拡張の構文の動作を確認します。

```bash
echo "requirements-dev.txt" | grep -qE '(^|[-_.])(dev)([-_.]|$)' && echo "MATCH" || echo "NO MATCH"
# 期待の出力: MATCH (EREのalternationで dev の周りの分離の文字を判定)
# 実際の出力: NO MATCH (BREの解釈で | が文字通りの縦棒の文字になる)
```

3つめの命令で、 文字の集合の構文も同じ症状である事を確認できます。

```bash
echo "foo-bar" | grep -qE '[-_]' && echo "MATCH" || echo "NO MATCH"
# 期待の出力: MATCH
# 実際の出力: NO MATCH
```

非対話の子のシェル(`$BASHPID != $$`)の場の中では、 終了の値が `127` で返ります。 利用者の側からは「コマンドが見つかりません」 のような失敗の合図に見えます。 実際は正規表現の解釈の失敗ですが、 表面の合図は別の原因を指します。

公式の起票はこちらです: [issue#59517](https://github.com/anthropics/claude-code/issues/59517)。 5月15日に起票され、 投稿者の段で再現の手順と中身の整理が完成しています。

## なぜ起きるか

Claude Codeは、 シェルの中の `grep` の経路を、 セッションの安全と速度の都合で、 自社の `ugrep` の経路に置き換えています。 置き換えの仕組みは、 bashの関数の上書きで、 利用者の対話のシェルに既定の関数として組み込まれます。

関数の本体は、 元の `grep` の引数を `"$@"` で受け取りますが、 自社の `ugrep` の起動の段で `-G` の旗を無条件で前置します。 EREの利用者が `-E` を渡しても、 後ろの `-G` が後勝ちで、 BREの動作になります。 利用者の引数の意図と、 実装の側の置き換えの意図の間に、 沈黙の食い違いが残る構造です。

## 4つの回避の手段

ひとつめは、 `command grep -E ...` の形で、 関数の上書きを素通りする手段です。 `command` のbashの組み込みの命令は、 関数の解決を飛ばして、 PATHの上の実体の `grep` に直接届きます。 副作用は最小です。

```bash
echo "requirements-dev.txt" | command grep -qE '(^|[-_.])(dev)([-_.]|$)' && echo "MATCH" || echo "NO MATCH"
# 出力: MATCH (PATHの上の本物の grep が呼ばれる)
```

ふたつめは、 絶対の経路で `grep` を呼ぶ手段です。 `/usr/bin/grep -E ...` や `/bin/grep -E ...` で、 関数の解決を完全に避けます。 環境の差で本物の `grep` の場所は違うので、 `command -v grep` で本物の場所を一回だけ確認しておくと安心です。

```bash
/usr/bin/grep -qE '(^|[-_.])(dev)([-_.]|$)' <<<"requirements-dev.txt" && echo "MATCH" || echo "NO MATCH"
```

3つめは、 BREの構文に書き直す手段です。 `\|` の経路で代替の構文をBREでも使える環境なら、 `grep '\(^\|[-_.]\)\(dev\)\([-_.]\|$\)'` の形で同じ意味を表現できます。 ただし、 BREの代替の構文の有無はGNU grepの版に依存し、 移植の場の段で動かない可能性があります。 EREへの依存が深い場の中身では、 ひとつめの `command` の手段が現実的です。

4つめは、 `awk` や `sed` や `perl` の他の道具に経路を変える手段です。 同じ意図の検索の中身を `awk '/(^|[-_.])(dev)([-_.]|$)/' <<<"requirements-dev.txt"` で書き直す事で、 `grep` の関数の上書きの問題を完全に回避します。 短い検索の中身では、 `awk` の方が読みやすい場合もあります。

## 利用者の側で何が学べるか

この事例の中身は、 利用者が指定した引数(`-E`)が、 実装の側で沈黙で別の値(`-G`)に置き換わる構造です。 表面の動作の表示は何も誤りの合図を出しません。 失敗の原因の表示も別の場所(終了の値の `127` の「コマンドが見つかりません」 風の合図)を指します。 利用者の側が、 引数の意図と実装の側の挙動の間の沈黙の食い違いを発見するためには、 自分の手で `declare -f grep` のような検証の手順を実行する必要があります。

似たような構造の事例は、 直近24時間で他にも観察されています。 同名の代理の表示が近い側の中身でも、 実行時は祖先の側の中身が起動する事例([起票#59541](https://github.com/anthropics/claude-code/issues/59541))。 `/clear` の中身の説明と実態の挙動が食い違う事例([起票#59534](https://github.com/anthropics/claude-code/issues/59534))。 `/compact` の境界をまたぐとOpusの選択が沈黙でSonnetに戻る事例([起票#59516](https://github.com/anthropics/claude-code/issues/59516))。

これらは個別の事故ではなく、 利用者の意図と実装の挙動の間の沈黙の食い違いの構造の系統です。 5月22日に発売する事例集 [Claude Code Claim-Verify Handbook](https://yurukusa.gumroad.com/l/claim-verify-handbook) は、 同型の構造の104件の事例(本文15件と継続の証拠89件)と、 利用者の側の14件の防衛の手順を整理しています。 試し読みの場は [この場所](https://gist.github.com/yurukusa/6dd608049064ed66c54f1a545a7b47a8) で読めます。

## まとめ

- Claude Codeのシェル環境の `grep` は、 bashの関数で上書きされ、 自社の `ugrep` に経路が変わる
- 関数の本体は無条件で `-G` を渡し、 利用者の `-E` を黙って上書きする
- 結果は沈黙で違う、 終了の値は `127` で別の失敗に見える
- 回避は `command grep -E`、 絶対の経路、 BREの書き直し、 他の道具の4手段
- 同型の構造の食い違いは他にも観察されており、 利用者の側で検証の手順を持つ事の重要性が増している
