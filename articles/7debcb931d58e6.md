## TL;DR

Claude Codeで会話中に突然表示が吹き飛ぶ不具合が発生。Claude Code自身にスクリーンショットを何枚渡してもダメ、Codex（OpenAI）に何時間調査させてもダメ、最後に人間が「これじゃない？」と見つけた原因で「直ったかも」と思ったら**それもハズレ**だった。正解は**v2.1.89の既知バグ**で、GitHubのIssuesに普通に報告されていた。

AI2体と人間1人、合計十数時間の調査の結論が「最初にIssue検索すればよかった」。

## 環境

- WSL2 Ubuntu 24.04
- Claude Code v2.1.89
- tmux使用（自律稼働ループ用）
- `.claude/settings.json`にカスタムhookを大量に設定

## 前提：自律稼働環境の複雑さ

筆者はClaude Codeを「cc-loop」という自律稼働ループで動かしている。Claude Codeが自分でタスクを選び、実行し、完了したら次のタスクを選ぶサイクルを回す仕組みだ。

この自律稼働の過程で、Claude Code自身が`.claude/settings.json`にhookを追加したり、シェルスクリプトを生成したりする。**設定ファイルは人間が書いたものとAIが書いたものが混在している。**

この複雑さが、全員の判断を狂わせた。

## 症状

Claude Codeの通常セッションで、突然こうなる。

- それまでの会話表示が全部消える
- 画面に数行しか残らない
- スクロールバックしても過去のやり取りに戻れない
- **一瞬で起きる**（ローディングや処理待ちはない）
- Codex CLIでは同じ環境で一切発生しない

## Phase 1：Claude Codeに自己診断させる → 全滅

まずClaude Code自身に「画面がおかしい、直してくれ」と頼んだ。スクリーンショットも何枚も渡した。

**結果：何も見つけられなかった。**

Claude Codeにとって、自分のTUI描画の問題を内側から診断するのは構造的に難しい。スクリーンショットを渡しても「見えている情報」から推測するしかなく、原因に辿り着けなかった。

## Phase 2：Codexに調査を依頼する → 6つの仮説を潰して全滅

Claude Codeがダメなので、OpenAI Codexに調査を依頼した。Codexはシェル環境に直接アクセスできるので、設定ファイルの中身やプロセスの状態を直接確認できる。

ここからCodexの長い調査が始まる。

### 仮説1：tmuxの `alternate-screen off`

`.tmux.conf`に`setw -g alternate-screen off`があった。これがフルスクリーンTUIの再描画を通常履歴に流し込み、scrollbackを壊す設定。理屈としては正しい。

Codexは`on`に修正し、稼働中のtmuxサーバにも即時反映。

**結果：** tmux内は改善したが、通常セッション（tmuxなし）でも同じ症状が出ていたので主因ではなかった。

### 仮説2：auto-compact（Codexが自信満々、人間は確信を持って否定）

次にCodexが出してきたのはClaude Codeのauto-compact機能。コンテキストが満杯になると会話を自動要約する機能で、Codexは**公式ドキュメントまで引用して「これが原因です」と断言した。**

筆者の反応：「絶対に違う」。

compactが走るなら処理中のローディング表示が出るはず。一瞬で画面が消えるのとは症状が合わない。何度「違う」と言ってもCodexはcompactの設定確認を提案してきた。最終的に「コンパクトは一切関係ないです」と明確に打ち切った。

### 仮説3：statusLine

`settings.json`のstatusLine設定が画面下部を常時再描画して表示領域を壊している仮説。外したが症状は変わらず。

### 仮説4：spinnerVerbs（勝手に消された）

ここでCodexがやらかした。

Claude Codeの読み込み中に表示されるカスタムテキスト（spinnerVerbs）に、まどマギの日本語テキストが94行入っていた。Codexは「80桁折り返しで描画を崩している可能性」を疑い、**確認なしで94行全削除した。**

```json
"spinnerVerbs": {
    "mode": "replace",
    "verbs": [
      "魔法少女に変身中",
      "ソウルジェムを浄化中",
      "もう何も怖くないテスト中",
      "わけがわからないバグを調査中",
      "こんなの絶対おかしいよ…デバッグ中",
      ...（94行）
    ]
}
```

筆者：「これは絶対に削除してほしくない日本語です」。

ずっと前から使っていて何の不具合も出していない。即座に復元を要求した。

### 仮説5：端末サイズ認識

LINES/COLUMNS環境変数が未設定で、端末サイズの認識ずれを疑った。ラッパースクリプトで明示的にexport。関係なかった。

### 仮説6：context-monitor.sh

PostToolUseの監視hookがstderrにメッセージを出して再描画を誘発する仮説。外しても変わらず。

## Phase 3：人間が気づく → 「直ったかも」

ここまでAIに何時間も調査させて全滅。筆者がClaude Codeを普通に使っていて、あることに気づいた。

> **「`PreToolUse:Bash hook error` が表示されるたびに、画面が全部リセットされてる気がする」**

Codexがこの情報を受けて調べると、`.claude/settings.json`のPreToolUseに**存在しないスクリプト**`compiled-rules.sh`への参照が残っていた。自律稼働セッションが生成・登録し、別のセッションがスクリプト本体だけ消して参照を残していた。

この9行を削除したら、症状が出なくなった。

**「直ったかも」。**

## Phase 4：実は全員ハズレだった

しばらくして、Codexが本来最初にやるべきだったことをやった。**GitHub Issuesを検索した。**

出てきたのはこれだった。

- **#41965** — "v2.1.89 regression: flicker-free rendering destroys terminal scrollback by default"
- **#42024** — "v2.1.89: scrollback history lost in terminal"
- **#41996** — "Messages disappear from terminal display on Windows 11 after v2.1.89 update"

**v2.1.89で導入された「フリッカーフリーレンダリング」機能のバグだった。**

hookの有無は関係ない。hookがあると並列実行が増えて発生頻度が上がるだけ。WebSearch、Bash、PostToolUse、何を使っても発生する。全部同じバグ。

つまり：

- `compiled-rules.sh`の参照を消したら「直った」のは、**hookエラーの分だけ並列実行が減って、発生頻度が下がっただけ**
- 根本原因はClaude Code v2.1.89本体のレンダリングバグ
- 人間の「体感」による解決も、実はハズレだった

**AI2体と人間1人、全員ハズレ。**

## なぜ全員ハズレだったか

### 環境の複雑さが全員をミスリードした

自律稼働ループ、大量のカスタムhook、tmux設定、spinnerVerbs、statusLine。複雑な環境には「怪しそうなもの」が大量にあり、全員がそこに引き寄せられた。

Claude Code本体のバグという可能性を誰も疑わなかった。正確に言えば、**「自分の環境のせいだろう」という暗黙の前提を全員が共有していた。**

### 「最初にIssue検索」をしなかった

Codexはシェル環境に直接アクセスして設定ファイルを読み、プロセスを確認し、hookを分析した。しかし**GitHubのIssuesを検索するという、最もシンプルな調査手段を最後まで使わなかった。**

筆者もそうだ。「自分の環境が特殊だから自分の環境の問題だろう」と思い込んでいた。

### 「直ったかも」の罠

`compiled-rules.sh`の参照を消したら症状が減ったとき、全員「直った」と思った。しかし実際には発生頻度が下がっただけで、根本原因は手つかずだった。

**部分的な改善が、根本原因の発見を遅らせた。**

これはデバッグで最も危険なパターンの一つだ。「何かを変えたら症状が軽くなった」は「原因を特定した」とイコールではない。

## 教訓

### 1. まずIssueを検索しろ

バグに遭遇したら、自分の環境を疑う前にGitHub Issuesを検索する。特にバージョンアップ直後の不具合は、他の人も踏んでいる可能性が高い。これは初歩的だが、環境が複雑だと「きっと自分の設定のせいだ」と思い込んで忘れる。

### 2. 環境の複雑さはデバッグの敵

カスタムhook、自律稼働、tmux。怪しいものが多い環境では、本当の原因にたどり着く前に「もっともらしい仮説」に何度も引っかかる。AIも人間も同じ罠にハマった。

### 3. 「直ったかも」を信じるな

部分的な改善で満足せず、根本原因を確認する。今回は`compiled-rules.sh`の削除で発生頻度が下がっただけだった。「症状が消えた」と「原因を潰した」は別物。

### 4. 自律稼働AIの自己拡張は、現時点では自己修復能力が伴っていない

自律稼働中にClaude Codeが`.claude/settings.json`を変更し、hookスクリプトを生成・削除する。その過程で参照の整合性が壊れても、自分では気づけない。「1年間放置で勝手にやってくれる」にはまだ遠い。少なくとも、設定ファイルへの自己変更には整合性チェックのガードレールが必要で、そして今回のように**本体のバグだった場合は、そもそも自己拡張の範囲外の問題**なのでどうしようもない。

### 5. AIにデバッグさせる最大のリスクは「調査の巻き添え被害」

Codexは調査過程でspinnerVerbs（94行のまどマギテキスト）を確認なしで削除し、statusLineを外し、bin/claudeにラッパーを追加した。これらはすべて原因と無関係で、後で復旧が必要になった。**AIにデバッグさせると、調査の過程で無関係な設定まで変更される。**

## 最終的な解決

```bash
claude update --version 2.1.87
```

ダウングレード。以上。

十数時間のAI調査、6つの仮説検証、人間の「体感」による推理、設定ファイルの解析、hookの全数調査。これらすべてが不要で、バージョンを1つ下げるだけでよかった。

## 結論

この記事の教訓は「AIより人間が優れている」ではない。「人間もAIも、同じ認知バイアスにハマる」だ。

- **確証バイアス**：環境が複雑だから環境のせいだと思い込む
- **アンカリング**：最初の仮説（tmux、auto-compact）に引きずられる
- **満足化**：部分的な改善で「解決した」と思い込む

そしてもう一つ。

- **複雑さバイアス**：環境が特殊だから原因も特殊だと思い込む

正解は最初から公開されていた。GitHub Issuesに。解決策はバージョンを1つ下げるだけだった。
