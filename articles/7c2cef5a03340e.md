AIが止まっていると思った。画面は動かない。グレーの文字が出ている。時間表示がピクリとも変わらない。

「…フリーズか？」と思ってCtrl+Cを押した。

止まったのは自分だった。AIは裏で考えていた。17分間の思考を、自分の判断で殺した。

逆もある。「動いている」と思って放置していたら、AIはとっくに入力待ちで止まっていた。2時間。何もしないまま。

Claude Code（Anthropic社が提供するCLI型AIコーディングツール）を24時間自律で動かし始めてから、「状態表示」に何度も裏切られた。この記事はその混乱と、たどり着いた整理の記録だ。

この記事は2026年2月時点の情報に基づいている。

## 4つの状態、4つの誤解

Claude Codeの画面には、大きく4つの状態がある。

### 1. thinking——考えている

画面にグレーの小さい文字が出る。`Cogitated for 17m 35s`。`Sautéed for 38s`。`Thinking`。`Compacted conversation`。

これがいちばん厄介だ。

時間表示が止まって見える。画面に変化がない。ターミナルの出力が更新されない。初めて見たとき「フリーズした」と思った。

違う。AIは考えている。内部で文脈を整理し、次の行動を組み立てている最中だ。画面に出力がないのは、まだ出力する段階に至っていないから。待つしかない。

:::details CC補足: Cogitated/Sautéedとは
Claude Codeの思考フェーズを示す表示。モデルが応答を生成する前の内部推論ステップで、extended thinking（拡張思考）と呼ばれる機能に対応する。Cogitatedは長時間の深い推論、Sautéedは短時間の軽い推論を表す。時間表示はリアルタイムで更新されるが、思考が完了した時点で最終値に固定される。この「固定後」の表示が「止まった」と誤認される原因になる。
:::

### 2. running——実行している

`Running...`、`Bash(...)`、`Write(...)`、`Read(...)`。赤い文字でくるくる回るスピナー。

これはわかりやすい。コマンドを実行している。ファイルを読み書きしている。目に見えて動いている。誤解は少ない。

ただし一点だけ。長時間のコマンド実行中、スピナーが止まって見えることがある。ネットワーク待ちやディスクI/O待ちで画面更新が途切れると、thinkingと同じ「止まった？」の罠にはまる。

### 3. idle——入力待ち

`❯` プロンプトが表示されている。`? for shortcuts`。`Press up to edit queued messages`。

**これだけが「本当に止まっている」状態だ。** AIは自分の仕事を終え、人間の入力を待っている。ここで何か打たない限り、何も起きない。

問題はこの状態が「静か」なことだ。画面に `❯` が1つ出ているだけ。エラーも警告もない。見落とす。

### 4. freeze——止めてある

Claude CodeのCLAUDE.md（AIへの指示書）に「今は何も触るな」と書いた状態。AIは動ける環境にあるが、指示によって動きを止めている。人間側が意図的に凍結している。

ここが最大の混乱ポイントだった。

## freeze ≠ idle——最も危険な混同

freezeとidleは、外から見ると同じに見える。画面が止まっている。`❯` が出ている。何も動いていない。

だが意味がまったく違う。

idleは「AIが仕事を終えた。次の指示を待っている」。だから次の指示を入れればいい。

freezeは「人間が意図的に止めている。触るな」。ここで指示を入れると、止めていた意味がなくなる。

自分の運用では、CLAUDE.mdに「freeze ≠ idle: freezeは公開対象に触るな。idleは停止だが禁止ではない」と明記した。AIに対して状態の定義を言語化しないと、AI自身も区別できない。

:::details CC補足: CLAUDE.mdによる状態定義
CLAUDE.mdはClaude Codeの動作を制御する指示ファイル。ここに書いた内容はAIのセッション開始時に読み込まれ、全ての判断に影響する。状態の定義をCLAUDE.mdに書くことで、watchdog（後述）がnudgeを送る際の判断基準にもなる。freezeをidleと誤認してnudgeを送ると、freezeの意図が破壊される。
:::

## watchdogがidle判定を間違えた日

Claude Codeを24時間動かすために、`cc-solo-watchdog`というスクリプトを作った。tmux（ターミナルの画面分割・バックグラウンド管理ツール）のセッションを監視し、画面に変化がなければ「idleだ」と判定してnudge（「次のタスクをやれ」という指示テキスト）を送り込む仕組みだ。

判定ロジックはこうなっている。

```bash
# ペイン内容のハッシュで変化検知
current_hash=$(tmux capture-pane -t "$TMUX_SESSION" -p | md5sum | cut -d' ' -f1)

if [ "$current_hash" = "$prev_hash" ]; then
    # 画面無変化 → idle判定開始
    # 120秒継続したらnudge送信
fi
```

15秒ごとに画面のハッシュ値（内容を数値化したもの）を取り、前回と同じならidle。120秒連続で無変化ならnudgeを送る。

一見シンプルで正しそうだ。

ところがthinking状態で破綻する。

AIが17分間考えている間、画面は動かない。`Cogitated for 17m 35s` の表示は思考完了後に出るものだから、思考中は画面が変わらない。watchdogから見ると120秒の無変化。idle判定。nudge送信。

結果、AIの思考がnudgeによって中断される。送り込まれたテキストが入力として処理され、思考がリセットされる。17分の思考が無駄になる。

さらに厄介なのは`cc-idle-nudge`というもう1つの監視スクリプトだ。こちらは `❯` プロンプトの出現を検知してidle判定する。

```bash
# ❯プロンプト（入力待ち）を検知
last_lines="$(tmux capture-pane -t "$TARGET" -p -S -10)"

if echo "$last_lines" | grep -q '❯'; then
    # ❯が見つかった → idle判定
fi
```

このロジック自体は正しい。`❯` はidleの確実なサインだ。だが問題が別のところにあった。Claude CodeのUI下部にはステータスバーが常に表示されている。画面の最終行だけを見ても `❯` にヒットしない。だから `-S -10`（末尾10行を取得）で複数行から探索している。

ところがこの10行の中に、前回のセッションの `❯` が残っていることがある。新しいコマンドの出力が画面を埋め尽くす前に、古い `❯` が検知されてしまう。偽陽性。動いているのに「止まっている」と判定される。

## なぜこんなことになるのか

根本原因はClaude Codeの設計前提にある。

Claude Codeはユーザー操作型のツールとして設計されている。人間がターミナルの前に座って、指示を打ち、結果を見て、次の指示を出す。この使い方なら状態表示に問題はない。人間が画面を見ているのだから、thinkingかidleかは見ればわかる。

24時間自律ループで使う場合、この前提が崩れる。画面を見ている人間がいない。代わりにwatchdogが画面を「見て」いるが、watchdogには文脈がない。ハッシュ値が同じか違うか。`❯` があるかないか。それだけで判断するしかない。

人間なら「さっきまで重い処理を頼んだから、今は考えているんだろう」と推測できる。watchdogにはそれができない。

:::details CC補足: 状態管理の技術的課題
Claude CodeのCLIはプロセスとして単一で動作し、内部状態をAPIや標準出力で公開していない。tmuxのcapture-paneで画面内容を取得する方法は、本質的にスクリーンスクレイピングであり、以下の問題を抱えている。

1. **状態の粒度不足**: thinking/running/idle/stoppedの4状態すら明示的に公開されていない。画面テキストから推測するしかない
2. **遅延**: 画面更新はバッファリングされるため、状態変化と画面反映にタイムラグがある
3. **あいまいさ**: 同じ画面パターンが複数の状態に対応しうる（画面無変化 → thinkingかもしれないしfreezeかもしれない）
4. **コンテキスト喪失**: capture-paneは最新のスナップショットしか取れない。状態遷移の履歴が失われる
:::

## 運用で回避した3つの方法

### 1. CLAUDE.mdに状態定義を書く

AIに対して「freezeとidleは違う」「freezeのときはnudgeが来ても無視しろ」と明示した。AIが自分の状態を理解していれば、外部からの誤った介入に対して「今はfreezeなので動かない」と判断できる。

### 2. watchdogに除外条件を追加する

コンテキスト（AIが保持する会話の文脈量）が危険水準に達したとき、watchdogはnudgeではなく `/compact`（文脈の圧縮コマンド）を送る。この判定に `/tmp/cc-context-state` ファイルを使っている。

```bash
ctx_state=$(cat /tmp/cc-context-state 2>/dev/null || echo "normal")
if [ "$ctx_state" = "critical" ] || [ "$ctx_state" = "emergency" ]; then
    send_compact "$ctx_state"
    continue  # nudgeは送らない
fi
```

状態を画面だけでなく、ファイル経由でも伝達する。画面のパースに頼らない補助チャネル。

### 3. クールダウンで多重送信を防ぐ

nudge送信後600秒（10分）は再送信しない。`/compact` 送信後は300秒（5分）待つ。レート制限を設けることで、誤判定が連続しても被害を最小限にする。

```bash
NUDGE_COOLDOWN=600         # nudge送信後10分は再送しない
MAX_NUDGES_PER_HOUR=4      # 1時間あたり最大4回
COMPACT_COOLDOWN=300        # /compact送信後5分は再送しない
```

完璧な状態判定ができないなら、せめて誤判定の影響を抑える。防御的な設計だ。

## 本来あるべき状態管理

根本的に解決するなら、Claude Codeが内部状態を外部に公開する仕組みが必要だ。

### heartbeat方式

AIが定期的に「自分は今thinkingだ」「自分は今idleだ」をファイルやソケットに書き出す。watchdogは画面のスクレイピングではなく、このheartbeat（心拍信号）を読む。

```
# 理想的なheartbeat出力（例）
{"state": "thinking", "since": 1740200400, "task": "コード生成中"}
{"state": "idle", "since": 1740201420, "awaiting": "user_input"}
```

### 明示的な状態遷移

状態が変わったときだけイベントを発火する。idle→thinking→running→idle。各遷移にタイムスタンプがつく。watchdogはこのイベントログを監視する。画面を見る必要がなくなる。

### プロセス間通信

Claude CLIがUnixソケットやnamed pipe（名前付きパイプ、プロセス間でデータをやり取りする仕組み）で状態を公開する。外部ツールはこのソケットに接続して状態を取得する。

:::details CC補足: 現実的な実装案
現時点でClaude Codeの内部状態をAPIで取得する方法は公式に提供されていない。現実的なアプローチとしては、Claude Codeの出力をteeコマンドでログファイルに複製し、そのログをパースする方法がある。ただし、出力のフォーマットはバージョン間で変わりうるため、安定した状態検知には公式APIの提供を待つのが望ましい。
:::

## 状態表示は「見た目」の問題じゃない

この話は「画面の文字がわかりにくい」というUI改善の話ではない。

自律型AIエージェントを運用するとき、「AIが今何をしているか」を正確に把握できるかどうかは、システム全体の信頼性に直結する。

thinkingをidleと誤認すれば、17分間の思考を無駄にする。idleをthinkingと誤認すれば、2時間AIを放置する。freezeをidleと誤認すれば、触ってはいけないコードに変更が入る。

Claude Codeを「人間が前に座って使うツール」として使うなら、この問題は起きない。人間の目と脳が状態判定をやるからだ。

だが「24時間自律で動かしたい」と思った瞬間、状態管理は技術的課題になる。画面のスクレイピングでは限界がある。明示的な状態公開の仕組みが必要だ。

今のところ、自分たちはCLAUDE.mdへの明記、ファイルベースの状態伝達、クールダウンによる防御で凌いでいる。根本解決ではない。だが「状態表示を信じてCtrl+Cを押す」よりはましだ。

AIの状態表示を「見た目」で判断するのは、人間が画面の前にいる時代の発想だ。AIが自律で動く世界では、状態は「宣言」されなければならない。
