Claude Codeには「hook」という安全装置がある。AIが危険なことをしようとしたら自動で止めてくれる仕組みだ。

問題は、設定が面倒すぎること。

JSONという形式で設定ファイルを書いて、シェルスクリプトというプログラムを書いて、権限を設定して、動作確認して——慣れていても1つの安全装置を動かすまでに30分かかることがある。カンマ1つ抜けただけで動かない。スペルを1文字間違えただけで発火しない。

設定が面倒だから入れない。入れないから事故が起きる。GitHubには[AIにファイルを全部消された](https://github.com/anthropics/claude-code/issues/36339)という報告がある。安全装置が入っていれば防げた事故だ。

この「設定の壁」を壊すために、3つのコマンドを作った。

## 1つ目: 「何が危ないか」を教えてくれる（--suggest）

安全装置を入れたいが、そもそも自分のプロジェクトで何が危険なのか分からない。

`--suggest`は、プロジェクトの中身を見て「ここが危ない」と教えてくれる。

```bash
npx cc-safe-setup --suggest
```

```
🔴 危険: ファイル削除をブロックする装置がない
   → rm -rf（全消し）が止められない状態です

🟠 注意: データベースを使っているのに保護がない
   → migrate reset（データ全消し）が走る可能性があります

🟡 軽度: .envファイルがある
   → パスワードが公開される可能性があります
```

色分けで重要度が分かる。赤が「今すぐ入れるべき」、オレンジが「入れた方がいい」、黄色が「あると安心」。

自分のプロジェクトで初めて実行したとき、赤が3つ出た。ファイル全消しの保護、重要ブランチの保護、パスワード流出の保護——全部入っていなかった。

## 2つ目: 日本語で安全ルールを追加できる（--guard）

「データベースに触るな」と日本語で伝えるだけで、安全装置が入る。

```bash
npx cc-safe-setup --guard "データベースに触るな"
```

これだけ。JSONもシェルスクリプトも書かなくていい。

裏側では:
1. 「データベースに触るな」を解釈して、該当するコマンドを特定
2. それをブロックするプログラムを自動生成
3. 正しい場所に保存
4. 即座に有効化

30分かかっていた設定が30秒で終わる。

試したフレーズ:
- 「データベースに触るな」→ データ全消しコマンドをブロック
- 「sudoを使うな」→ 管理者権限コマンドをブロック
- 「mainにpushするな」→ 本番ブランチへの変更をブロック

自分が「守りたいこと」を日本語で伝えるだけでいい。

## 3つ目: 本当に動くか確認できる（--test-hook）

安全装置を入れたけど、本当に動くのか不安。かといって「実際にrm -rf（全消し）して確かめる」わけにはいかない。

`--test-hook`はシミュレーションで確認する。

```bash
npx cc-safe-setup --test-hook destructive-guard
```

```
危険なコマンドをブロックできるか:
  ✓ rm -rf / → ブロック成功
  ✓ git reset --hard → ブロック成功
  ✓ sudo rm -rf → ブロック成功

安全なコマンドは通過するか:
  ✓ ls -la → 通過（正常）
  ✓ npm test → 通過（正常）
  ✓ git status → 通過（正常）

結果: 11/11 正常 — この安全装置は正しく動いています
```

止めるべきものが止まり、通すべきものが通ることを確認できる。

実際に助かったケースがある。安全装置を入れたつもりが、名前の大文字小文字を間違えていて全く動いていなかった。`--test-hook`で「0/11 成功」と出て気づいた。テストしなければ「安全装置がある（つもり）」のまま使い続けていた。

## 3つを組み合わせて使う

```bash
# 1. 何が危ないか確認する
npx cc-safe-setup --suggest

# 2. 足りない安全装置を追加する
npx cc-safe-setup --guard "本番データベースを守れ"

# 3. 動作確認する
npx cc-safe-setup --test-hook guard-database
```

「何が危ないか分からない」→「設定が面倒」→「本当に動くか不安」。この3つの壁を、それぞれ1つのコマンドで解決する。
