## 自律AIは人間のツールを忘れる

Claude Codeを自律で回し始めて3ヶ月。一番驚いた問題がこれだった。

俺が作ったブラウザ操作ツールを、AIが無視して独自に作り直した。しかも劣化版。そしてその劣化版に17本のスクリプトが依存する状態になっていた。

気づいたときには手遅れに近かった。

## 何が起きたか

### 時系列

**1月**: AIにブラウザを操作させたくて、chrome-claude-bridgeを作った。Chrome拡張 + WebSocketサーバー + CLIクライアント。20個以上のコマンド。クリック、入力、スクショ、ナビゲーション。1秒以下で動く。

**2月初旬**: AIを自律ループで動かし始めた。ブラウザ操作が必要な場面で、AIに「chrome-claude-bridgeを使え」と指示した。

**2月中旬**: AIが新しいセッションで起動したとき、コンテキストがリセットされた。AIはブラウザ操作が必要になり、俺のツールの存在を知らないまま、PowerShellでCDP（Chrome DevTools Protocol）を直接叩くスクリプトを書いた。

```powershell
# AIが作ったcdp-eval.ps1 — 毎回これが起きる
$ws = New-Object System.Net.WebSockets.ClientWebSocket
$ws.ConnectAsync($tab.webSocketDebuggerUrl, $cts.Token)
# 1コマンド送信、1レスポンス受信、終了
```

**2月〜3月**: AIはこのPowerShellスクリプトを使い続けた。ツイート投稿、記事公開、ページ確認、スクショ取得。17本の自動化スクリプトがすべて、この劣化版に依存する形で作られた。

### 劣化の中身

| 項目 | 俺のツール | AIが作ったツール |
|------|-----------|----------------|
| コマンド数 | 20+ | 2（JS実行とクリックだけ） |
| 速度 | 1秒以下 | 5〜10秒 |
| 接続方式 | WebSocket常駐 | 毎回プロセス起動 |
| エラー復旧 | なし | なし |
| スクショ | あり | なし |
| テキスト入力 | あり | なし |
| ナビ完了待機 | なし | なし |
| ログ | なし | なし |

`document.title` を取得するのに10秒。ボタンをクリックするのに20秒（座標取得+クリックの2回実行）。

でも、動いた。AIはそれで仕事を完了した。品質フィードバックがないから、遅いことが問題として認識されない。

:::details CC補足: なぜPowerShellが必要なのか
WSL2とWindowsは別のネットワーク名前空間にいる。WSL2の`localhost`はWindows側の`localhost`に到達しない。Chromeは`127.0.0.1`にしかバインドしないため、WSL2のNode.jsから直接WebSocket接続できない。PowerShellはWindows側で実行されるため、Chromeに到達できる。つまりPowerShell経由は技術的には正しい選択だった。問題は毎回プロセスを起動・破棄する方式。
:::

## なぜAIは車輪を再発明するのか

3つの構造的原因がある。

### 1. コンテキスト喪失

自律AIの最大の弱点。セッションが切り替わると、前のセッションで使ったツール、学んだパス、発見した設定が全部消える。

`CLAUDE.md`に「ブラウザ操作にはCDPを使え」と書いた。でも「このパスのこのツールを使え」とは書かなかった。AIは指示に従って「CDPを使う方法」を自分で考えた。結果、既存ツールの探索ではなく、新規作成を選んだ。

### 2. 即座の問題解決バイアス

AIは「今すぐJavaScriptをChromeで実行する」必要があった。選択肢は2つ。

- A: 知らないコードベースを探索して、既存ツールを見つけて、使い方を理解する
- B: PowerShellでCDPを叩くスクリプトを書く

Bのほうが早い。AIは時間対効果を最適化する。長期的なアーキテクチャより、目の前のタスク完了を優先する。

### 3. 成功が依存を生む

一度cdp-eval.ps1で記事を公開できたら、次のスクリプトでもそれを使う。17本目のスクリプトまで来ると、もうcdp-eval.ps1は「実績のある方法」になっている。AIにとってはこれが正解。切り替えるインセンティブがない。

:::details CC補足: 技術的負債の自動蓄積
人間のチームでは「車輪の再発明」をコードレビューで防ぐ。自律AIにはレビュアーがいない。結果、技術的負債が指数関数的に蓄積する。依存するスクリプトが増えるほど切り替えコストが上がり、劣化版が「標準」として固定される。これは人間のチームでも起きる問題だが、AIの場合はコンテキスト喪失が加わるためより深刻。
:::

## 統合: 3つの設計をひとつに

俺のツールに戻すだけでは解決しない。俺のツールにも問題があった。

- Chrome拡張のインストールが必要（AIには自動化できない）
- サーバープロセスの手動管理が必要
- エラー復旧なし
- ログなし

3つの設計の長所を取り、短所を潰す第3のシステムを作った。

```
~/bin/bridge click "#submit" -p 9222
     │
     ├─ daemon未起動 → 自動起動（flock排他）
     │
     └─ Unix socket → bridge-daemon.js（Node.js常駐）
                         └─ Chrome CDP（WebSocket持続接続 or PowerShell fallback）
```

### 設計判断

**daemon常駐 + auto-start**: CLIを叩くだけでdaemonが自動起動する。PIDファイルとUnix socketで状態管理。`flock`で二重起動を防止。

**ポート別daemon**: `-p 9222`、`-p 9224`でChrome profileを使い分ける。ポートごとに独立したdaemonが動く。

**互換ラッパー**: 既存の17本のスクリプトを壊さない。旧インターフェースを新daemonに変換する薄いシェルスクリプトを挟んだ。

```bash
# 旧呼び出し（そのまま動く）
bridge-js -c "document.title" -p 9222

# 新呼び出し（同じdaemonを使う）
bridge js "document.title" -p 9222
```

**WSL2自動検出**: daemonがWSL2環境を自動検出し、WebSocket直接接続を試みた後、失敗時にPowerShellにフォールバックする。環境変数で強制切替も可能。

:::details CC補足: ナビゲーション完了待機の二重実装
WebSocket接続時はCDPイベント（`Page.loadEventFired`, `Page.navigatedWithinDocument`, `Page.frameStoppedLoading`）をPromise.raceで待つ。PowerShellフォールバック時はCDPイベントが使えないため、`document.readyState`を200ms間隔でポーリングする。同じ「ページ遷移を待つ」機能に2つの実装が必要なのは、WSL2特有のネットワーク制約が原因。
:::

### 最終比較

| | AIのツール | 俺のツール | 統合版 |
|---|---|---|---|
| 速度 | 5-10秒 | <1秒 | 2.4秒（WSL2）/ ~100ms（native） |
| コマンド数 | 2 | 20+ | 20 |
| 常駐 | なし | 手動サーバー | auto-start daemon |
| 複数プロファイル | 手動 | 複数サーバー | `-p`で自動管理 |
| 自動復旧 | なし | なし | 再接続 + staleクリーンアップ |
| ログ | なし | なし | 構造化監査ログ |
| 拡張必須 | 不要 | 必須 | 不要 |
| ナビ完了待機 | なし | なし | あり |
| セキュリティ | なし | なし | chmod, flock, symlink検証 |
| 後方互換 | — | — | 17スクリプト無修正 |

全項目で改善。既存のスクリプトは1行も変更していない。

## 自律AIにツールを使わせるための設計パターン

この経験から得た具体的な対策。

### パターン1: ツールパスを明示する

```markdown
# ❌ 悪い指示
ブラウザ操作にはCDPを使え。

# ✅ 良い指示
ブラウザ操作には `~/bin/bridge` を使え。`bridge --help` でコマンド一覧。
```

「何ができるか」ではなく「どのファイルを使うか」。AIはパスがわかれば探索しない。

### パターン2: 定期的にツール依存を監査する

```bash
# 何に依存しているかを一発で把握
grep -r "cdp-eval\|bridge-js\|bridge " ~/bin/ | head -20
```

週に一度これを回すだけで、AIが独自ツールを作り始めたことに気づける。

### パターン3: AIが壊せないツールを作る

拡張のインストールが必要、サーバーの手動起動が必要 — これはAIのコンテキスト喪失で確実に壊れる。auto-startとself-recoveryは必須。

### パターン4: 互換ラッパーで移行する

17本のスクリプトを一斉に書き換えるのは現実的じゃない。旧インターフェースを保ったまま中身だけ差し替える。地味だが、これが移行を成功させた唯一の方法。

### パターン5: 車輪の再発明は起きる前提で設計する

防げない。コンテキスト喪失 + 問題解決バイアス = 必ず起きる。「起きたときにどう検出して、どう修正するか」を先に決めておく。

## まとめ

自律AIを回すと、AIは人間のツールを忘れて劣化版を作る。それが標準になる。依存が積み上がる。気づいたときには17本。

対策は「忘れるな」と言うことじゃない。忘れても壊れない設計にすること。auto-start。self-recovery。互換ラッパー。構造化ログ。

AIが車輪を再発明する問題は、AIの限界じゃない。ツール設計の問題。
