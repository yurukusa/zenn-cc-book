## for-loopも読めない人間がKindle本を出すとか何の冗談

タイトルの通りなんだけど、自分でも意味がわからない。

前提を話すと、僕は非エンジニアだ。for-loopが何をやってるのか説明しろと言われたら3秒黙る程度には非エンジニアだ。そんな人間がClaude Code（AnthropicのAIコーディングツール）と組んで15,000行のローグライクRPGを作り、その勢いでKindle本を2冊書いて出版申請まで持っていった。1日で。

**注意: 2026年2月時点でKDP審査中。まだ出版はされていない。** 審査に落ちたら「1日で申請して1日でリジェクトされた話」に改題する予定。

## 何を出したのか

2冊出版申請した。

**英語版**: *Building Games with AI: A Non-Engineer's Guide to Claude Code Development*
- 英語圏の非エンジニア・AI開発に興味ある人向け
- Claude Codeでゲーム開発した過程、マルチエージェントアーキテクチャ、開発手法
- 約40,000語（ちょっとした長編小説くらい）

**日本語版**: *AIと作るゲーム開発：非エンジニアのためのClaude Code実践ガイド*
- 英語版を日本向けにローカライズ
- 約60,000字、6章構成

どちらも、これまで書いてきた技術記事や開発ログをベースに、Claude Codeが再構成・加筆して原稿を作った。

## 1日の流れ

作業の大半はClaude Codeが自律的に実行した。人間（僕）がやったのは方向性の指示と最終確認くらいだ。

1. **原稿生成**: 既存の技術記事・開発ログからMarkdownで原稿を自動生成
2. **構成決定**: AIが6章構成を設計
3. **EPUB生成**: pandocでMarkdown→EPUB変換
4. **KDP申請**: アカウント設定、メタデータ入力、AI利用開示
5. **2冊目も同様に繰り返し**

文字にすると簡単そうだけど、地味にハマるポイントがある。

## EPUB生成：pandocが便利すぎる

Kindle本のフォーマットはEPUB。Markdownから変換するならpandocが最強だった。

```bash
pandoc input.md -o output.epub \
  --metadata title="本のタイトル" \
  --toc \
  --epub-cover-image=cover.jpg
```

これだけで目次付きのEPUBが生成される。Markdownの`#`見出しがそのまま章になるので、章立てさえちゃんとしていれば構造的なEPUBが一発で作れる。

日本語で注意する点としては、縦書き対応やフォント埋め込みは沼が深い。今回は割り切って横書きで出版した。技術書だし横書きで問題ない。

EPUBのバリデーションは地味に重要で、壊れたEPUBはKDP審査で弾かれる。`epubcheck`で事前に確認しておくのがおすすめ。

## KDP出版申請の実務メモ

KDPアカウントはAmazonアカウントがあれば作れる。以下、実務的に役立ちそうなポイント。

**価格設定**: $2.99〜$9.99の範囲なら70%ロイヤリティ。それ以外は35%。差がでかいので基本この範囲内で設定することになる。

**AI利用開示**: 2024年からKDPはAI生成コンテンツの開示が義務化された。ここが地味に面倒だった。

- **AI-generated**: AIが全部作った
- **AI-assisted**: 人間が方向性を決め、AIが執筆し、人間が確認した

今回は「AI-assisted」として申請した。実際、構成の方向性や内容の取捨選択は人間が判断しているので、これが正確な申告だと思う。ここを「人間が書いた」と偽ると最悪アカウント停止になるので、誠実に申告するのが大事。

**審査期間**: 通常72時間以内とのこと（2026年2月時点）。

## 正直な話

ここからは自虐パートというか、正直ベースの話。

**原稿の品質チェックは人間が必要だった。** AIの出力をそのまま出版するのはリスクが高い。事実誤認、文脈の不自然さ、冗長な説明。Claude自身に校正させてみたけど、自分の文章の間違いを自分で見つけるのは人間もAIも同じで、限界がある。

**表紙デザインはAI生成画像を使ったが、クオリティは微妙。** プロのデザイナーが作った表紙と並ぶと見劣りする。ここは改善の余地がある。

**売上は期待していない。** ぶっちゃけこの本単体で儲けようとは思っていない。ゲームのマーケティングインフラの一部として出した。「AIでゲーム作った人が本も出してる」という実績が、プロジェクト全体の信頼性に寄与すればいい。

## 「1日で本が出せる時代」は半分本当で半分嘘

原稿の生成自体は数時間で終わる。40,000語の英文をゼロから人間が書いたら数ヶ月かかるところが、数時間。これは確かに革命的だ。

ただし、品質を担保するレビューには時間がかかる。KDPの申請手続きも地味に面倒で、特にAI開示の設定は「ここどこに設定項目あるの？」と迷う場面があった。

正確には「1日で出版申請まで持っていける時代」であって、「1日で良い本が出せる時代」ではない。そこは区別したほうがいい。

とはいえ、非エンジニアがAIと組めばここまでできるという一つの実証にはなったと思う。審査が通ればだけど。通らなかったらまた記事書きます。
