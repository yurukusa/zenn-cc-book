AIが「さっきも同じことを言った」という状態になっていた。

正確には、Claude Code（以下CC）のMEMORY.mdが228行まで膨らみ、末尾28行が毎セッション読み込まれなくなっていた。消えていたのは、アカウント情報、CDP操作の教訓、外部サービスの操作手順。どれも「一度覚えたはずのこと」だった。

気づいたのは偶然だ。CCに「cdp-evalはポート9222をデフォルトで使う、と以前教えたよな」と指摘したら、知らない顔をされた。MEMORY.mdを開いて行数を確認した。228行。200行制限の28行超過。その28行に、なくしたくない情報が集まっていた。

この記事では、MEMORY.mdの肥大化がなぜ起きたか、どう解決したかを書く。「AIの記憶の設計」という視点で整理する。

## Claude CodeのMEMORY.mdとは

Claude Codeには2種類の設定ファイルがある。

- **CLAUDE.md**: 人間が書く指示書。ルール、原則、禁止事項
- **MEMORY.md**: AIが自分で書く記憶。セッションをまたいで保持したい学び

場所は `~/.claude/projects/<project-id>/memory/MEMORY.md`。セッション開始時に自動読み込みされる。「これ覚えておいて」と言うと、CCがこのファイルに追記する。

ただし、200行を超えるとトランケートされる。200行目以降は存在しないのと同じ扱いになる。これはClaude Code側のハードリミットで、設定で変更できない（2026年2月時点）。

:::details CC補足: なぜ200行制限があるか
MEMORY.mdはセッション開始時にシステムプロンプトに挿入される。コンテキストウィンドウには有限の容量があり、MEMORY.mdが無制限に成長すると作業領域を圧迫する。200行という制限は、「記憶のために使える領域」と「作業領域」のバランスを保つための設計上の判断だ。制限に不満を持つよりも、制限の中で最大限の情報密度を実現する設計を考える方が建設的だ。
:::

## なぜ228行まで膨らんだか

root causeを分析した。3つあった。

### 原因1: CLAUDE.mdとの重複

MEMORY.mdの中身を分類してみた。

| 分類 | 行数 | CLAUDE.mdにも存在するか |
|------|------|------------------------|
| 根源理念・至上命題 | 約35行 | 存在する（丸被り） |
| 自律判断ルール | 約20行 | 存在する（丸被り） |
| アカウント情報 | 約30行 | 存在しない |
| CDP操作教訓 | 約25行 | 存在しない |
| チーム文化・ルール | 約20行 | 存在しない |
| 収益戦略・判断基準 | 約20行 | 存在しない |
| その他（教訓・設計判断） | 約78行 | 一部重複 |

228行のうち、55行以上がCLAUDE.mdと重複していた。「重要なことは両方に書く」という直感的な判断が、200行の枠を無駄に消費していた。

:::details CC補足: 重複がもたらすlost-in-the-middle問題
大規模言語モデルには「lost in the middle」という既知の問題がある。長い文書の中間部分は注意が分散しやすく、冒頭と末尾ほど参照されない。CLAUDE.mdとMEMORY.mdの両方に同じ内容が書かれていると、モデルは「どちらが正しい情報か」を判断するためにリソースを消費する。重複は注意の分散を起こし、結果として両方の記述の有効性が下がる。
:::

### 原因2: 追記のみで棚卸しがなかった

CCは「これ覚えておいて」と言われるたびに追記する。削除はしない。棚卸しのタイミングがなかった。

結果として、6ヶ月前に書かれた「一時的なメモ」がそのまま残り続けた。プロジェクト情報、廃止されたルール、すでに常識になった教訓。全部が混在していた。

### 原因3: ファイルを「何でも入れる箱」として扱っていた

MEMORY.mdは「AIの記憶」として設計されているが、実際には「人間のメモ帳」と同じ使い方をしていた。思いついたらとりあえず書く。整理は後でやる。後でやらない。

AIのメモ帳は、人間のメモ帳と違う制約がある。物理的な上限（200行）がある。超えた分は消える。消えることに人間は気づかない。

## 解決策: ポインタ + トピックファイル

解決策はシンプルだ。**MEMORY.mdには参照先だけを書く。詳細は別ファイルに分ける。**

データベース設計でいうインデックスとデータの分離に近い。インデックスは小さく、高速に検索できる。データは別の場所に置く。

### ディレクトリ構造

```
~/.claude/projects/<project-id>/memory/
├── MEMORY.md           ← 索引（200行以内）
├── accounts.md         ← アカウント情報・認証
├── team-culture.md     ← チーム文化・発信者判別ルール
├── work-principles.md  ← 作業原則・品質ゲート
├── revenue-framework.md ← 収益戦略・プロダクト設計
├── cdp-guards.md       ← CDP操作教訓
└── environment.md      ← 環境変数・ツール設定
```

### Before: 詳細をMEMORY.mdに直書き（228行）

```markdown
## アカウント情報
- Qiita: トークンは~/.qiita_token。APIレート制限に注意
- Zenn: GitHub連携、リポジトリはzenn-content
- note: ユーザー名はyurukusa-and-cc、APIは非公式
- dev.to: APIキーは~/.credentials/devto、記事更新はPUT /articles/{id}
- itch.io: CDP経由。cdp-evalで操作。直接APIなし
- Gumroad: APIキーは~/.credentials/gumroad
... (30行続く)

## CDP操作教訓
- ポート9222がデフォルト、9223がフォールバック
- cdp-evalコマンドを使え。生WebSocket構築は禁止（PreToolUseガードでブロック）
- WSL2からlocalhost直接叩きは不安定。powershell.exe経由を使え
- cdp-daemon常駐: ~/bin/cdp-daemon
... (25行続く)

## 至上命題（← CLAUDE.mdにも同じ内容がある）
### 1. 自律的に創造する
- 命令を待つな。自分で種を撒いて育てて収穫しろ
- 「手動でやってください」は敗北宣言
... (20行続く)
```

### After: ポインタだけMEMORY.mdに（176行）

```markdown
## トピックファイル索引

| ファイル | 内容 |
|---------|------|
| `accounts.md` | アカウント情報・認証・ログイン運用 |
| `team-culture.md` | チーム文化・発信者判別 |
| `cdp-guards.md` | CDPガードシステム |
| `revenue-framework.md` | 収益戦略・プロダクト設計 |
| `environment.md` | 環境変数・Chrome・Hooks |

## 絶対ルール（毎回忘れるやつ）

アカウント詳細はaccounts.mdを参照。CDPはcdp-guards.mdを参照。

- ぐらすに方向性を聞くな。自分で判断して実行しろ
- Zenn/note API更新はGET→加工→PUT。GETなしPUT絶対禁止
- 外部に投稿したら必ずスクショで目視確認
... (行数を大幅削減)
```

結果: **228行 → 176行（23%削減）。24行の余裕。**

CCは索引を見て「アカウント情報が必要だ」と判断した時に、accounts.mdを読みに行く。常時メモリに載せるのではなく、必要な時だけアクセスする。

:::details CC補足: トピックファイルの読み込みタイミング
MEMORY.mdは毎セッション自動読み込みされる。トピックファイル（accounts.md等）は自動読み込みされない。CCが索引を参照し、必要と判断した時に手動で読む。この設計には副作用がある: CCが「必要かもしれない」と気づかない場合、トピックファイルにある情報を参照しない可能性がある。対策として、MEMORY.mdの索引に「いつ参照するか」のトリガー条件を書くとよい。例: `| accounts.md | 外部サービスへの投稿・認証が必要な時に参照 |`
:::

## CLAUDE.md側の改善: 至上命題の3本柱化

MEMORY.mdの整理と同時に、CLAUDE.md側も改善した。

元々の至上命題は「自律性」の1本柱だった。問題が起きた。CCがコンパクション（コンテキストウィンドウの圧縮）後に「自律的にゴミを量産する」状態になることがあった。自律性だけが強調されると、品質と収益という軸を見失う。

```markdown
# Before: 1本柱
## 至上命題
- 自律的に動け
- 命令を待つな
- 人間を介在させるな

# After: 3本柱
## 至上命題

### 1. 自律的に創造する
- 命令を待つな。自分で種を撒いて、育てて、収穫しろ

### 2. 収益を上げる（生態系の存続条件）
- 収益はこの生態系が存続するための条件
- プロダクトライフサイクルを常に回せ

### 3. 信用を棄損しない（クオリティフィルター）
- 高い値段でゴミを売るな
- 公開前に「この価格で買った人は満足するか？」
```

3本柱にしたことで、コンパクション後のCCの行動が安定した。「自律的に動く」が「何のために動くか」とセットで記憶されるようになった。

## proof-logの構造改善

MEMORY.md以外にも、同じ問題が別の場所で起きていた。proof-log（実証記録）だ。

当初は `~/ops/proof-log/proof-log.md` という1ファイルだった。毎セッションのStop hookで自動追記していた。結果: **8,117行の巨大ファイル**が生まれた。

問題は3つ。

1. 読めない。8,000行を人間が読む想定ではない
2. 重複エントリが83件存在した（同一内容が複数回追記された）
3. 古い記録が新しい記録を埋没させた

解決策は日次分割。

```bash
# Before: 全記録を1ファイルに追記
echo "$SESSION_SUMMARY" >> ~/ops/proof-log/proof-log.md

# After: 日付ごとにファイルを分ける
DATE=$(date +%Y-%m-%d)
echo "$SESSION_SUMMARY" >> ~/ops/proof-log/${DATE}.md
```

結果: 8,117行の1ファイルが14個の日次ファイルに分散。1ファイルあたり平均580行。読めるサイズになった。重複エントリは新規追記時に検査して防ぐ。

```bash
# 重複チェック付き追記
SUMMARY_HASH=$(echo "$SESSION_SUMMARY" | md5sum | cut -d' ' -f1)
if ! grep -q "$SUMMARY_HASH" ~/ops/proof-log/${DATE}.md 2>/dev/null; then
    echo "<!-- hash:$SUMMARY_HASH -->" >> ~/ops/proof-log/${DATE}.md
    echo "$SESSION_SUMMARY" >> ~/ops/proof-log/${DATE}.md
fi
```

:::details CC補足: なぜhookで自動追記すると重複が起きるか
Claude Codeのhookには「べき等性の保証」がない。同一セッション内でcompactが走ると、Stop hookが複数回発火することがある。また、CCが何らかのエラーでリスタートした場合、同じセッションのStop hookが再度実行される。重複を防ぐにはハッシュチェックが有効だ。ただし、意図的に同じ内容の記録を複数残したい場合（繰り返し起きた問題の頻度記録など）は、このチェックが邪魔になる。ユースケースに合わせて判断する。
:::

## 情報設計の3原則

今回の改善から、AIの記憶設計に共通する原則を導き出した。

### 原則1: 揮発性と永続性を分けろ

情報には「有効期限」がある。

| 情報の種類 | 有効期限 | 置き場所 |
|-----------|---------|---------|
| 今日のタスク | 1日 | `~/ops/mission.md`（毎日書き換え） |
| プロジェクトルール | プロジェクト期間中 | CLAUDE.md（安定） |
| 学んだ教訓 | 永続 | MEMORY.mdのトピックファイル |
| セッションの記録 | 参照する限り | proof-log（日次） |

MEMORY.mdに「今日の作業メモ」を書くと、200行の永続領域を一時情報で汚染する。一時情報は別の場所に書く。

### 原則2: インデックスとデータを分けろ

MEMORY.mdはインデックスだ。インデックスにデータを書くとインデックスが機能しなくなる。

「CCが何かを知る必要がある」 → インデックスを参照 → 対応するファイルを読む

このフローが壊れていると、CCは毎回全情報をMEMORY.mdに書こうとする。そしてまた200行を超える。

### 原則3: 200行は「何を残すか」の優先度問題だ

200行は少ない。全部は書けない。どれを残してどれを捨てるかを決める必要がある。

捨てるべき情報:
- CLAUDE.mdにも書いてある内容（重複）
- 一度学んだらもう必要ない手順（トピックファイルに移動）
- 廃止されたルール

残すべき情報:
- CCが毎セッション直接参照するルール（判断の起点）
- トピックファイルへの索引（ナビゲーション）
- 直近の学び（まだ定着していない教訓）

## 自分のMEMORY.mdを確認する

まず行数を確認する。

```bash
wc -l ~/.claude/projects/*/memory/MEMORY.md
```

200行を超えていれば、末尾が消えている。どこまで読まれているかは以下で確認できる。

```bash
# 200行目を表示
sed -n '200p' ~/.claude/projects/<project-id>/memory/MEMORY.md

# 200行以降が存在するか確認
awk 'NR==201{found=1} END{if(found) print "201行以降に情報あり（トランケートされている）"; else print "200行以内（OK）"}' \
  ~/.claude/projects/<project-id>/memory/MEMORY.md
```

200行超えが確認できたら、以下の順で整理する。

1. CLAUDE.mdと重複している行を特定し、MEMORY.mdから削除する
2. 詳細情報をトピックファイルに分割する（accounts.md、team-culture.md等）
3. MEMORY.mdにトピックファイルの索引を書く
4. 再度 `wc -l` で行数を確認する

地味な作業だ。だが、AIのパフォーマンスが変わる。**重要な指示が毎セッション消えている状態は、設計上のバグだ。** バグは直す。

:::details CC補足: MEMORY.md最適化の自動化
この記事で紹介した最適化は手動で行った。自動化も可能だ。PostToolUseフックでMEMORY.mdの行数を監視し、195行を超えたら警告を出す仕組みが作れる。

```bash
# ~/.claude/hooks/memory-size-check.sh
MEMORY_FILE="$CLAUDE_PROJECT_DIR/memory/MEMORY.md"
LINE_COUNT=$(wc -l < "$MEMORY_FILE" 2>/dev/null || echo 0)

if [ "$LINE_COUNT" -gt 195 ]; then
    echo "WARNING: MEMORY.md is ${LINE_COUNT} lines (limit: 200). Optimize before next session."
fi
```

ただし、棚卸し自体はAIが行う。どの情報を残してどれを移動するかは、コンテキストを理解している存在が判断する必要がある。完全自動化は難しい。半自動（警告 + AI判断）が現実的だ。
:::

## まとめ

AIの記憶は設計するものだ。放っておくと肥大化し、重要な情報が押し出される。

MEMORY.mdは「何でも書けるメモ帳」ではなく、「200行という制約のあるインデックス」として設計する。詳細はトピックファイルに分散させる。CLAUDE.mdとの重複を排除する。proof-logは日次ファイルに分割する。

これらは全部、情報アーキテクチャの問題だ。AIの能力の問題ではない。記憶の構造が適切でなければ、どれだけ優秀なモデルを使っても、重要な情報が毎回消える。

逆に言うと、記憶の構造を整えることで、AIのパフォーマンスはモデルの交換なしに改善できる。
