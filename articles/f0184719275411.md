## TL;DR

`instructions.md` にタスクを日本語で書き、`claude-watch` デーモンがそれを検知してClaude Codeを自動起動する。タスクの進捗は `status.md` に記録される。この仕組みにより、夜間バッチ実行や離席中の自律作業が可能になる。

この記事では、このワークフローの設計パターンと実践的なノウハウを解説する。

## 背景：なぜファイル駆動なのか

Claude Codeは対話的なツールとして設計されている。人間がプロンプトを入力し、AIが応答し、人間が次の指示を出す。このループは強力だが、制約がある。

**人間がボトルネックになる。**

- コードレビューを待つ間、Claude Codeは何もしない
- 10個のタスクがあっても、1個ずつ指示を出す必要がある
- 夜間や離席中に作業を進められない

解決策として構築したのが `instructions.md` 駆動のワークフローだ。

## システム全体像

```
┌──────────────────────────────────────────────┐
│  人間の作業                                   │
│  1. instructions.md にタスクを記述            │
│  2. 保存して離席                              │
└──────────┬───────────────────────────────────┘
           │ ファイル変更（mtime更新）
           ▼
┌──────────────────────────────────────────────┐
│  claude-watch（常駐デーモン）                  │
│  - 5秒ごとにmtimeをポーリング                 │
│  - instruction-id で重複実行防止              │
│  - claude -p --dangerously-skip-permissions   │
│    でヘッドレス実行                            │
└──────────┬───────────────────────────────────┘
           │ Claude Code セッション
           ▼
┌──────────────────────────────────────────────┐
│  Claude Code（自律実行）                      │
│  - instructions.md のタスクを解釈             │
│  - 各タスクを実行（コード生成、テスト等）      │
│  - status.md に進捗を記録                     │
│  - 完了後、セッション終了                     │
└──────────────────────────────────────────────┘
```

## claude-watch の実装

### コアとなるポーリングループ

```bash
#!/bin/bash
# claude-watch.sh
# instructions.md の変更を検知してClaude Codeを自律実行するデーモン
# ファイルシステムのポーリングにしている理由：
#   inotifyはWSL2で不安定なため、確実性を優先してmtime比較を採用

WATCH_DIR="${1:-.}"
INSTRUCTIONS="$WATCH_DIR/instructions.md"
STATUS="$WATCH_DIR/status.md"
LOG="$WATCH_DIR/.claude-watch.log"
POLL_INTERVAL=5
LAST_MTIME=0
LAST_ID=""

log() {
    echo "[$(date '+%Y-%m-%d %H:%M:%S')] $1" | tee -a "$LOG"
}

# instructions.md からinstruction-idを抽出する
# 同じIDのタスクを二重実行しないためのキー
get_instruction_id() {
    grep -oP '(?<=instruction-id:\s).*' "$INSTRUCTIONS" 2>/dev/null || echo ""
}

log "claude-watch started. Watching: $INSTRUCTIONS"

while true; do
    if [ -f "$INSTRUCTIONS" ]; then
        current_mtime=$(stat -c %Y "$INSTRUCTIONS" 2>/dev/null || echo "0")
        current_id=$(get_instruction_id)

        # mtimeが変わった AND instruction-idが前回と異なる場合に実行
        if [ "$current_mtime" != "$LAST_MTIME" ] && \
           [ "$current_id" != "$LAST_ID" ] && \
           [ -n "$current_id" ]; then

            log "Change detected. ID: $current_id"
            LAST_MTIME="$current_mtime"
            LAST_ID="$current_id"

            # status.md に開始を記録
            echo "## 実行中: $current_id" > "$STATUS"
            echo "開始時刻: $(date '+%Y-%m-%d %H:%M:%S')" >> "$STATUS"
            echo "状態: running" >> "$STATUS"

            # Claude Codeをヘッドレスモードで実行
            # --dangerously-skip-permissions は自律実行に必須
            # -p でプロンプトを直接渡す
            claude -p \
                --dangerously-skip-permissions \
                "$(cat "$INSTRUCTIONS")" \
                2>&1 | tee -a "$LOG"

            # 完了を記録
            echo "状態: completed" >> "$STATUS"
            echo "完了時刻: $(date '+%Y-%m-%d %H:%M:%S')" >> "$STATUS"
            log "Task completed: $current_id"
        fi
    fi

    sleep "$POLL_INTERVAL"
done
```

### systemdサービスとして登録

常駐させるにはsystemdのユーザーサービスとして登録する。`ExecStart` にclaude-watch.shのパスを指定し、`Restart=on-failure` で自動復帰させる。`systemctl --user enable claude-watch` で自動起動を設定すれば、ログイン時に自動的に監視が始まる。

## instructions.md の書き方

### 基本構造

```markdown
---
instruction-id: 2026-02-08-feature-auth
project: /home/user/myapp
---

## タスク

1. ユーザー認証機能を追加する
   - JWT ベースのトークン認証
   - ログイン/ログアウトのエンドポイント
   - パスワードハッシュは bcrypt を使用

2. テストを書く
   - 正常系：ログイン成功、トークン検証
   - 異常系：不正なパスワード、期限切れトークン

3. 完了後、status.md にサマリーを記載する

## 制約

- 既存のAPIエンドポイントを壊さない
- requirements.txt に依存を追記する
- テストが全て通ることを確認してからstatus.mdに完了と書く
```

### 設計のポイント

#### 1. instruction-id は必須

`instruction-id` がないとclaude-watchは実行しない。これは安全装置であり、エディタの自動保存でインストラクションが意図せず再実行されることを防ぐ。IDは `日付-概要` の形式が分かりやすい。

#### 2. タスクは番号付きで順序を明示する

Claude Codeは番号付きリストを順序として解釈する。依存関係があるタスクは番号順に並べる。

```markdown
## タスク

1. データベーススキーマを作成する
2. マイグレーションを実行する
3. CRUD APIを実装する（1, 2に依存）
4. テストを書いて実行する（3に依存）
```

#### 3. 制約は明示的に書く

「壊さないでほしいもの」「使うべき技術」「やってはいけないこと」を明記する。Claude Codeは制約がないと最適解を自分で判断するが、既存プロジェクトでは意図しない選択をすることがある。

```markdown
## 制約

- Python 3.11 を使用（3.12の機能は使わない）
- フレームワークはFastAPI（Flaskに変更しない）
- データベースはSQLite（PostgreSQLにしない）
- 既存のテストを壊さない
```

#### 4. 完了条件を書く

何をもって「完了」とするかを明記する。これがないとClaude Codeが中途半端な状態で止まることがある。

```markdown
## 完了条件

- [ ] 全テストがpass
- [ ] lintエラーがゼロ
- [ ] status.md にサマリーを記載
- [ ] 新規ファイルがgit addされている
```

## status.md による進捗管理

Claude Codeの実行中と完了後の状態を `status.md` で追跡する。

### status.md の構造

```markdown
## 実行中: 2026-02-08-feature-auth
開始時刻: 2026-02-08 23:00:00
状態: completed
完了時刻: 2026-02-08 23:45:00

### サマリー
- JWT認証を `src/auth/` に実装
- エンドポイント: POST /login, POST /logout, GET /me
- テスト12件追加、全てpass

### 注意事項
- SECRET_KEY は環境変数 JWT_SECRET から読む設定にした
```

朝起きてstatus.mdを確認するだけで、夜間に何が行われたかが一目でわかる。

## タスクバッチ：複数タスクの連続実行

大きな作業を複数のinstructions.mdに分割し、順番に実行する仕組み。

```bash
#!/bin/bash
# batch_execute.sh
# 複数のinstructions.mdを順番にclaude-watchに投入する
# 各タスクの完了をstatus.mdで確認してから次に進む

TASKS_DIR="$1"
WORK_DIR="$2"

for task_file in "$TASKS_DIR"/task_*.md; do
    task_name=$(basename "$task_file" .md)
    echo "=== Starting: $task_name ==="

    # instructions.md にコピー
    cp "$task_file" "$WORK_DIR/instructions.md"

    # claude-watchが検知して実行するのを待つ
    # status.md の「状態: completed」を監視
    while true; do
        if grep -q "状態: completed" "$WORK_DIR/status.md" 2>/dev/null; then
            echo "=== Completed: $task_name ==="
            # 結果をアーカイブ
            cp "$WORK_DIR/status.md" "$TASKS_DIR/results/${task_name}_result.md"
            break
        fi
        sleep 10
    done
done

echo "All tasks completed."
```

タスクファイルの命名規則は `task_01_setup.md`、`task_02_implement.md` のように番号で順序を示す。

## 失敗パターンと対策

### パターン1：タスクが曖昧すぎる

```markdown
# 悪い例
## タスク
- UIを良くする
- パフォーマンスを改善する

# 良い例
## タスク
- ダッシュボードのテーブルにソート機能を追加する
  - カラムヘッダクリックで昇順/降順切り替え
  - デフォルトは作成日降順
```

曖昧な指示はClaude Codeが「何が正解か」を判断できず、的外れな変更をする原因になる。

### パターン2：スコープが大きすぎる

1つのinstructions.mdに20個のタスクを書くと、後半のタスクでコンテキストが溢れて品質が下がる。目安は **1セッションで5タスク以内**。大きな作業はバッチ実行で分割する。

### パターン3：既存コードの把握不足

Claude Codeは実行のたびに新しいセッションで起動する。前回のセッションで何をしたか覚えていない。対策として、instructions.md に関連ファイルのパスを明記する。

```markdown
## コンテキスト

関連ファイル:
- src/api/routes.py — 既存APIルート（ここに追加する）
- src/models/user.py — Userモデル（これを使う）
- tests/conftest.py — テストフィクスチャ（ここにDB設定がある）
```

### パターン4：テスト未実行で完了扱い

「テストを実行して確認する」と書いてもスキップされることがある。「テストを実行せずにcompletedと書くことを禁止する」のように強い言葉で制約を書くと改善する。

### パターン5：ファイルの破壊的変更

Claude Codeが既存ファイルを壊すことがある。「作業ブランチを作成し、各タスク完了後にコミットする」を事前準備として指示することで対策する。

## 実践例：夜間バッチ実行

実際にこのシステムを使って夜間に10記事のマーケティング素材を生成した例。

```markdown
---
instruction-id: 2026-01-20-marketing-batch
project: /home/user/game-project
---

## タスク

1. outputs/reddit_gamedev.md にReddit r/gamedev用の投稿を生成する
   - ゲームの技術的な側面にフォーカス
   - 800-1200ワード

2. outputs/reddit_indiegames.md にReddit r/IndieGames用の投稿を生成する
   - ゲームの魅力にフォーカス
   - 400-600ワード

3. outputs/steam_description.md にSteamストアの説明文を生成する
   - 日英両方

4. 各ファイル生成後、status.md に完了を記録する

## 制約

- 個人情報を含めない（名前、住所、勤務先等）
- ハンドルネーム「ゆるくさ」のみ使用可
```

23時に instructions.md を保存して寝る。朝7時に status.md を確認すると全タスクが完了している。生成物をレビューして微調整するだけ。

## まとめ

`instructions.md` 駆動開発のポイントは3つ。

1. **タスクは具体的に、スコープは小さく** — 曖昧さを排除し、1セッション5タスク以内
2. **制約と完了条件を明記** — Claude Codeが「何をすべきでないか」「いつ完了か」を判断できるように
3. **status.md で追跡** — 実行結果を確実に記録し、朝起きて確認できる状態にする

このパターンにより、人間がボトルネックにならない開発フローが実現する。
