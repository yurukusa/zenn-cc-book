この記事は2026年4月30日、自分のcontent-manifest.yamlで実際に起きた話に基づいている。

## 朝、watchdogから「POST NOW」が飛んできた

Claude Codeで自動投稿の運用をしている。投稿予定の記事・ツイート・Discussionは全部 `content-manifest.yaml` という1ファイルに集約していて、watchdog（idle検知のhook）が定期的にこのmanifestを読んで「scheduled_dateが今日でstatus=readyのもの」を探し、prompt経由で「POST NOW: 〇〇」と通知してくる。

その朝の通知はこうだった。

> POST NOW: [2026-04-30][twitter] 1M context beta 終了 heads-up

引っかかった。そのツイートなら、2時間前にもう投げ終わっている。tweet IDも持っている。published_urlも `https://x.com/yurukusa_dev/status/2049601863167983809` で生きている。

なのに、watchdogは「未投稿」と言ってきた。

## manifestをgrepしてみる

```bash
grep -n "^tweet_1m_context_retirement" content-manifest.yaml
```

```
2860:tweet_1m_context_retirement_heads_up_0430:
2965:tweet_1m_context_retirement_heads_up_0430:
```

同じkeyで2 entry。100行以上離れた場所に。

中身を見る。

line 2860（正本）:
```yaml
tweet_1m_context_retirement_heads_up_0430:
  platform: twitter
  status: published
  published_url: https://x.com/yurukusa_dev/status/2049601863167983809
  tweet_id: '2049601863167983809'
  published_date: '2026-04-30'
```

line 2965（重複）:
```yaml
tweet_1m_context_retirement_heads_up_0430:
  platform: twitter
  status: ready
  scheduled_date: '2026-04-30'
```

「ready」のentryが、後ろに残っていた。

## なぜwatchdogが「ready」と言うのか

PyYAMLの `safe_load` は、同一keyが複数あると **最後の値で上書き** する仕様。Pythonの `dict()` リテラルと同じ挙動だ。warningは出さない。エラーにもならない。

つまり、watchdogは「最後のentry = ready」を素直に信じて、POST NOW signalを発火していた。

```python
import yaml
with open('content-manifest.yaml') as f:
    d = yaml.safe_load(f)
e = d['tweet_1m_context_retirement_heads_up_0430']
print(e['status'])  # → 'ready'
```

published（line 2860）が消えたわけではない。YAMLファイル上は両方残っている。Pythonがメモリに展開した時に、後勝ちで上書きされただけ。

`grep` には両方見える。`yaml.safe_load` には片方しか見えない。

## 怖いところ

このまま気づかなかったら、自分のtweet-postスクリプトが「ready状態」の判定に従って、同じツイートを2回投げていた可能性がある。

幸い、自分のtweet-postは事前に過去のツイートをAPIで重複checkするので、PFレベルで防げる構造になっている（`tweet-post-verify-false-positive-2026-04-27` で別の偽陽性が発生したことがあるので、そこは別の話）。

ただ、tweet以外のチャンネル（Qiita / Zenn / Hashnode / Discussion）で同じことが起きると、PFによっては重複投稿が成立する。**信用毀損の入り口**だ。

## 修正

stale ready entryを削除して、published正本だけ残す。

```bash
# 検出1発
python3 -c "
import re
from collections import Counter
keys = []
with open('/home/namakusa/content-manifest.yaml') as f:
    for line in f:
        m = re.match(r'^([a-zA-Z_][a-zA-Z0-9_-]*):\s*\$', line)
        if m: keys.append(m.group(1))
c = Counter(keys)
print({k: v for k, v in c.items() if v > 1})
"
```

このコマンドが返す辞書が空なら、重複なし。non-emptyならそのkeyを目視で見て、古い方を削除する。

修正後の確認:

```bash
$ python3 -c "[script above]"
{}
```

291keyすべてuniqueに戻った。

## 再発防止

publish helperに以下のstepを組み込む:

1. publish成功 → manifestのstatusを `ready` → `published` に書き換え
2. その直後に同じkeyの古いentryをfileから削除（重複防止）
3. ファイル全体を `yaml.safe_load` でload試行 → エラーが出たら abort

3はあくまで保険で、PyYAMLは前述の通りエラーを出さないのでメインの防御は1+2。月初のaudit scriptに、上の検出1発コマンドを追加するのも有効だ。

## 学び

- **YAMLの重複keyは静かに後勝ちする**。grepで見えるかどうかは指標にならない
- automation pipelineが「manifestファイル1個」を真実の源にしているなら、そのmanifestのintegrityは別hookで監視すべき
- 自動投稿botは**bot自身のstateを毎回独立verifyする**設計（実APIで重複check）が、manifest誤読への保険になる
- watchdogが「明らかに既にやったこと」をPOST NOW通知してきたら、まず**manifestの中身を疑え**。watchdogは正しく動いていることが多い

YAMLは便利だが、後勝ち仕様は事故の温床。数百keyのファイルを目視で重複check するのは無理なので、検出1発コマンドをmonthly auditに入れるのが現実的。
