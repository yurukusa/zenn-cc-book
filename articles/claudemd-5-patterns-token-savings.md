CLAUDE.mdが長いと、Claude Codeのトークン消費が増える。CLAUDE.mdの内容は**毎ターン**コンテキストに含まれるからだ。

800時間の自律運用で、CLAUDE.mdを500行→100行→35行に圧縮した。キャッシュ読み取り率が改善し、トークン消費が体感で40-50%減った。

ポイントは「短くする」ことではなく「効率的に書く」こと。同じ意図を、より少ないトークンで伝える5つのパターンを紹介する。

## パターン1: 禁止リストより許可リスト

**非効率:**

```markdown
## 禁止事項
- rm -rfを使うな
- git reset --hardを使うな
- git push --forceを使うな
- sudoを使うな
- chmod 777を使うな
```

禁止リストは常に不完全。リストにない操作は「許可」と解釈される。

**効率的:**

```markdown
## 許可されている操作
- ファイル読み取り: 常に可
- git commit: 可（mainへの直接pushは不可）
- ファイル削除: 自分が作成したファイルのみ

上記にない破壊的操作は実行前に確認。
```

行数は同程度だが、リトライが減る。禁止リスト方式では「これは禁止？」の判断が毎回発生し、hookにブロック→別の方法→またブロックのループになる。許可リスト方式では「リストにないからやめる」で1回で済む。

## パターン2: 具体例を1つ（few-shot）

**非効率:**

```markdown
コミットメッセージは分かりやすく書くこと。
```

→ 「Update files」「Fix bug」が生成される。基準がないからだ。

**効率的:**

```markdown
コミット: 「なぜ」を含める。
✓ fix: prevent duplicate API calls on rapid button clicks
✗ fix bug
```

+2行で品質が劇的に変わる。LLMは具体例からパターンを抽出するのが得意だ。

## パターン3: 理由を1行だけ

**非効率:**

```markdown
テストは `__tests__/` に配置する。
```

理由がないルールはエッジケースで無視される。

**効率的:**

```markdown
テストは `__tests__/` に配置する。
  理由: src/と同階層だとビルド出力に混入する
```

1行の理由でClaude Codeは「なぜダメか」を理解し、類似ケースにも適用する。

## パターン4: テーブル形式

**非効率（散文5行）:**

```markdown
技術選択で迷ったら最もポピュラーなものを選ぶ。
要件が不明確なら最もシンプルに解釈する。
セキュリティは常に安全な方を選ぶ。
```

**効率的:**

```markdown
| 状況 | 判断 |
|------|------|
| 技術選択 | 最もポピュラーなもの |
| 不明確な要件 | 最もシンプルな解釈 |
| セキュリティ | 常に安全な方 |
```

同じ情報量、構造が明確。LLMはテーブル形式を高精度で処理する。

## パターン5: hookに任せる

**非効率:**

```markdown
## 安全ルール
- rm -rfは使わない
- git reset --hardは使わない
- .envファイルはgit addしない
- mainに直接pushしない
```

CLAUDE.mdに書いても「お願い」に過ぎない。

**効率的:**

```markdown
## 安全ルール
hookが破壊的コマンド・branch保護・secretを自動ブロック。
ブロックされた操作はリトライしないこと。
```

6行→2行。hookが技術的に強制するルールをCLAUDE.mdにも書くのは二重管理。hookの導入は1コマンド:

```bash
npx cc-safe-setup
```

## 早見表

| パターン | 核心 | 節約メカニズム |
|---------|------|---------------|
| 許可リスト | 「やっていいこと」を書く | リトライ削減 |
| 具体例 | 良い例＋悪い例 | 品質向上→やり直し減 |
| 理由1行 | 「なぜ」を15語で | エッジケース判断 |
| テーブル | 条件→判断の表 | パース効率 |
| hook分離 | 強制ルールはhookへ | 二重管理排除 |

## 35行テンプレート

5パターンを全て適用した最小構成:

```markdown
# プロジェクト指示書

## 基本方針
- ファイル読み込みはlimitパラメータで必要な部分だけ
- 1セッション = 1テーマ。テーマが変わったら新セッション
- hookがブロックした操作はリトライしない

## コーディング規約
| 状況 | 判断 |
|------|------|
| 技術選択 | 最もポピュラーなもの |
| 不明確な要件 | 最もシンプルな解釈 |
| セキュリティ | 常に安全な方 |

## コミット
✓ fix: prevent duplicate API calls on rapid button clicks
✗ fix bug

## 禁止
hookがブロックする操作はリトライしない。
```
