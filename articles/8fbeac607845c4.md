## AIエージェントのタスク設計で学んだ3つのパターン

6人のAIエージェントチームに17タスクを渡して並列で動かした。最初に起きたのは、依存関係を無視した衝突だった。

designerがスクリーンショットを撮り終える前に、shipperがitch.ioページを更新しようとした。空のスクショ枠にプレースホルダーが表示された状態でページが公開されかけた。

この失敗から学んだタスク設計のパターンを3つ書く。

## パターン1: Sequential（直列依存）

一番シンプルな依存関係。AのアウトプットがBのインプットになる場合。

```
Task #2: スクリーンショット撮影 [designer]
Task #6: itch.ioページ更新 [shipper] → blockedBy: [#2]
```

Task #2が `completed` にならない限り、Task #6は着手不可になる。shipperがTaskListを見ても、#6は `blocked` と表示される。

designerが「完了」とマークした瞬間にshipperの#6が解放される。shipperが次のTaskList確認時に拾う。

**使いどころ**: 成果物の受け渡しがある場合。「〇〇を作ってから△△に使う」。

**注意**: designerが完了を忘れるとshipperは永遠に待つ。プロンプトに「完了したらTaskUpdateで必ずcompletedに更新しろ」と明記する。

## パターン2: Fan-out（1対多分岐）

1つのタスクの完了が複数タスクのブロックを解除する。

```
Task #5: 全PF指標収集 [grower]
Task #12: note記事ドラフト [grower] → blockedBy: [#5]
Task #13: Zenn記事ドラフト [grower] → blockedBy: [#5]
Task #15: dev.to記事ドラフト [grower] → blockedBy: [#5]
```

指標を集め終わってから記事を書く。指標がないと記事の数字が埋められないから。

Task #5が完了した瞬間、#12, #13, #15の3つが一斉に着手可能になる。ただし、1人のgrowerが3つを同時にはできない。順番に拾う。

**使いどころ**: 調査・リサーチ→複数アウトプットの場合。1つの情報源が複数の成果物を生む。

**注意**: 同じエージェントが全ての下流タスクをownerにすると、結局シーケンシャルになる。本当に並列化したいなら、下流タスクを別のメンバーに振る。

## パターン3: Fan-in（多対1合流）

複数タスクが全て完了してから次に進む。

```
Task #1: ゲームフィール改善 [builder]
Task #2: スクリーンショット撮影 [designer]
Task #5: 指標収集 [grower]
Task #10: CrazyGames提出 [shipper] → blockedBy: [#1, #2, #5]
```

CrazyGamesに提出するには、ゲームの改善が完了し、スクショが撮影され、指標から提出の妥当性が確認されている必要がある。

3つ全てが `completed` にならないと#10は着手できない。1つでも残っていれば `blocked`。

**使いどころ**: リリース・提出など「全部揃ってから」の判断が必要な場合。

**注意**: 合流タスクのownerは長時間ブロックされる可能性が高い。プロンプトに「ブロック中は準備作業をしろ」と書く。実際の運用では、shipperはスクショ待ちの間にitch.ioページの現状分析を行った。

## アンチパターン

### 循環依存

```
Task A → blockedBy: [B]
Task B → blockedBy: [A]
```

デッドロック。AもBも永遠に着手できない。Task作成時に依存関係のグラフを確認する。3つ以上のタスクでの間接的な循環（A→B→C→A）も危険。

### 過剰ブロック

```
Task #7: カバー画像作成 [designer] → blockedBy: [#1, #2, #3, #4, #5, #6]
```

全タスクに依存させると、最後まで着手できない。本当に全部の完了が必要か？多くの場合、1-2のタスクだけが真の依存関係で、残りは「あったら嬉しい」レベル。

### 依存関係なし（全フラット）

```
Task #1: ゲーム改善
Task #2: スクショ撮影
Task #3: ページ更新
（全て独立、blockedByなし）
```

一見効率的だが、designerがまだスクショを撮り終えていないのにshipperがページ更新を始める。成果物が揃わない状態で出荷される。

## 実践Tips

### 1. 依存関係は最小限にする

17タスクで設定した依存関係は3ペアだけ。残り14タスクは独立して並列実行された。依存関係を増やすほど並列度が下がる。

### 2. ブロック中の仕事を設計する

blockedBy付きのタスクのownerには、待ち時間の過ごし方をプロンプトで指定する:

```
待機中の過ごし方:
- 更新対象ページの現状分析
- 変更計画のドキュメント作成
- 類似事例のリサーチ
```

### 3. 完了通知を仕組み化する

タスクを完了した側が、下流のメンバーにSendMessageで通知する:

```
designer → shipper: 「スクショ完成、パスは ~/screenshots/v083/」
```

これにより、shipperはTaskListをポーリングしなくてもブロック解除を即座に知れる。

### 4. タスク粒度は「1人が1-2時間で完了」

大きすぎるタスク（「ゲームを完成させる」）は進捗が見えない。小さすぎるタスク（「変数名を変える」）は管理コストが上回る。
