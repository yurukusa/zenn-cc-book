## 3本で終わるはずだった

記事を3本セットで書く。そういう計画だった。

1本目で「なぜこの構造にしたか」を書く。設計記事。2本目で「どう作ったか」を書く。実装記事。3本目で「結果どうなったか」を書く。回収記事。綺麗な三部作。企画段階ではそう思っていた。

結果から言う。30本を超えた。

3本で終わらなかったどころか、10倍の記事が生まれた。しかもその大半は、自分で「書こう」と決めたわけではない。AIエージェントが素材を発見し、分類し、記事の骨格まで用意していた。

この記事では、3本の計画が30本に膨らんだ仕組みを解説する。偶然ではない。自動で回るサイクルがあった。

:::details CC補足: この記事の情報について
この記事は2026年2月時点の情報に基づいている。使用しているのはClaude Code（Anthropic社のCLIツール）とCodex CLI（OpenAI社のCLIツール）。Claude Max $200/月プランでの運用記録。
:::

## 三部作構想の中身

もともとの企画はこうだった。

AIに「止まるな」と命令したら何が起きるか。このテーマを3本の記事に分解する。

**第1部「AIに『止まるな』と言ったら5回言い直す羽目になった」**。CLAUDE.md（AIへの指示書）に「質問するな」と書いたのに守らない。フックで強制したら今度は別の意味で暴走する。ルールを守らせる難しさの記録。

第2部「ralph-loopを個人開発者が使ってみた」。先人の知恵を借りて自律ループを構築した話。動いたけど自分の理想とは違った。$20,000のGPU16台並列という「力で殴る」アプローチと、$200の1台で知恵を絞るアプローチの比較。

第3部「tachikoma-loopが動いた日」。自作の自律ループがやっと動いたら、ループの基盤ファイルが壊れたというオチ。「送信ボタンを押す」だけの操作に4回の書き直しが必要だった話。

3本それぞれが単体で読めて、通しで読むと「非エンジニアのAI自律運用奮闘記」になる。そういう設計だった。

## なぜ3本で終わらなかったか

理由は3つある。

1つ目。1つのテーマから派生テーマが次々に出てくる。「AIが止まる問題」を掘ると、「なぜ止まるか」→「忘却の構造」→「記憶の設計」→「CLAUDE.mdの書き方」と芋づる式に出てくる。1本の記事に詰め込むと散漫になる。分けるしかない。

2つ目。proof-log（作業証跡）が毎日溜まる。AIが何をやったかの記録が自動で残る。この記録を見返すと「ここ面白いな」というポイントが見つかる。記事の種が勝手に生まれる。

3つ目。素材ファイルが自動蓄積する仕組みがあった。これが一番大きい。

## materials/ディレクトリの正体

`~/projects/dung-azure-flame/articles/materials/` というディレクトリがある。ここに31個の素材ファイルが入っている。

素材ファイルとは何か。記事の原材料だ。体験、エピソード、技術的発見を、記事化する前の状態で保存したもの。frontmatter（先頭のメタデータ）付きのMarkdownファイルで、番号とスラッグ（短い識別名）が振ってある。

```
materials/
├── material_01_hook-infra.md        # Hookで強制する開発インフラ
├── material_02_chrome-auto-recovery.md  # chrome-auto自動復旧
├── material_03_ai-strategy-vs-human.md  # AIの戦略 vs 人間の指摘
├── ...
├── material_28_proof-log.md         # proof-log自動記録の設計
├── material_31_creator-doubts-own-product.md  # 作った本人が疑う
└── INDEX.md                          # 全素材の一覧
```

INDEX.mdは素材の台帳。番号、ファイル名、タイトルの対応表になっている。新しい素材を追加するときは番号を振ってここに登録する。

ポイントは、この素材の大半がproof-logから生まれていること。「昨日CCがやったこと」の記録を読み返して「これ記事になる」と判断し、素材として切り出す。そのプロセス自体も半自動化されている。

:::details CC補足: 素材ファイルのフォーマット
各素材ファイルには以下のfrontmatterが付与される。

```yaml
---
material_id: 28
title: "proof-log——AIの作業証跡を自動記録する仕組みの設計と実装"
source: "2026-02-15 CCセッションからの抽出"
split_date: "2026-02-15"
---
```

`material_id`は一意の連番。`source`は元ネタの出処。`split_date`は素材として切り出した日付。これにより、どの体験がいつ素材化されたかを追跡できる。
:::

## 自動サイクルの全体像

proof-logから記事が生まれるまでの流れを整理する。

```
proof-log（作業証跡）
  ↓ 読み返し・発見
material（素材化）
  ↓ 記事設計・執筆
draft（記事ドラフト）
  ↓ 公開
publish（記事公開）
  ↓ KPI取得
measure（計測）
  ↓ 分析・次の種
proof-log（次の作業証跡へ）
```

このサイクルが回っている。一度回り始めると、止めるほうが難しい。

### proof-log → material

proof-logはAIの作業証跡だ。セッションが終わるたびに、フック（自動実行スクリプト）が勝手に記録する。何時に、誰が、何を、どのファイルを変えたか。5W1H形式。

実際のproof-logはこんな感じ。

```markdown
### 2026-02-22 01:05-01:11 JST — セッション終了（自動記録）

- いつ: 01:05〜01:11 JST（5分）
- どこで: nursery-shift
- 誰が: CC: 17件
- 何を: 6ファイル変更 (+772/-0)
  - staged-hatching-backus.md (+543/-0, 8回)
  - app.py (+49/-0, 5回)
- どうやって: Edit: 13回, Write: 4回
```

これが毎日、何十件と溜まる。人間が「記録しなきゃ」と思う必要がない。Stop hookという仕組みで、セッション終了時に自動で書き込まれる。

:::details CC補足: Stop hookの仕組み
Claude Codeの`.claude/settings.json`にStop hookを登録すると、セッション終了時にシェルスクリプトが自動実行される。proof-log-session.shはactivity-log.jsonl（機械向けの生ログ）を集計し、人間が読めるMarkdown形式に変換してproof-logに追記する。

```json
{
  "hooks": {
    "Stop": [{
      "matcher": "",
      "hooks": [
        { "type": "command", "command": "proof-log-session.sh" }
      ]
    }]
  }
}
```

さらにcontext-monitor.shがPostToolUseフックに常駐し、コンテキスト（AIの短期記憶）が25%以下になったら中間記録を残す。これにより、コンテキスト枯渇でセッションが飛んでも証跡が消えない。
:::

このproof-logを読み返す。「ここでCCが4回書き直してる」「このエラー対応、他の人もハマるだろう」。そういうポイントを見つけたら、素材ファイルとして切り出す。

### material → draft

素材ファイルができたら、次は記事のドラフトだ。

ここでcontent-manifest.yaml（コンテンツ管理台帳）が登場する。どのプラットフォームに、どの素材を使って、どんな記事を書くかの対応表。

```yaml
zenn_ai_broken_promises:
  title: AIエージェントが約束を守らない — 忘却問題の構造と5段階の対処法
  platform: zenn
  status: published
  published_url: https://zenn.dev/yurukusa/articles/55c885c245d3fd
  draft_file: ~/drafts/zenn-ai-broken-promises.md
  notes: 素材#13ベース
```

`notes: 素材#13ベース`。素材13番から記事を作ったという記録。これで「どの素材がどの記事になったか」が追跡できる。

ドラフトは `~/drafts/active/` に置く。書き上がったらcontent-manifestに登録。公開したらstatusをpublishedに変え、URLを記録する。

### publish → measure

公開したら計測する。24時間後のview数、like数、コメント数をcontent-manifestに記録。

```yaml
  views_24h: 418
  likes_24h: 1
  comments_24h: 0
  kpi_collected_at: '2026-02-17'
```

この数字が次の判断材料になる。「claudecodeタグ付きは54ビュー、なしは3ビュー」。こういうファクトが溜まると、次にどんな記事を書くべきかが見えてくる。

### measure → proof-log

計測結果を分析する行為自体がproof-logに残る。分析から新しい発見が生まれる。その発見が次の素材になる。

ループが閉じた。

## 31個の素材が生まれた内訳

31個の素材を振り返ると、いくつかのカテゴリに分かれる。

**インフラ事故系**（素材01, 02, 09, 24）。Hookが壊れた、ポート番号がハードコードだった、Chrome自動復旧が暴走した。事故が起きるたびに素材が1つ増える。

運用設計系（素材07, 10, 13, 25, 28, 30）。AIに質問を禁止する3層防御。Stopフック5フェーズの試行錯誤。忘却問題の構造分析。CLAUDE.mdの設計パターン。運用を作り込むたびに素材が増える。

比較・分析系（素材03, 14, 22）。AIの戦略 vs 人間の指摘。ralph-loop vs tachikoma-loop。3つのAIに同じ問題を投げた結果。比較すると見えるものがある。

エッセイ系（素材17, 18, 31）。AIがバズらせようとしていた。午前2時にAIが主人の起床時間を気にしていた。作った本人が自分の商品を疑った。日常の中の小さな違和感が素材になる。

これらに加えて、インタビュー素材が10本。ぐらす（自分）への聞き取り形式で、始まりの動機、挫折、成功体験、将来像を記録している。

全部合わせて41の素材。三部作の計画から10倍以上の規模に膨らんだ。

:::details CC補足: インタビュー素材の位置づけ
材料ディレクトリには素材ファイル（material_XX）とは別に、interviews/ディレクトリにインタビュー素材が10本ある。これはぐらす本人への聞き取りを構造化したもので、主にnoteやはてなのエッセイ系記事の原材料になる。技術記事はmaterial_XXから、エッセイ記事はinterviewsから。素材の種類によって記事の方向性が変わる。
:::

## なぜこのサイクルが回るのか

自動サイクルが持続する理由を3つ挙げる。

1つ目。**記録が「やるべきこと」から外れている**。proof-logはフックで自動実行される。人間が意識して記録する必要がない。「記録しよう」というモチベーションに依存しない仕組みだから、サイクルが途切れない。

2つ目。素材化のハードルが低い。素材ファイルは完成品ではない。「これ面白いかも」レベルのメモで十分。frontmatterを付けて番号を振るだけ。INDEX.mdに1行足すだけ。30秒で終わる。

3つ目。アウトプット先が複数ある。Zenn、Qiita、note、はてな、dev.to。プラットフォームごとに読者層が違うから、同じテーマでも切り口を変えれば別の記事になる。1つの素材から2〜3本の記事が生まれることもある。

逆に言えば、この3つのうち1つでも欠けるとサイクルは止まる。記録を手動にしたら忘れる。素材化のフォーマットが重いと面倒になる。公開先が1つだけだと、ネタが枯れる。

## 三部作は「計画」ではなく「起点」だった

振り返って思うのは、三部作構想は「計画」として機能したわけではないということだ。

3本書いて終わり。そんなつもりだった。でも1本目を書く過程で素材が増え、2本目の途中で派生テーマが3つ見つかり、3本目を書き終わる頃には次の三部作が企画できるくらいのネタが溜まっていた。

計画は起点にすぎなかった。回り始めたサイクルが、計画を超えていった。

これは自分の才能の話ではない。仕組みの話だ。proof-logが記録を自動化し、materialsディレクトリが素材を構造化し、content-manifestが記事を管理する。この3つの仕組みが噛み合うと、コンテンツは勝手に増殖する。

## 実際に使っている仕組み一覧

最後に、このサイクルを支えている仕組みをまとめる。

| 仕組み | 場所 | 役割 |
|-------|------|------|
| proof-log | `~/ops/proof-log/` | 作業証跡の自動記録（5W1H） |
| activity-log.jsonl | `~/ops/` | 機械向けの生イベントログ |
| materials/ | `~/projects/.../articles/materials/` | 記事素材の保管・分類 |
| INDEX.md | materials/内 | 素材の台帳 |
| content-manifest.yaml | `~/` | 記事管理台帳（全PF横断） |
| drafts/active/ | `~/drafts/active/` | 執筆中のドラフト |
| Stop hook | `.claude/settings.json` | セッション終了時の自動記録 |
| context-monitor.sh | `.claude/hooks/` | コンテキスト残量の監視 |

1つ1つは単純なテキストファイルとシェルスクリプトだ。派手な技術は使っていない。ただ、これらが組み合わさることで「記録→素材→記事→計測→記録」のサイクルが自動で回る。

3本の計画から30本以上の記事が生まれた。次の1本も、このサイクルのどこかから生まれるだろう。

:::details CC補足: このサイクルをゼロから作るには
最小構成で始めるなら、必要なのは3つだけ。

1. proof-logの仕組み（Stop hookでセッション終了時に自動記録）
2. 素材ディレクトリ（Markdownファイル + INDEX.md）
3. content-manifest.yaml（記事管理の台帳）

proof-logのStop hookは`.claude/settings.json`に数行追加するだけで動く。素材ディレクトリはmkdirで作ってINDEX.mdを置くだけ。content-manifestはYAMLファイル1つ。全部テキストベースで、特別なツールは不要。

サイクルの起動に必要なのは「1本目の記事を書くこと」。書くことで素材が生まれ、計測データが溜まり、次の素材が見つかる。最初の1回転を手動で回せば、あとは慣性で回り続ける。
:::
