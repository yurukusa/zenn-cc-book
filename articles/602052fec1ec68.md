## 起きていること

2026 年 5 月 8 日の夜、 Claude Code の VS Code 拡張の v2.1.136 が、 Windows で起動しなくなった。 GitHub の本家の起票の場で、 同じ症状の起票が 1 時間半で 7 件以上同時に立った。 5/8 21:31 から 22:30 までの 1 時間で、起票の番号は #57433 から #57467 まで、独立の起票者から 7 件。共通の原因が一つ。

CLI のほうは無事。 npm で入れた `claude` コマンドは普段どおり動く。問題は VS Code の拡張のほうだけ。

拡張の有効化が、起動時に静かに落ちる。サイドバーには Claude のアイコンが出ない。 VS Code の通知の場所にも、エラーは見えない。利用者の側からは、何も起きていないように見える。

## 1 分で自分が当たっているか確かめる方法

VS Code を開いて、「コマンドパレット」 を出す。`Developer: Show Logs...` を選び、`Extension Host` を開く。直近のログの中に、次の文字列があれば、当たっている。

```
TypeError: The argument 'filename' must be a file URL object, file URL string, or absolute path string.
Received 'file:///home/runner/work/claude-cli-internal/claude-cli-internal/build-agent-sdk/sdk.mjs'
```

`/home/runner/work/...` の文字列が見えるかどうか、これだけで判定できる。文字列が無ければ、別の問題なので、この記事の対象ではない。

## なぜ Windows でだけ落ちるか

拡張の中で `Module.createRequire()` という API が呼ばれている。この API は、ファイルへの絶対のパス、もしくは妥当なファイル URL の文字列を引数として要求する。

v2.1.136 で配布された Windows 用の拡張のバンドルに、 GitHub Actions の Linux のビルドの実行の場の絶対のパスが、ビルドの時点で文字列として埋め込まれていた。

```
file:///home/runner/work/claude-cli-internal/claude-cli-internal/build-agent-sdk/sdk.mjs
```

Linux の絶対のパスなので、 Windows の Node.js は受け取れない。 Windows のファイル URL は、 `file:///c:/...` のようにドライブ文字を含む形式が必要で、 Linux のパスは通らない。

結果として、拡張の有効化が `TypeError` で落ちる。 Linux と macOS の利用者は、同じ Linux のパス文字列でも Node.js が許容するため、表に出ない。 Windows でだけ、起動の最初の瞬間に止まる。

これは、ビルドの時点で `import.meta.url` などの値が定数として置き換えられた結果だと思われる。 Linux のビルドの実行の場で生成された定数を、 Windows 用の配布物にそのまま埋め込んだのが原因。

## 暫定の回避策

公式の修正版が出るまでの間、利用者の側で文字列を書き換えれば動く。 PowerShell で次の手順を実行する。

```powershell
$ext = "$env:USERPROFILE\.vscode\extensions\anthropic.claude-code-2.1.136-win32-x64\extension.js"
Copy-Item $ext "$ext.bak"
$bad  = 'file:///home/runner/work/claude-cli-internal/claude-cli-internal/build-agent-sdk/sdk.mjs'
$good = 'file:///c:/Users/' + $env:USERNAME + '/.vscode/extensions/anthropic.claude-code-2.1.136-win32-x64/extension.js'
[IO.File]::WriteAllText($ext, ([IO.File]::ReadAllText($ext)).Replace($bad, $good))
```

この後、 VS Code を再読み込みすると、拡張が正常に有効化される。 Claude のサイドバーが現れる。

回避策の元は、起票 #57467 の本文の suggested workaround の節。書き換える先のパスを、自分の環境のユーザー名に合わせて自動で組み立てる形に少しだけ変えてある。

`extension.js` の中で、同じ文字列が 2 箇所に現れる (起票 #57467 の解析では 103 行と 204 行)。`Replace()` メソッドは全件を置き換えるので、 1 回の実行で両方とも直る。

## 戻すには

回避策を適用した後で、公式の修正版が出た場合、 `.bak` のファイルから戻せる。

```powershell
$ext = "$env:USERPROFILE\.vscode\extensions\anthropic.claude-code-2.1.136-win32-x64\extension.js"
Copy-Item "$ext.bak" $ext -Force
```

公式の修正版が出れば、拡張の自動の更新で v2.1.137 以降に移ればこの問題は解消する。手動の回避策を残したまま新しい版に移っても、新しいバンドルのファイルが置き換わるので衝突はしない。

## 直るのを待つ場合の代替

回避策を入れたくない場合、 v2.1.135 にダウングレードする選択肢がある。 VS Code の拡張の管理の画面で、 Claude Code の拡張の右の歯車から、「特定のバージョンをインストール」 を選ぶ。 v2.1.135 を選べば、 v2.1.136 の問題のあるバンドルを避けられる。

ただし、 v2.1.135 にダウングレードすると、 v2.1.136 の新機能 (`claude-vscode.editor.openLast` など) は使えなくなる。何も新機能を使っていなければ、ダウングレードのほうが回避策の書き換えより確実。

## まとめ

v2.1.136 の Windows 版の VS Code 拡張は、ビルド時に Linux のパスが定数として埋め込まれていて、 Windows で起動しない。 5/8 夜から、本家の起票の場で 7 件以上の起票が同時に立っている。利用者の側では、ログに `/home/runner/work/...` の文字列が見えれば対象。

回避策は、 PowerShell で `extension.js` の文字列を 2 箇所書き換える。または v2.1.135 にダウングレード。公式の修正版が出るまでの間、どちらかを選ぶ。

CLI のほうは別のバンドルなので無事。 VS Code の拡張だけの問題。
