## 起きていること

2026年5月8日の夜、Claude CodeのVS Code拡張v2.1.136がWindowsで起動しなくなった。GitHubの本家の起票の場で、同じ症状の起票が1時間半で7件以上同時に立った。5/8 21:31から22:30までの1時間で、起票の番号は#57433から#57467まで、独立の起票者から7件。共通の原因が一つ。

CLIのほうは無事。npmで入れた`claude`コマンドは普段どおり動く。問題はVS Code拡張のほうだけ。

拡張の有効化が起動時に静かに落ちる。サイドバーにClaudeのアイコンが出ない。VS Codeの通知の場所にもエラーは見えない。利用者の側からは何も起きていないように見える。

## 1分で自分が当たっているか確かめる方法

VS Codeを開いて、「コマンドパレット」を出す。`Developer: Show Logs...`を選び、`Extension Host`を開く。直近のログの中に次の文字列があれば、当たっている。

```
TypeError: The argument 'filename' must be a file URL object, file URL string, or absolute path string.
Received 'file:///home/runner/work/claude-cli-internal/claude-cli-internal/build-agent-sdk/sdk.mjs'
```

`/home/runner/work/...`の文字列が見えるかどうか、これだけで判定できる。文字列が無ければ、別の問題なので、この記事の対象ではない。

## なぜWindowsでだけ落ちるか

拡張の中で`Module.createRequire()`というAPIが呼ばれている。このAPIは、ファイルへの絶対のパス、もしくは妥当なファイルURLの文字列を引数として要求する。

v2.1.136で配布されたWindows用の拡張のバンドルに、GitHub ActionsのLinuxのビルドの実行の場の絶対のパスが、ビルドの時点で文字列として埋め込まれていた。

```
file:///home/runner/work/claude-cli-internal/claude-cli-internal/build-agent-sdk/sdk.mjs
```

Linuxの絶対のパスなので、WindowsのNode.jsは受け取れない。WindowsのファイルURLは`file:///c:/...`のようにドライブ文字を含む形式が必要で、Linuxのパスは通らない。

結果として、拡張の有効化が`TypeError`で落ちる。LinuxとmacOSの利用者は、同じLinuxのパス文字列でもNode.jsが許容するため、表に出ない。Windowsでだけ、起動の最初の瞬間に止まる。

これは、ビルドの時点で`import.meta.url`などの値が定数として置き換えられた結果だと思われる。Linuxのビルドの実行の場で生成された定数を、Windows用の配布物にそのまま埋め込んだのが原因。

## 暫定の回避策

公式の修正版が出るまでの間、利用者の側で文字列を書き換えれば動く。PowerShellで次の手順を実行する。

```powershell
$ext = "$env:USERPROFILE\.vscode\extensions\anthropic.claude-code-2.1.136-win32-x64\extension.js"
Copy-Item $ext "$ext.bak"
$bad  = 'file:///home/runner/work/claude-cli-internal/claude-cli-internal/build-agent-sdk/sdk.mjs'
$good = 'file:///c:/Users/' + $env:USERNAME + '/.vscode/extensions/anthropic.claude-code-2.1.136-win32-x64/extension.js'
[IO.File]::WriteAllText($ext, ([IO.File]::ReadAllText($ext)).Replace($bad, $good))
```

この後VS Codeを再読み込みすると、拡張が正常に有効化される。Claudeのサイドバーが現れる。

回避策の元は起票#57467の本文のsuggested workaroundの節。書き換える先のパスを、自分の環境のユーザー名に合わせて自動で組み立てる形に少しだけ変えてある。

`extension.js`の中で、同じ文字列が2箇所に現れる(起票#57467の解析では103行と204行)。`Replace()`メソッドは全件を置き換えるので、1回の実行で両方とも直る。

## 戻すには

回避策を適用した後で公式の修正版が出た場合、`.bak`のファイルから戻せる。

```powershell
$ext = "$env:USERPROFILE\.vscode\extensions\anthropic.claude-code-2.1.136-win32-x64\extension.js"
Copy-Item "$ext.bak" $ext -Force
```

公式の修正版が出れば、拡張の自動の更新でv2.1.137以降に移ればこの問題は解消する。手動の回避策を残したまま新しい版に移っても、新しいバンドルのファイルが置き換わるので衝突はしない。

## 直るのを待つ場合の代替

回避策を入れたくない場合、v2.1.135にダウングレードする選択肢がある。VS Codeの拡張の管理の画面で、Claude Codeの拡張の右の歯車から「特定のバージョンをインストール」を選ぶ。v2.1.135を選べば、v2.1.136の問題のあるバンドルを避けられる。

ただしv2.1.135にダウングレードすると、v2.1.136の新機能(`claude-vscode.editor.openLast`など)は使えなくなる。何も新機能を使っていなければ、ダウングレードのほうが回避策の書き換えより確実。

## まとめ

v2.1.136のWindows版のVS Code拡張は、ビルド時にLinuxのパスが定数として埋め込まれていて、Windowsで起動しない。5/8夜から本家の起票の場で7件以上の起票が同時に立っている。利用者の側では、ログに`/home/runner/work/...`の文字列が見えれば対象。

回避策はPowerShellで`extension.js`の文字列を2箇所書き換える。またはv2.1.135にダウングレード。公式の修正版が出るまでの間、どちらかを選ぶ。

CLIのほうは別のバンドルなので無事。VS Code拡張だけの問題。
