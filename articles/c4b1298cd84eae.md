## 起きたこと

2026年4月16日から5月15日までの30日間で、Claude Codeの`/ultrareview`の機能に対するGitHub Issuesの問題報告が累計198件に達した。6.6件/日の継続の発生率、113件がopen、85件がclosed、そのうち5件は`not planned`のdispositionで技術者の説明のコメントなしで閉じられている。

加えて、上位の技術者であるBoris Cherny (bcherny) の本件の関連のコメントは累計0件。直近の30日間で198件の問題報告が累積している機能に対して、公式のClaude Codeの事故の対応の中心の人物が一度もコメントしていない状態である。

利用者の側の影響は、4つの失敗の様式で整理できる。そのうち3つが、課金は進行するがreviewの出力は返らない構造になっている。つまり、利用者はExtra Usageの課金を払いながら結果を受け取れない状態である。

## 再現できる数値の取得

ご自身の端末で本記事の数値を取得する経路は次の通り。

```bash
gh api '/search/issues?q=repo:anthropics/claude-code+ultrareview+created:>=2026-04-16'
```

結果として`total_count`が198件で返る。stateによる分離は次の通り。

```bash
gh api '/search/issues?q=repo:anthropics/claude-code+ultrareview+created:>=2026-04-16+state:open'
gh api '/search/issues?q=repo:anthropics/claude-code+ultrareview+created:>=2026-04-16+state:closed'
```

直接の課金の言及 (Extra Usage) の限定の取得は次の通り。

```bash
gh api '/search/issues?q=repo:anthropics/claude-code+ultrareview+%22Extra+Usage%22'
```

結果は20件。これは利用者が報告の本文でExtra Usageの状態を明示的に書いた件数で、Shape A (free-runの消費) を含めた実際の課金の発生の母数はその数倍と推定される。

## 4つの失敗の様式

**Shape A — 中断 + free-runのquotaの消費**: 163件の中の初期の21日間の窓で119件 (73%)。最も高い発生率。jobがdispatchに到達し、per-fileの分析のbudgetでfree-runのquotaを消費した後、findingsを返さずに分析のloopを抜ける。利用者は中断の合図を見るが、quotaは消費済の状態。

**Shape B — 完了のstatus + 空のfindings + Extra Usageの課金**: Issue [#56775](https://github.com/anthropics/claude-code/issues/56775) で 7,000行の差分に対して `status: completed` と `findings: []` を返した後で約20米ドルのExtra Usageの課金が確定。「完了」と表示されるがreviewの結果がない状態。

**Shape C — 内部の例外 + 部分の課金**: Issue [#58791](https://github.com/anthropics/claude-code/issues/58791) で、大きな差分で内部の例外が発生してJSONのreturnの経路に到達せず、部分の作業の課金が利用者に確定する。

**Shape D — 同日の繰り返しの失敗の全件の課金**: Issue [#58019](https://github.com/anthropics/claude-code/issues/58019) で、利用者が同日に3件のpull requestに対して`/ultrareview`を実行、全件でcrash、6回のExtra Usageの課金が発生、findingsは0件の状態。

## 根本原因

4つの様式の中核の根本原因は、dispatchとfindingsのdeliveryの間に取引の境界が無い事である。課金の事象はdispatchで発火する。findingsの事象は完了で発火する。findingsの事象が失敗した場合や空の結果を返した場合に、課金の事象を巻き戻す経路が無い。

利用者の側から見ると、ある機能を起動して、課金が発生して、結果が返ってこない状態。これは商業の場の対面の取引で言えば、店舗が代金を受け取った後で商品を渡さずに店を閉める行為に近い構造。

## 検出のhookの実装

検出のhookの設計は、advisoryが既定で、環境の変数`CC_ULTRAREVIEW_STRICT=1`でexit code 2の強制に上げる構造。`Stop`の事象で`/ultrareview`の起動の窓のfindingsの数とExtra Usageの差分を取得、findingsが0件かつExtra Usageが$5を超える場合に警告。

```bash
#!/usr/bin/env bash
# ultrareview-empty-paid-charge-detector.sh
# Stop の事象で Shape B と Shape C の検出

set -euo pipefail

INVOCATION_LOG="${CLAUDE_HOME:-$HOME/.claude}/logs/ultrareview-$(date +%Y%m%d).jsonl"
FINDINGS_COUNT=$(jq -r 'select(.event == "task-notification" and (.command // "" | contains("ultrareview"))) | .findings | length' "$INVOCATION_LOG" 2>/dev/null | tail -1)
USAGE_DELTA=$(jq -r 'select(.event == "extra-usage-delta") | .amount_usd' "$INVOCATION_LOG" 2>/dev/null | tail -1)

if [[ -z "${FINDINGS_COUNT:-}" || -z "${USAGE_DELTA:-}" ]]; then
  exit 0
fi

if [[ "${FINDINGS_COUNT}" -eq 0 && $(echo "${USAGE_DELTA} > 5" | bc -l) -eq 1 ]]; then
  echo "ALERT: /ultrareview returned 0 findings against \$${USAGE_DELTA} Extra Usage charge."
  echo "Reference: https://github.com/anthropics/claude-code/issues?q=is%3Aissue+ultrareview"
  if [[ "${CC_ULTRAREVIEW_STRICT:-0}" == "1" ]]; then
    exit 2
  fi
fi

exit 0
```

このhookは [cc-safe-setup](https://github.com/yurukusa/cc-safe-setup) (MIT、730件以上の安全装置のhookの集まり) と同じ場所に保管。`~/.claude/settings.json`の`Stop` hookの配列にそのまま組み込み可能。

## 回避の手順

直近の30日間で技術者の応答が0件の状態で、`/ultrareview`を安全に扱うための手順は次の3つ。

ひとつ目、Extra Usageの課金の利用者は、500行を超える差分に対する`/ultrareview`の起動を回避する。500行以下の失敗の発生率はclusterの中で大きく下がる。

ふたつ目、free-runのquotaの利用者は、73%のShape Aの発生率を前提に予算を設定する。失敗のquotaの消費を吸収可能な時のみ起動する。routineのreviewは`git diff`か`cline review`の代替の経路を選ぶ。

みっつ目、失敗の事例はGitHub Issuesに報告する。clusterの真の規模は、報告の数を直接の取得の経路で利用者が確認した時のみ可視で、公式のtriageのduplicate auto-closeの経路でUIのcluster表示の数は実際より少なく見える。

## 監視している3つの合図

clusterは200件の閾値の手前で残り2件。6.6件/日の継続で、数日以内の超過の確度が極めて高い。直近の応答が変化した時の入力は次の3つ。

ひとつ目、198件のいずれか1件に対して、bcherny か Anthropic の技術者がbilling-without-deliveryのconditionを明示的にacknowledgmentする。

ふたつ目、point releaseでdispatchとfindingsのdeliveryのtransactional boundaryが追加される (失敗のrollback)。

みっつ目、changelogの項目で「ultrareview billing reconciliation」の言及。

いずれか1件が発火するまで、上記のhookは最も安価なmitigationの経路。
