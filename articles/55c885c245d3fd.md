## 「決めたこと、忘れてくれるんですよ」

Claude Codeを毎日使い倒して数週間。正直に言う。

> Claudeがめちゃくちゃ俺が言ったことや決めたことを忘れてくれる。MCPやskillsがインストールされていても活用されない。話し合ってルールを決めても次のタイミングには忘れる

これは自分の生の感想。AIエージェントと一緒に作業していて、一番ストレスが溜まるのがこれ。

「質問するな、自分で決めろ」とCLAUDE.mdに書いた。5分後に「どちらがよろしいでしょうか？」と聞いてくる。

「MCPツールを優先的に使え」と明記した。次のセッションで存在自体を忘れてBashでゴリ押し始める。

「止まるな」と書いたら「了解です。」で止まる。いや止まってるじゃん。「止まるな」の意味を3回書き直した。

バグじゃない。これは**構造的な問題**だ。


## なぜAIは約束を忘れるのか

「ちゃんと伝えたのに」と思うかもしれないけれど、LLMの動作原理を考えると、忘れるのが正常で覚えているほうが例外になる。理由は4つある。

### 1. コンテキストウィンドウの限界

LLMが一度に「見える」テキストの量には上限がある。会話が長くなるほど古い情報は押し出される。100ページの本のうち、今開いているページしか読めないようなもの。最初のほうで交わした約束は、後半には視界から消えている。

### 2. コンパクション（要約圧縮）

Claude Codeはコンテキストが溢れそうになると、過去の会話を要約して圧縮する。このとき**ルールや約束が要約から落ちる**ことがある。「PUTの前にGETしろ」という教訓が、「API操作を行った」に圧縮されたら、もう教訓としては機能しない。

### 3. セッション交代

新しいセッションは、前のセッションの約束を知らない。昨日のセッションで「次からこうしよう」と合意しても、今日のセッションにとっては他人事。引き継ぎのない新入社員が毎日来るようなもの。

### 4. CLAUDE.mdの優先度問題

CLAUDE.mdに書いたルールは「読まれる保証」がない。読まれても、会話が進むにつれてコンパクションで落ちる可能性がある。設定ファイルに書いた＝守られる、とはならない。


## 実際に起きた事故

### Zenn記事5本消失事件

「Zennの記事にフッターを追加して」と頼んだ。AIはZenn APIに `PUT` リクエストを送った。フッターのテキストだけを。

RESTの `PUT` はリソース**全体**を置き換える。本文がフッター1行に上書きされた。

教訓を書いた。「PUTの前に必ずGETせよ」と。

翌日。**同じエージェントが同じミスをやった。** 今度は5記事全部。読者のコメントで気づいた。

教訓は書いてあった。読まれなかった。

### フック実装直後の自己違反

「ユーザーに質問するな」というフックを実装した。その直後のメッセージで「聞かせてほしい。」と言って止まった。

フックの実装自体は正しかった。でもフックが発火するのは**ツール呼び出し時**であって、会話テキストの中で質問するのは止められない。穴があった。


## 信頼性の5段階 — どこまでやれば「守らせる」ことができるか

試行錯誤の結果、AIにルールを守らせる方法には明確な信頼性の階層があることがわかった。

| レベル | 方法 | 信頼性 | 弱点 |
|:---:|------|:---:|------|
| 1 | CLAUDE.mdにルールを書く | 低 | コンパクションで落ちる |
| 2 | MEMORY.mdに記録する | 中低 | 参照するかはAI次第 |
| 3 | 教訓をYAMLで構造化して警告を出す | 中 | 警告はするが強制力なし |
| 4 | **フック（PreToolUse/PostToolUse）** | **高** | コンパクション後も自動発火 |
| 5 | MCPサーバで外部から制御 | 高 | 設定が複雑 |

ポイントは、**レベル1〜3は全て「AIが自発的に参照する」ことに依存している**ということ。AIが忘れたら終わり。

レベル4以降は違う。フックはClaude Codeのランタイムに組み込まれていて、**AIの記憶に関係なく**発火する。コンパクションされようが、セッションが交代しようが、ツール実行のたびに必ず走る。


## 実装例：フックで「忘却耐性」を作る

### レベル1：CLAUDE.mdにルールを書く（信頼性：低）

```markdown
# CLAUDE.md
## ルール
- ユーザーに質問するな。自分で判断しろ
- PUTの前にGETせよ
- MCPツールを優先的に使え
```

書くだけ。最初は読まれる。会話が長くなると忘れられる。

### レベル4：PreToolUseフックで強制する（信頼性：高）

```bash
#!/bin/bash
# no-ask-human.sh — 質問ブロックフック
# 発火条件: PreToolUse（AskUserQuestion実行前）
# 効果: exit 1 でツール実行をブロック

# 環境変数でオーバーライド可能（有人作業時用）
if [ "${CC_ALLOW_QUESTIONS:-0}" = "1" ]; then
    exit 0
fi

echo "BLOCKED: Do not ask the human." >&2
echo "判断の順序: 自分で判断 → 不明なら ~/pending_for_human.md に記録 → 別タスクに進む" >&2
exit 1
```

これを `.claude/settings.json` に登録する。

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "AskUserQuestion",
        "hooks": [
          {
            "type": "command",
            "command": "/home/user/.claude/hooks/no-ask-human.sh"
          }
        ]
      }
    ]
  }
}
```

AIが「質問しよう」としてAskUserQuestionツールを呼ぶ → フックが割り込んで `exit 1` → ツール実行がブロックされる → AIは質問できない。

**AIがルールを忘れても、フックは忘れない。**

### もう一つの例：ファイル編集後に構文チェックを強制する

```bash
#!/bin/bash
# syntax-check.sh — 編集後に構文検証を自動実行
# 発火条件: PostToolUse（Edit/Write実行後）

INPUT=$(cat)
FILE_PATH=$(echo "$INPUT" | jq -r '.tool_input.file_path // empty')

if [[ -z "$FILE_PATH" || ! -f "$FILE_PATH" ]]; then
    exit 0
fi

EXT="${FILE_PATH##*.}"

case "$EXT" in
    py)
        python -m py_compile "$FILE_PATH" 2>&1 || \
            echo "Python syntax ERROR in: $FILE_PATH" >&2
        ;;
    sh|bash)
        bash -n "$FILE_PATH" 2>&1 || \
            echo "Shell syntax ERROR in: $FILE_PATH" >&2
        ;;
    json)
        jq empty "$FILE_PATH" 2>&1 || \
            echo "JSON syntax ERROR in: $FILE_PATH" >&2
        ;;
esac

exit 0
```

「構文チェックしてね」と頼む必要がない。ファイルを編集するたびに自動で走る。AIが構文チェックの存在を忘れても問題ない。


## レベル3の実装例：教訓のコード化

フックほどの強制力はないけれど、「ただのテキスト」よりは効くのがレベル3の構造化教訓。

```yaml
# ~/.brain/lessons/api-put-safety.yaml
id: api-put-safety
title: "PUT前にGETを必ず実行"
severity: critical
trigger_patterns:
  - "PUT"
  - "api.*update"
checklist:
  - "GETで現在のデータを取得したか？"
  - "差分だけでなく全フィールドを含めているか？"
context: "2026-02-09にZenn記事5本がPUT上書きで消失した"
```

教訓をYAMLで定義して、トリガーパターンにマッチしたら警告を出す。これをPreToolUseフックから呼べばレベル4に昇格する。呼ばなくてもAIが参照すれば効くので、コストパフォーマンスは悪くない。


## 「AIにルールを守らせる」は「人間にルールを守らせる」と同じ問題

ここまで書いてきて気づいたことがある。

AIエージェントのルール忘却問題は、組織マネジメントの問題と**完全に同じ構造**をしている。

- **口頭で伝える**（CLAUDE.md）→ 忘れる。人間だって口頭指示は忘れる
- **マニュアルに書く**（MEMORY.md）→ 読まれない。人間だってマニュアルは読まない
- **研修で教える**（構造化教訓）→ 効果は一時的。人間だって研修内容は忘れる
- **システムで強制する**（フック）→ 従わざるを得ない。人間だってシステム制約には逆らえない

つまり、AIエージェントの管理は**組織設計の問題**だ。

「ちゃんと伝えたのに守ってくれない」と嘆くのは、マネージャーが「ちゃんと言ったのに部下が守らない」と嘆くのと同じ。個人の意志に依存するルールは破られる。依存しない仕組みを作ったルールだけが生き残る。

AIエージェントの時代に必要なのは、プロンプトエンジニアリングだけじゃない。**仕組みエンジニアリング**だ。


## まとめ：忘れる前提で設計する

AIエージェントは忘れる。これはバグじゃなくて仕様。LLMのアーキテクチャ上、避けられない。

だから「忘れないようにする」ではなく「**忘れても大丈夫なようにする**」が正しい設計方針。

1. **重要度が低いルール** → CLAUDE.mdに書く（忘れても被害が小さい）
2. **中程度のルール** → 構造化教訓で警告を出す（忘れても気づける）
3. **絶対に守らせたいルール** → フックで強制する（忘れても発火する）

「信頼するけど検証する（Trust but verify）」。冷戦時代の核軍縮交渉で使われたこのフレーズが、まさかAIエージェント運用の最適解になるとは思わなかったけど。

AIに約束を守らせたかったら、約束じゃなくて仕組みを作ろう。人間相手でも、AI相手でも、結局やることは同じだった。
