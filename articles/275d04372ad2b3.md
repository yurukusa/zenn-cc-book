Claude Code を140時間以上動かしてきた。APIも叩いた。それなりに分かっているつもりだった。

Anthropicが公式に出している無料の認定コース（Anthropic Academy）を全12コース受けてみた。クイズで間違えた問題を見返したら、「知ったつもり」がいくつも見つかった。

この記事はシリーズの第1回。テーマはプロンプトキャッシュ。

## 短いプロンプトにキャッシュを付けても無駄だった

`cache_control` を500トークン程度のシステムプロンプトに付けていた。「コスト削減できてるはず」と思い込んでいた。

エラーは出ない。警告もない。黙って素通りする。

**キャッシュは1024トークン未満だと無視される。**

これがAcademyのクイズで暴かれた最初の「知ったつもり」だった。自分はキャッシュの存在は知っていたが、最低サイズの制約を完全に見落としていた。

:::details CC補足: プロンプトキャッシュの基本メカニズム
Anthropic APIのプロンプトキャッシュは「プレフィックスベース」の仕組みだ。リクエストの先頭から連続するブロックを保存し、次回同じプレフィックスが来たときに再計算をスキップする。

Claudeはリクエスト処理後、内部計算結果を**全て破棄**する。毎回ゼロから計算し直す。キャッシュはこの破棄を防ぐ仕組みだ。

最小サイズが1024トークンに設定されている理由は、キャッシュ自体にオーバーヘッド（保存・インデックス・検索コスト）があるため。短すぎるプロンプトではオーバーヘッドが節約分を上回り、かえって非効率になる。

コスト構造:
- **キャッシュヒット時**: 入力トークン料金の10%（90%削減）
- **キャッシュ書き込み時**: 入力トークン料金の125%（初回は25%割高）
- **TTL（Time To Live）**: デフォルト5分（アクセスごとにリセット）。`"ttl": "1h"` を `cache_control` に指定すれば1時間に延長可能（追加コストあり）

キャッシュはデフォルトで無効。`cache_control`を明示的に追加して初めて動作する。
:::

## ブレークポイントの置き場所も間違えていた

ブレークポイント（キャッシュマーカー）の存在は知っていた。置く場所を間違えていた。

自分はsystem promptだけにブレークポイントを置いていた。15個のtool定義が毎回再処理されていた。

正解は**tools配列の最後のtool定義**に `cache_control` を付与すること。こうするとsystem prompt + 全tool定義がまとめて1つのプレフィックスとしてキャッシュされる。

```json
{
  "tools": [
    { "name": "tool_1", "description": "..." },
    { "name": "tool_2", "description": "..." },
    {
      "name": "tool_15",
      "description": "最後のツール",
      "cache_control": { "type": "ephemeral" }
    }
  ]
}
```

このコードで何が起きるか。`cache_control`を最後のtoolに置くことで、「ここまでの全内容をキャッシュしてくれ」というマーカーになる。次のリクエストで同じtools配列を送ると、15個分のtool定義の処理がスキップされる。

:::details CC補足: プレフィックスキャッシュの配置戦略
キャッシュはプレフィックス（先頭からの連続ブロック）を対象とする。ブレークポイントは「ここまでキャッシュしてくれ」というマーカーだ。

内部的にはtools → system prompt → messagesの順で結合される。つまり最後のtoolにブレークポイントを置くと、tools全体がキャッシュ対象になる。

**複数ブレークポイント戦略**（最大4つまで設定可能）:
1. 最後のtool定義 → tools全体をキャッシュ
2. system promptの最後 → system promptもキャッシュ
3. 会話履歴のある時点 → 過去のやり取りもキャッシュ

3段階でキャッシュすれば、会話が進んでmessagesが変わっても、toolsとsystem promptのキャッシュは有効なままだ。

ブレークポイントを付与できる場所: テキストブロック、画像ブロック、tool_use、tool_result、tool schema、system prompt。

重要な注意点: キャッシュが有効になるのは、ブレークポイントまでの内容が**完全に同一**の場合のみ。1文字でも変わるとキャッシュミスが発生する。
:::

## 「短手形式」ではキャッシュが使えない

もう一つ見落としていたポイントがある。APIリクエストのフォーマットの問題だ。

system promptを文字列で直接渡す短手形式（`"content": "あなたはアシスタントです"`）では、`cache_control`を付与する場所がない。

キャッシュを使うにはテキストブロックの長手形式にする必要がある。

```json
{
  "system": [
    {
      "type": "text",
      "text": "あなたはアシスタントです...",
      "cache_control": { "type": "ephemeral" }
    }
  ]
}
```

短手形式で書いていたコードをキャッシュ対応にするだけで、構造を書き換える必要がある。小さい話だが、知らないとハマる。

## フォローアップリクエストの罠

キャッシュが効くのは、ブレークポイントまでの内容が**完全に同一**の場合だけだ。

会話のたびにsystem promptのタイムスタンプを更新していたらどうなるか。毎回キャッシュミスする。25%の書き込みコストだけ払って、90%の削減は受けられない。むしろ損する。

キャッシュの恩恵を受けるには、「変わらない部分」と「変わる部分」を明確に分離する設計が必要だ。tool定義やsystem promptの固定部分をプレフィックスに、会話メッセージなどの動的部分を後方に配置する。

## コスト計算してみた

具体的にどれくらい差が出るか。

15個のtool定義（合計約3000トークン）を含むリクエストを100回送るケースで考える。

**キャッシュなし:**
- 3000トークン × 100回 = 300,000入力トークン分のコスト

**キャッシュあり:**
- 初回書き込み: 3000 × 1.25 = 3,750トークン分
- 残り99回ヒット: 3000 × 0.1 × 99 = 29,700トークン分
- 合計: 33,450トークン分

約89%の削減。100回のリクエストで266,550トークン分のコストが消える。

ただしこれは「完全ヒット」の理想値。実際にはsystem promptの微変更やTTL切れでミスが発生する。それでも、tool定義のような固定ブロックが大きいほど効果は大きい。
