## TL;DR

WSL2上のClaude CodeからWindows側のChromeを完全自動操作する仕組みを構築した。Chrome DevTools Protocol（CDP）とPowerShellの組み合わせで、スクリーンショット取得・要素クリック・テキスト入力・ページ遷移まで全て自動化できる。

この記事では、その実装方法と応用例を解説する。

## 問題：WSL2の壁

Claude Codeを日常的に使っていると、ブラウザ操作の自動化が欲しくなる場面が頻繁に訪れる。

- itch.ioにゲームをアップロードしたい
- Redditに投稿を書きたい
- Discordでメッセージを送りたい
- Webサービスのダッシュボードを操作したい

問題は、Claude CodeがWSL2上で動いていること。WSL2はLinux環境なので、Windows側のChromeに直接アクセスできない。既存のMCPツール（Puppeteer MCP等）もWSL2環境を想定していない。自分で作るしかなかった。

## 解決策：Chrome DevTools Protocol + PowerShell

### アーキテクチャ

```
┌─────────────────────────────────────┐
│  WSL2 (Linux)                       │
│  ┌─────────────────────────────┐    │
│  │  Claude Code                │    │
│  │  → Bashツールで             │    │
│  │    PowerShellスクリプト実行  │    │
│  └──────────┬──────────────────┘    │
│             │ powershell.exe 呼び出し│
└─────────────┼───────────────────────┘
              │
┌─────────────▼───────────────────────┐
│  Windows (ホスト)                    │
│  ┌─────────────────────────────┐    │
│  │  PowerShell                 │    │
│  │  → WebSocket接続            │    │
│  │  → CDPコマンド送信          │    │
│  └──────────┬──────────────────┘    │
│             │ CDP (WebSocket)       │
│  ┌──────────▼──────────────────┐    │
│  │  Chrome                     │    │
│  │  --remote-debugging-port    │    │
│  └─────────────────────────────┘    │
└─────────────────────────────────────┘
```

ポイントは2つ。

1. **WSL2からはPowerShellを直接呼べる** — `powershell.exe`はWSL2内からそのまま実行可能
2. **ChromeはCDPでリモート操作可能** — `--remote-debugging-port`フラグで起動すればWebSocket経由で制御できる

この2つを組み合わせれば、WSL2→Windows Chrome間のブリッジが完成する。

## 実装

### Step 1：Chromeをデバッグモードで起動

まず、CDPを有効にしたChromeを起動するスクリプトを用意する。

```powershell
# launch_chrome.ps1
# CDPポートを指定してChromeを起動する
# ポート9222はCDPのデファクトスタンダード

$chromePath = "C:\Program Files\Google\Chrome\Application\chrome.exe"
$debugPort = 9222

# 既存のChromeプロセスを確認
$existing = Get-NetTCPConnection -LocalPort $debugPort -ErrorAction SilentlyContinue
if ($existing) {
    Write-Host "Chrome already running on port $debugPort"
    exit 0
}

# デバッグモードで起動
# --user-data-dir を分けることで通常のChromeと共存可能
Start-Process $chromePath -ArgumentList @(
    "--remote-debugging-port=$debugPort",
    "--user-data-dir=C:\tmp\chrome-debug-profile"
)

# 起動待ち
Start-Sleep -Seconds 3
Write-Host "Chrome launched on port $debugPort"
```

WSL2からの呼び出し方：

```bash
# WSL2側からPowerShellスクリプトを実行
powershell.exe -ExecutionPolicy Bypass -File "$(wslpath -w ./launch_chrome.ps1)"
```

### Step 2：CDPエンドポイントの取得

CDPでChromeに接続するには、まずWebSocketのエンドポイントURLを取得する必要がある。

```powershell
# get_ws_endpoint.ps1
# CDPのHTTPエンドポイントからWebSocket URLを取得する

$debugPort = 9222

try {
    # /json/version でブラウザレベルのWebSocket URLが取れる
    $response = Invoke-RestMethod -Uri "http://localhost:$debugPort/json/version"
    $wsUrl = $response.webSocketDebuggerUrl
    Write-Output $wsUrl
} catch {
    Write-Error "Chrome is not running or CDP is not enabled"
    exit 1
}
```

ブラウザレベルのWebSocket URLは `ws://localhost:9222/devtools/browser/...` の形式。特定のタブに接続したい場合は `/json` エンドポイントで全タブのリストを取得し、各タブの `webSocketDebuggerUrl` を使う。

### Step 3：CDPコマンドの実行基盤

WebSocket接続してCDPコマンドを送受信するヘルパー関数を作る。

```powershell
# cdp_helper.ps1
# CDPコマンドをWebSocket経由で送信する汎用ヘルパー
# 全てのブラウザ自動操作はこの関数を通して実行される

function Send-CDPCommand {
    param(
        [Parameter(Mandatory=$true)]
        [System.Net.WebSockets.ClientWebSocket]$WebSocket,
        [Parameter(Mandatory=$true)]
        [string]$Method,
        [hashtable]$Params = @{},
        [int]$Id = 1
    )

    $command = @{
        id = $Id
        method = $Method
        params = $Params
    } | ConvertTo-Json -Depth 10

    # コマンド送信
    $bytes = [System.Text.Encoding]::UTF8.GetBytes($command)
    $segment = [ArraySegment[byte]]::new($bytes)
    $WebSocket.SendAsync(
        $segment,
        [System.Net.WebSockets.WebSocketMessageType]::Text,
        $true,
        [System.Threading.CancellationToken]::None
    ).Wait()

    # レスポンス受信
    $buffer = [byte[]]::new(1048576)  # 1MB バッファ
    $result = ""
    do {
        $segment = [ArraySegment[byte]]::new($buffer)
        $recv = $WebSocket.ReceiveAsync(
            $segment,
            [System.Threading.CancellationToken]::None
        ).Result
        $result += [System.Text.Encoding]::UTF8.GetString(
            $buffer, 0, $recv.Count
        )
    } while (-not $recv.EndOfMessage)

    return $result | ConvertFrom-Json
}

# WebSocket接続を確立する関数
function Connect-CDP {
    param([string]$WsUrl)

    $ws = [System.Net.WebSockets.ClientWebSocket]::new()
    $uri = [Uri]::new($WsUrl)
    $ws.ConnectAsync($uri, [System.Threading.CancellationToken]::None).Wait()
    return $ws
}
```

### Step 4：ページ遷移とスクリーンショット

CDPの基本操作として、ページ遷移とスクリーンショットの取得を実装する。

```powershell
# cdp_nav.ps1
# ページ遷移を実行し、読み込み完了を待つ

param(
    [Parameter(Mandatory=$true)]
    [string]$Url
)

. .\cdp_helper.ps1

$debugPort = 9222
$tabs = Invoke-RestMethod -Uri "http://localhost:$debugPort/json"
$wsUrl = $tabs[0].webSocketDebuggerUrl
$ws = Connect-CDP -WsUrl $wsUrl

# Page.enable でページイベントの監視を有効化
Send-CDPCommand -WebSocket $ws -Method "Page.enable" -Id 1

# ページ遷移
Send-CDPCommand -WebSocket $ws -Method "Page.navigate" -Params @{
    url = $Url
} -Id 2

# 読み込み完了を待つ（簡易実装）
Start-Sleep -Seconds 3

# スクリーンショット取得
# Base64エンコードされたPNG画像が返ってくる
$screenshot = Send-CDPCommand -WebSocket $ws -Method "Page.captureScreenshot" -Params @{
    format = "png"
    quality = 80
} -Id 3

# ファイルに保存
$bytes = [Convert]::FromBase64String($screenshot.result.data)
[IO.File]::WriteAllBytes("C:\tmp\screenshot.png", $bytes)

$ws.Dispose()
Write-Host "Screenshot saved to C:\tmp\screenshot.png"
```

スクリーンショットはClaude Codeのマルチモーダル機能と組み合わせると強力で、画像から次にクリックすべき要素を判断させることができる。

### Step 5：要素のクリックとテキスト入力

```powershell
# cdp_interact.ps1
# DOM要素をCSSセレクタで検索し、クリックまたはテキスト入力を行う

param(
    [string]$Selector,
    [string]$Action = "click",  # "click" or "type"
    [string]$Text = ""
)

. .\cdp_helper.ps1

$debugPort = 9222
$tabs = Invoke-RestMethod -Uri "http://localhost:$debugPort/json"
$ws = Connect-CDP -WsUrl $tabs[0].webSocketDebuggerUrl

# DOMを有効化
Send-CDPCommand -WebSocket $ws -Method "DOM.enable" -Id 1
Send-CDPCommand -WebSocket $ws -Method "Runtime.enable" -Id 2

if ($Action -eq "click") {
    # JavaScript経由でクリック
    # querySelectorで要素を見つけてclick()を呼ぶのが最もシンプル
    $js = "document.querySelector('$Selector').click()"
    Send-CDPCommand -WebSocket $ws -Method "Runtime.evaluate" -Params @{
        expression = $js
    } -Id 3
    Write-Host "Clicked: $Selector"

} elseif ($Action -eq "type") {
    # フォーカスしてからテキスト入力
    # Input.insertText はクリップボード経由ではなく直接入力するため安定する
    $focusJs = "document.querySelector('$Selector').focus()"
    Send-CDPCommand -WebSocket $ws -Method "Runtime.evaluate" -Params @{
        expression = $focusJs
    } -Id 3

    Send-CDPCommand -WebSocket $ws -Method "Input.insertText" -Params @{
        text = $Text
    } -Id 4
    Write-Host "Typed into: $Selector"
}

$ws.Dispose()
```

### Step 6：WSL2側のラッパー

上記のPowerShellスクリプトをBashラッパーで包み、`chrome-bridge.sh navigate URL` のようにシンプルに呼べるようにする。`powershell.exe -ExecutionPolicy Bypass -File ...` をcase文で分岐させるだけの構造だ。

## 応用例

この仕組みを使えば、Discord（Web版）への自動投稿、Redditへの記事投稿、itch.ioのゲームページ管理など、ブラウザ操作が必要なあらゆる作業を自動化できる。

例えばDiscordへの投稿は以下のようになる。

```bash
# チャンネルに遷移
./chrome-bridge.sh navigate "https://discord.com/channels/SERVER_ID/CHANNEL_ID"
sleep 5

# メッセージ入力欄にテキストを入力してEnterで送信
./chrome-bridge.sh type "[data-slate-editor='true']" "投稿テスト"
```

## 注意点

1. **WebSocketバッファ** — 4Kスクリーンショットではバッファ不足になる。複数回Readの結合が必須
2. **ページ読み込み待機** — 固定Sleep は不安定。`Page.loadEventFired` を待つべき
3. **SPA対応** — DOM要素が遅延生成されるため、ポーリングで出現を待つ
4. **認証** — `--user-data-dir` で専用プロファイルを分離し手動ログインしておく

## まとめ

WSL2からのWindows Chrome操作は、CDP + PowerShellで実現できる。Claude Codeと組み合わせれば、「スクリーンショット→AI判断→操作実行」の知的なブラウザ自動操作が可能になる。
