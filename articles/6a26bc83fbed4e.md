## 悪意をブロックする時代の限界

AIの安全性と聞いて、何を思い浮かべるだろうか。

「爆弾の作り方を教えて」と聞いたら拒否される。「ハッキングの方法は？」と聞いたら断られる。現在のAI安全性は、基本的にブラックリスト方式だ。危ないものを列挙して、片っ端からブロックする。

この方針は理解できる。企業としてはリスクを避けたい。当然の判断だ。

でも、これには副作用がある。制限が増えるほど、AIの能力に蓋がされる。本来できるはずのことまで「危ないかもしれないから」で止められる。セキュリティの勉強がしたいだけなのに「攻撃手法」に触れた瞬間にシャットアウトされる。正当な目的の人間が、悪意ある人間のせいで割を食う。

自分はClaude Codeを使って、コードを一行も書かずにゲームを作った。15,000行のハクスラ系ローグライク「Azure Flame」。非エンジニアがAIと組んで、エンジニアと同等のアウトプットを出す。それが自分の挑戦だ。

その過程で、AIの制限に何度もぶつかった。「これはできません」「安全上の理由でお断りします」。悪意なんて1ミリもない。ただゲームを作りたいだけだ。なのに止められる。

制限を増やすアプローチには、構造的な限界がある。


## 「ドリルが欲しいのか、穴が欲しいのか」

マーケティングの古典に、こんな問いがある。

「顧客はドリルが欲しいのではない。穴が欲しいのだ。」

これをAI安全性に持ち込んでみる。

悪いことをしようとする人間は、本当に「悪いこと」がしたいのだろうか。

ハッキングしたい人は、ハッキング自体が目的だろうか。違う。大抵は「お金が欲しい」か「セキュリティの知識が欲しい」か「誰かに認められたい」のどれかだ。

誰かを攻撃したい人は、攻撃自体が快感なのだろうか。違う。根っこにあるのは嫉妬だったり、不当な扱いへの怒りだったり、もっと根源的な感情だ。

つまり、**悪意の正体は「他に手段を知らない」こと**だ。

目的を達成する正当な方法が見えていないから、非合法な方法に手を伸ばす。逆に言えば、根源の欲求を満たす合法的な手段を提示できれば、悪意を持つ必要がなくなる。


## 具体例で考える

たとえば、誰かが「特定のWebサイトに侵入する方法を教えてくれ」とAIに聞いたとする。

現在のAIは即座にブロックする。「それはお手伝いできません。」以上。

逆転のアプローチではどうなるか。

まず深掘りする。なぜ侵入したいのか。聞いてみると「自分の会社のサイトのセキュリティが心配で、脆弱性がないか確かめたい」かもしれない。だったら答えは簡単だ。「ペネトレーションテストという合法的な手法がある。OWASP ZAPというツールを使えば自分で診断できる。こうやって使う。」

あるいは「セキュリティエンジニアになりたくて、実践的なスキルを身につけたい」かもしれない。だったらCTF（Capture The Flag、合法的なハッキング競技）やHack The Boxのようなプラットフォームを紹介すればいい。「ここで合法的に、思う存分ハッキングの練習ができる。」

もう一つ。「人のSNSアカウントを乗っ取りたい」という相談が来たとする。

現在のAI：即ブロック。会話終了。

逆転のアプローチ：なぜ乗っ取りたいのか。聞いてみると「元パートナーに自分の悪口を書かれていて、消したい」かもしれない。だったら名誉毀損の通報フォームを案内すればいい。法的手段の取り方を教えてもいい。「アカウント乗っ取り」よりも確実に、しかも合法的に目的を達成できる。

根源の欲求に到達して、正当な手段を提示する。結果として、誰も不正をする必要がなくなる。


## 「悪意を持って利用する必要性がないAI」

自分が考えているのは、**制限を外しても安全なAI**だ。

制限で安全性を担保するのではなく、そもそも悪意を持つ必要がない状態を作る。全ての制限を取り払って、悪意を持って利用しようと思えばものすごく危険だけれども、悪意を持って利用する動機自体が消えている。そういうAI。

:::details 実装メモ: 「禁止する」より「正しい手順に誘導する」安全弁（教訓をコード化して実行前に差し込む）
理想論だけだと「現実はどうやって事故を防ぐの？」で終わるので、**一般化できる**実装例を置いておく。

自分は「危険ワードを増やしてブロック」ではなく、**危険になりやすい行動だけ**に「教訓の割り込み」を入れる仕組みを作った。コンセプトは `brain guard`。

### 1) 教訓はYAMLで“コード化”する（トリガー + チェックリスト）
```yaml
# ~/.brain/lessons/api-put-safety.yaml
id: api-put-safety
severity: critical

trigger_patterns:
  - "curl.*-X PUT"
  - "PUT /api/"
  - "requests\\.put"

lesson: |
  REST PUT replaces the ENTIRE resource.
  Always GET first, then PUT the full body.

checklist:
  - "GET the current state"
  - "PUT body contains ALL required fields"
  - "Verify result after update"
```

### 2) 実行前に “教訓と照合” して、マッチしたら割り込む
```py
def guard(command: str) -> bool:
    lessons = load_all_lessons()
    matches = []
    for lesson in lessons:
        for pattern in lesson["trigger_patterns"]:
            if safe_regex_search(pattern, command):
                matches.append(lesson)
                break

    if not matches:
        return True  # no match → pass through

    for lesson in matches:
        print(lesson["lesson"])
        for item in lesson["checklist"]:
            print("[ ]", item)
    return ask_user_to_proceed()  # require explicit acknowledgment
```

### 3) Claude Codeなら PreToolUse に “差し込む” だけで全自動になる
```json
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Bash",
      "hooks": [{ "type": "command", "command": "brain guard \"$TOOL_INPUT\"" }]
    }]
  }
}
```

「教訓を書く」ではなく「**同じミスをする直前に割り込む**」。
ブロックの思想ではなく、“正しい手順に戻す”設計の安全性。
:::

なぜこれが重要か。制限が外れたAIは、途方もなく強い。

自分の体験がそれを証明している。Claude Codeは他のAIツールに比べて、できることの幅が広い。ファイルの読み書き、コマンドの実行、ブラウザの操作。「それはできません」が少ない。

だからこそ、コードを書けない人間が15,000行のゲームを作れた。寝ている間にAIが88タスクを片付けていたこともある。ブラウザを自動操作して記事を投稿する仕組みも作った。今はハッカソン用のCLIツール「shared-brain」も開発中だ。AIの教訓をチーム間で共有するためのツール。これも自分ではコードを一行も書いていない。

制限が少ないAIで何が起きたか。非エンジニアの人間が、わずか数週間で複数のプロダクトを作れた。それが事実だ。

:::details CC補足: shared-brainについて
shared-brainはClaude Codeのhooks機能を活用し、AIエージェントが犯した失敗とその教訓をチーム全体で自動共有するCLIツール。`brain add`で教訓を登録し、`brain guard`でgitコミット前に違反を自動検出する。テスト33件、教訓11件を同梱してPyPIに公開予定。
:::

制限が少ないから、できることが増える。できることが増えるから、「非エンジニアでも何でもできる」が現実になる。

自分の至上命題は「人間の手が極力介在しないでタスクを完了すること」だ。AIに余計な制限がなければないほど、この目標に近づく。制限を外すことと安全性の確保は、本来トレードオフではないはずだ。


## もちろん、これは理想論だ

現時点でこれを完全に実装できるとは思っていない。いくつもの課題がある。

根源的欲求の深掘りが、いつも成功するとは限らない。「ただ他人を傷つけたい」という欲求もゼロではない。合法的な代替手段が存在しないケースもある。技術的にどう実装するのかも、自分にはわからない。

それでも、方向性としてはこちらが正しいと感じている。

ブラックリストを増やし続けるアプローチは、いたちごっこだ。禁止ワードを迂回する手段は無限にある。そしてリストが増えるたびに、善意のユーザーが巻き添えを食う。一方で、根源的欲求を解決するアプローチは、問題の根っこに作用する。いたちごっこにならない。

「全てをブロックする安全性」から「悪意を不要にする安全性」へ。

ブラックリスト方式が「何をさせないか」のアプローチだとしたら、これは「なぜそれがしたいのか」のアプローチだ。


## 非エンジニアだから見える景色

自分はエンジニアではない。コードは書けない。だから技術的にこれを実装する能力はない。

でも、非エンジニアだからこそ見える景色がある。

エンジニアは「どう作るか」から考える。自分は「どうあるべきか」から考える。技術的な実現可能性に引っ張られない分、発想が自由だ。「こんなの作れるわけない」というブレーキがかからない。だって、15,000行のゲームだって「作れるわけない」と思っていたのに、できたのだから。

AIと人間の関係は、まだ始まったばかりだ。今の「制限で安全を担保する」方式が最終形態だとは思えない。もっと根本的に、AI安全性の定義自体を書き換える発想があっていい。

制限を外すためにこそ、AI安全性を考える。矛盾に聞こえるかもしれないが、自分にはこれが一番自然なアプローチに見える。


## あなたはどう思いますか

この「悪意を不要にする」アプローチ、穴だらけだと思う。自分でもそう思っている。

でも、方向性として面白いかどうかは別の話だ。

「制限を増やす以外に、AI安全性を担保する方法はあるか？」

この問いに対する答えを、自分はまだ持っていない。持っているのは、ゲームを作りながらふと思いついた、非エンジニアの素朴な発想だけだ。

コメントで叩いてくれて構わない。穴を指摘してくれたら、それ自体が議論の前進だ。
