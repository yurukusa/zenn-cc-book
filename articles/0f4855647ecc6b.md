## hookが動かない。原因は8パターンに絞れる

設定したはずなのに動かない。エラーも出ない。何が起きているのかわからない。

700時間以上の自律稼働で遭遇した全パターンを診断順に並べる。上から順に試せば、大半の問題は10分以内に解決する。

## まず`--doctor`を実行する

原因を推測する前に、自動診断を走らせる。

```bash
npx cc-safe-setup --doctor
```

13項目を自動チェックして、問題のある箇所を特定する。

```
✓ jq installed (jq-1.7)
✓ settings.json exists
✓ settings.json is valid JSON
✓ "hooks" section exists in settings.json
✓ PreToolUse: 3 hook(s) registered
✗ Hook script not found: ~/.claude/hooks/my-hook.sh
✓ PostToolUse: 1 hook(s) registered
✓ hooks directory exists (16 scripts)
✓ Claude Code: 2.1.84 (Claude Code)
```

`✗`が出た行が原因。以下は`--doctor`で検出できるものと、できないものを含めた全パターン。

## 原因1: jqがインストールされていない

hookはJSON形式で入力を受け取る。`jq`がないとパースできず、何も起きない。

```bash
jq --version                  # 確認
sudo apt install jq           # Ubuntu/Debian/WSL
brew install jq               # macOS
```

`--doctor`で検出される。

## 原因2: 実行権限がない

実行権限がないと、エラーなく静かに失敗する。

```bash
ls -la ~/.claude/hooks/          # 確認
chmod +x ~/.claude/hooks/*.sh    # 修正
```

**WSLユーザーは特に注意**。Windows側からファイルをコピーした場合、実行権限が落ちていることがある。WSL内で作成し直すか、`chmod +x`を明示的に実行する。

## 原因3: matcherが合っていない

`matcher`がツール名と一致していない。

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [{"type": "command", "command": "bash ~/.claude/hooks/guard.sh"}]
      }
    ]
  }
}
```

`matcher`の仕様:
- `"Bash"` — Bashツールだけにマッチ
- `"Write"` — Writeツールだけにマッチ
- `""` — **全ツール**にマッチ（空文字列）

**大文字小文字は区別される**。`"bash"`ではマッチしない。`"Bash"`と書く。

よくある間違い:
- `"bash"` → `"Bash"`
- `"write"` → `"Write"`
- `"edit"` → `"Edit"`
- `"*"` → `""`（ワイルドカードではなく空文字列）

## 原因4: settings.jsonの場所が違う

設定ファイルは2か所ある。

| ファイル | 適用範囲 |
|---|---|
| `~/.claude/settings.json` | 全プロジェクト共通 |
| `プロジェクト/.claude/settings.json` | そのプロジェクトのみ |

ローカル設定がグローバル設定を上書きしている可能性がある。

```bash
# グローバル設定の確認
cat ~/.claude/settings.json | jq '.hooks'

# プロジェクトローカル設定の確認
cat .claude/settings.json | jq '.hooks' 2>/dev/null
```

両方にhookセクションがある場合、ローカルが優先される。

## 原因5: 構文エラーで全ツールがロックされる

**これが最も危険なパターン**。

hookスクリプトにbash構文エラーがあると、スクリプトはexit code 2を返す。Claude Codeのhook仕様では:

| exit code | 意味 |
|---|---|
| 0 | 許可（hookチェックOK） |
| 2 | **ブロック**（ツール実行を拒否） |
| それ以外 | hookエラーとして無視 |

構文エラー → bashがexit 2で終了 → Claude Codeは「hookがブロックした」と解釈。

`matcher: ""`（全ツールマッチ）の設定でこれが起きると、**全ツールの実行がブロック**される。ファイルの読み書き、bashコマンド、すべてが止まる。

自律稼働中に構文エラーを混入し、3時間のロックアウトを経験した。hookを修正するためのファイル編集すらブロックされる。

**予防策**:

```bash
# hookの構文チェック
npx cc-safe-setup --validate
```

```
✓ destructive-guard
✓ branch-guard
✓ secret-guard
✗ my-custom-hook — syntax error near line 12
```

`--validate`はすべてのhookスクリプトに対してbashの構文チェックを実行する。CIに組み込むか、hook編集後に必ず実行する。

## 原因6: 環境変数PATHが通っていない

hookスクリプト内で外部コマンドを呼んでいる場合、PATHが通っていないと失敗する。

Claude Codeのhookは最小限の環境で実行される。`~/.bashrc`や`~/.zshrc`で設定したPATHは反映されない場合がある。

```bash
#!/bin/bash
export PATH="/usr/local/bin:/usr/bin:/bin:$PATH"

# NG: コマンド名だけ
result=$(my-custom-tool check)
# OK: フルパス指定
result=$(/usr/local/bin/my-custom-tool check)
```

## 原因7: hookの入力JSON形式が違う

hookイベントによって、入力として渡されるJSONの構造が異なる。

**PreToolUse**:
```json
{
  "tool_name": "Bash",
  "tool_input": {
    "command": "rm -rf /tmp/test"
  }
}
```

**PostToolUse**:
```json
{
  "tool_name": "Bash",
  "tool_input": {
    "command": "rm -rf /tmp/test"
  },
  "tool_result": "..."
}
```

`PreToolUse`用に書いたスクリプトを`PostToolUse`にも使い回す場合、`tool_result`フィールドの有無で挙動が変わることがある。

デバッグには入力JSONをファイルに書き出すのが確実。

```bash
#!/bin/bash
INPUT=$(cat)
echo "$INPUT" >> /tmp/hook-debug.json  # 入力を記録
echo "$INPUT" | jq -r '.tool_input.command'
```

## 原因8: Claude Codeのバージョンが古い

hooks機能はClaude Code v2.1以降で利用可能。古いバージョンでは`settings.json`にhookを書いても無視される。

```bash
claude --version
```

v2.1未満の場合はアップデートする。

```bash
npm update -g @anthropic-ai/claude-code
```

## 緊急復旧: `--safe-mode`

hookの問題で作業が完全にブロックされた場合、全hookを一括無効化できる。

```bash
npx cc-safe-setup --safe-mode      # 全hook無効化
# → Safe mode ON. 6 hooks disabled.
# Claude Codeを再起動

npx cc-safe-setup --safe-mode off  # 復旧
```

原因5のロックアウト事故では、この`--safe-mode`で復旧した。hookを使うなら覚えておく。

## 予防: `--validate`を習慣にする

hookを編集したら、必ず構文チェックを実行する。

```bash
npx cc-safe-setup --validate
```

全hookの構文チェックと危険なexit codeパターンの検出を行う。hookを変更するたびに実行する習慣をつける。自律稼働中にhookが壊れると、人間が気づくまで数時間ロストする。

## 診断フローチャート

1. `npx cc-safe-setup --doctor` を実行 → 問題が見つかればそれを修正
2. `--doctor`で問題なし → matcherの大文字小文字を確認
3. matcherも正しい → settings.jsonの場所（グローバル vs ローカル）を確認
4. 設定場所も正しい → hookスクリプト内でデバッグログを出力して入力JSONを確認
5. 全ツールがブロックされている → `--safe-mode`で復旧してから`--validate`で構文チェック
