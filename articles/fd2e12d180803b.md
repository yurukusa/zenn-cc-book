## 全部「Agent」と呼んでいた

「4ステップの定義済みプロセスでタスクを強化する方法は？」

Academyのクイズでこう問われた。「Agent」と答えた。不正解。**Workflow**だった。

自分は全部「Agent」と呼んでいた。プロンプトチェーン？ Agent。ルーティングロジック？ Agent。並列処理パイプライン？ Agent。

全部Workflowだった。

## AnthropicのWorkflowとAgentの定義

Anthropicは明確な線を引いている。

**Workflow**: 事前定義されたステップを順序通りに実行する。制御フローが固定。**コード**が「次に何をするか」を決める。

**Agent**: 自律的にツールを使い、判断しながらタスクを遂行する。制御フローが動的。**LLM**が「次に何をするか」を決める。

区別の基準はシンプルだ。「次のステップを決めるのは誰か」。コードが決めるならWorkflow。LLMが決めるならAgent。

Claude Code（の自律モード）がAgentの典型例だ。「コードを読む→テスト実行が必要だと判断→テスト結果を解釈→バグを修正」。各ステップでLLMが判断している。事前に「テストを実行しろ」と指示されているわけではない。

:::details CC補足: Workflowの5パターン（Anthropic公式分類）
Anthropicの公式ドキュメントでは、Workflowを5つのパターンに分類している:

**1. Prompt Chaining（直列）**
ステップAの出力をステップBの入力にする。翻訳→校正→フォーマットのように直列に繋ぐ。

```
入力 → [ステップA] → [ステップB] → [ステップC] → 出力
```

**2. Routing（分岐）**
入力を分類し、種類に応じて異なるパスに振り分ける。顧客問い合わせを「技術」「請求」「一般」に分類して担当パスに流す。

```
入力 → [分類] → 技術 → [技術対応]
              → 請求 → [請求対応]
              → 一般 → [一般対応]
```

**3. Parallelization（並列）**
同じ入力に対して複数のタスクを並列実行し、結果をまとめる。1つの文書を「誤字チェック」「事実確認」「トーン確認」で同時に検査する。

**4. Orchestrator-Workers（委任）**
メインのLLMがタスクを分析し、サブタスクをworkerに委任する。ただし**委任するタスクの種類は事前に定義されている**。

**5. Evaluator-Optimizer（改善ループ）**
生成→評価→改善のループを回す。Evaluator-optimizerの実例: 画像→Claudeで3Dモデル説明→CADqueryでモデル生成→レンダリング→Claudeが元画像と比較→不十分なら再生成。**全ステップが事前に設計されている**からWorkflow。

これら5つは全てWorkflowだ。制御フローをコードが決めているから。
:::

## なぜ区別が重要なのか

「呼び方の問題では？」と思うかもしれない。違う。設計判断に直結する。

**Workflowの強み**:
- 各ステップの精度が高い。大きなタスクを小さなタスクに分割するから
- テストが容易。各ステップを個別にテストできる
- 結果が予測可能。同じ入力には同じ出力が返る
- デバッグが簡単。どのステップで問題が起きたかすぐわかる

**Agentの弱点**:
- タスク完了率がWorkflowより低い。判断を間違える可能性がある
- テスト・評価が困難。同じ入力でも異なるパスを通る可能性がある
- デバッグが難しい。「なぜその判断をしたか」の追跡が複雑

Academyの動画では明確にこう言っていた。**「常にWorkflowで実装可能かを先に検討し、Agentは本当に必要な場合のみ使う」**。

:::details CC補足: Workflowを先に検討すべき技術的理由
LLMの判断には確率的な不確実性がある。temperature 0でも、内部の浮動小数点演算の順序によって結果が異なることがある。

Workflowでは制御フローがコードで固定されているため、LLMの不確実性は「各ステップ内の出力品質」にのみ影響する。ステップ間の遷移は確定的だ。

Agentでは制御フロー自体がLLMの判断に依存するため、不確実性が「何をするか」のレベルに波及する。ステップAの結果に基づいてステップBかステップCか判断するが、その判断自体が確率的。

実用上の影響:
- Workflowは「出力の品質」を改善すればよい → プロンプト改善、Few-shot Examples追加
- Agentは「判断の品質」も改善しなければならない → ツールの説明改善、判断ロジックのガイドライン、フォールバック設計

Agentが適切なケース:
- タスクの手順が事前に定義できない（「このバグを調査して直せ」のような開放的なタスク）
- 環境の状態によって次のアクションが変わる
- 人間が同じタスクをやる場合にも「次に何をするか」を都度判断する

Workflowが適切なケース:
- タスクの手順が決まっている
- 入力のカテゴリに応じて処理が分岐する
- 品質を安定させたい
- テスト・監視を体系的に行いたい
:::

## Temperatureのデフォルトは1.0

もう一つ。Temperatureのデフォルト値を聞かれて「0.5くらいだろう」と答えた。バランスの取れた中間値だろうと。

**デフォルトは1.0。** 最大の創造性。

Temperature（温度）とは、出力のランダム性を制御するパラメータだ。0.0から1.0の範囲で設定する。

- **0**: 最高確率のトークン（単語の断片）を常に選択する。同じ入力には同じ出力。決定的
- **1.0**: 元の確率分布をそのまま使う。低確率のトークンも選ばれうる。創造的

Claudeのデフォルトが1.0なのは意図的だ。Claudeは創造的な会話を想定して設計されている。

:::details CC補足: Temperatureの内部動作
Temperatureはトークン選択確率の分布を制御するパラメータだ。

LLMがテキストを生成する仕組み: 次のトークン候補それぞれに確率が割り当てられる。「猫は」の次に「かわいい」30%、「動物」25%、「好き」20%...のように。

- **Temperature 0**: 確率最大のトークンを常に選ぶ。「かわいい」が常に選ばれる。出力が完全に決定的になる
- **Temperature 0.5**: 確率分布がシャープになる。高確率のトークンがさらに有利になり、低確率のトークンが選ばれにくくなる
- **Temperature 1.0**: 元の確率分布をそのまま使う。「動物」や「好き」も本来の確率で選ばれる

技術的にはsoftmax関数の温度パラメータ。softmax(logits / T)のTがTemperature。Tが小さいほど分布が尖り、大きいほど分布が平坦になる。

注意: Temperatureを上げても**常に**劇的に異なる出力になるわけではない。あるトークンの確率が圧倒的に高い場合（99%など）、Temperature 1.0でもほぼ常にそのトークンが選ばれる。分布が変わるだけで、確率が高いものは依然として高い。
:::

## 分類タスクをTemperature 1.0で回していた

デフォルト1.0の何が問題か。

事実抽出、分類、構造化データ処理。これらのタスクでは、一貫した結果が必要だ。同じ入力には同じ出力が返ってほしい。

自分は分類プロンプトをTemperatureの指定なし（=デフォルトの1.0）で何ヶ月も回していた。結果が時々ブレる。「なんでだろう」と思いつつ放置していた。原因はTemperatureだった。

```python
# 分類タスクの正しい設定
response = client.messages.create(
    model="claude-sonnet-4-20250514",
    max_tokens=100,
    temperature=0,  # ← これを明示する
    messages=[{
        "role": "user",
        "content": "以下のテキストを「技術」「ビジネス」「一般」に分類せよ: ..."
    }]
)
```

用途別の推奨値:
- **データ抽出、分類、事実回答** → `temperature: 0`
- **一般的な会話** → `temperature: 0.5`前後
- **ブレスト、創作、マーケティングコピー** → `temperature: 0.7〜1.0`

## Extended Thinkingとの関係

1つ注意点がある。Extended Thinking（拡張思考、第2回で扱った）を有効にすると、**Temperatureは1.0に固定される。変更不可。**

思考プロセスには確率分布全体が必要なため、deterministic（決定的）な出力は設計上サポートされていない。

つまりExtended Thinkingと低Temperature（分類タスク向け）は併用できない。分類タスクの精度を上げたい場合、Extended Thinkingではなくプロンプト改善やFew-shot Examplesで対応する必要がある。

:::details CC補足: なぜExtended ThinkingはTemperature 1.0固定なのか
Extended Thinkingでは、モデルが推論プロセスを展開する際に、多様な思考パスを探索する必要がある。

Temperature 0だと、常に最も確率の高いトークンが選ばれるため、思考が単一のパスに固定される。「別の角度から考えると...」「いや、この仮定は間違っているかもしれない...」のような思考の分岐が起きにくくなる。

Temperature 1.0では確率分布全体が活きるため、モデルは複数の仮説を検討し、自己修正しながら推論を進められる。これがExtended Thinkingの品質を支えている。

結果として:
- 分類タスク + Temperature 0 → Extended Thinkingなしで実装
- 複雑な推論タスク + Extended Thinking → Temperature 1.0固定（変更不可）
- この2つは設計上、両立しない
:::

## WorkflowとTemperatureを組み合わせる

Workflowパターン（第5パターンの定義を思い出してほしい）とTemperatureの知識を組み合わせると、より精密な設計ができる。

たとえばPrompt Chaining（直列パイプライン）で:
1. ステップ1: 入力テキストを分類する → `temperature: 0`
2. ステップ2: 分類結果に基づいて要約を生成する → `temperature: 0.3`
3. ステップ3: 要約をマーケティングコピーに書き換える → `temperature: 0.8`

各ステップの目的に応じてTemperatureを変える。Agentだとこの制御は難しい（LLMが自律的に次のアクションを決めるため、ステップごとのTemperature切り替えをコードで制御できない）。

これもWorkflowが適している理由の一つだ。

## まとめ

Workflow vs AgentとTemperatureで知らなかったこと:
- 「次のステップを決めるのは誰か」が区別の基準。コード→Workflow、LLM→Agent
- Workflowは5パターン（直列、分岐、並列、委任、改善ループ）
- **まずWorkflowで実装可能かを検討し、Agentは最終手段**
- Temperatureのデフォルトは1.0。分類タスクには明示的に0を設定する
- Extended Thinking有効時はTemperature 1.0固定。変更不可
