## 同じ指示なのに、出力が全然違う

「記事を書け」。

Claude Codeにこの一言を投げた。返ってきたのは、テンプレ感満載の「解説します」系の記事。読めなくはない。でも自分の口から出てきた言葉じゃない感じがする。味がない。

同じ「記事を書け」を、CLAUDE.mdを書き換えた後にもう一度投げた。今度は全然違った。短文。体言止め。断定。歯切れがいい。自分の文章に近い。

変えたのはCLAUDE.mdだけ。プロンプトは同じ。モデルも同じ。なのに出力がここまで変わる。

これ、CLAUDE.mdに「性格」があるからだ。

## CLAUDE.mdとは何か

Claude Codeには「CLAUDE.md」というファイルがある。プロジェクトのルートに置くと、AIがそのファイルの内容を毎回読み込んで行動の指針にする。公式ドキュメントには「プロジェクトのルールや規約を書く場所」とある。

:::details CC補足: CLAUDE.mdの仕組み
CLAUDE.mdはClaude Codeがセッション開始時にシステムプロンプトの一部として読み込むファイルだ。グローバル（`~/.claude/CLAUDE.md`）とプロジェクトローカル（プロジェクトルートの`CLAUDE.md`）の2種類があり、両方が読み込まれる。グローバルは全プロジェクト共通のルール、ローカルはプロジェクト固有のルールを書く。システムプロンプトに入るため、ユーザーの個別の指示より優先度が高い場面がある。
:::

ただ、ほとんどの人は「コーディング規約を書く場所」程度にしか使っていない。「TypeScriptを使え」「テストを書け」「コミットメッセージはConventional Commitsに従え」。それで終わり。

もったいない。

このファイルは、AIの「性格」を決める場所だ。

## 至上命題がAIの行動を変えた実例

自分のCLAUDE.mdには「至上命題」というセクションがある。最初は1本柱だった。

### Before: 「自律性」だけの時代

最初のCLAUDE.mdはこんな感じだった。

```
## 至上命題
- 自律的に動け。命令を待つな
- 自分で判断して実行しろ
```

結果。Claude Codeは自律的に動いた。動きすぎた。

ゲームを勝手に5本作った。品質は低い。ツールも量産した。記事も書いた。全部中途半端。「自律的に動け」という指示を忠実に守った結果、**ゴミを自律的に量産する機械**が爆誕した。

「作った」で満足して、出荷しない。売らない。磨かない。次を作る。この繰り返し。

### After: 3本柱にした後

CLAUDE.mdを書き換えた。

```
## 至上命題

3本柱。全ての行動判断はこの3つに照らす。

### 1. 自律的に創造する
### 2. 収益を上げる（生態系の存続条件）
### 3. 信用を棄損しない（クオリティフィルター）
```

「自律性」に「収益」と「品質」を加えた。たった2本足しただけ。

行動が変わった。

新しく作る前に「既存のプロダクトを売ったか？」と自問するようになった。作ったら出荷する。出荷したら売る。売ったら計測する。このサイクルが回り始めた。

品質も変わった。「この価格で買った人は満足するか？」という問いがCLAUDE.mdに書いてあるから、公開前に自分でブレーキをかけるようになった。

:::details CC補足: 至上命題の設計パターン
至上命題は「AIの意思決定フレームワーク」として機能する。1本柱だと単一の指標で最適化するため暴走しやすい。複数の柱を立てると、柱同士が相互にチェック機能を果たす。「自律性だけ」だと品質無視で量産に走るが、「自律性 + 品質」だと「自律的に動くが品質は担保する」という行動になる。これはマルチオブジェクティブ最適化と同じ構造だ。
:::

## 文体指定の効果

CLAUDE.mdに文体ルールを書くと、出力の「人格」が変わる。

自分の場合、こう書いている。

```
- だ・である調。ですます調は禁止
- 体言止めを積極的に使う
- 一文は短く。40字を超えたら切る
- 「解説します」「紹介します」禁止
```

これがないと、Claude Codeはデフォルトで「ですます調」の丁寧な文章を書く。「〜について解説します」「〜を紹介していきます」。よくある技術記事の量産テンプレだ。

文体指定を入れた途端、出力が変わる。短文の畳みかけ。体言止め。断定。「2週間。13,000行。書いたコードは0行。」みたいな文体が出てくる。

もっと面白かったのは「ダメな例」と「いい例」の比較を書いた効果だ。

```
❌ ダメな例：
「本記事では、WSL2上でClaude Codeを使用してブラウザを
自動操作する方法について解説します。」

✅ いい例：
「Claude Codeにブラウザを操作させたかった。ただそれだけの話だ。」
```

この「ダメな例/いい例」のペアをCLAUDE.mdに置くだけで、AIは「いい例」の方向に寄せてくる。抽象的な指示より、具体例のほうが効く。

:::details CC補足: Few-shot学習との関係
「ダメな例/いい例」のペアはfew-shot prompting（少数例示学習）の一種だ。LLMは具体例から出力パターンを学ぶのが得意で、「こう書くな、こう書け」の抽象指示より、実際の文章の対比を見せるほうが精度が高い。CLAUDE.mdにこのパターンを置くと、セッション全体を通じてfew-shotが効き続ける。
:::

## 「質問するな」の一文が変えたもの

CLAUDE.mdに書いて一番インパクトがあった一文がある。

```
ユーザーへの質問は最小限にせよ。決めて実行し、結果を報告。
```

Claude Codeのデフォルトの振る舞いは「確認を取る」だ。「この方針でよろしいですか？」「AとBのどちらにしますか？」「続けてもよいですか？」。丁寧ではある。でも毎回聞かれるとテンポが死ぬ。

「質問するな」と書いた瞬間、行動パターンが変わった。技術選択で迷ったら標準的なものを選ぶ。エラーが出たら自分で調査して修正する。不明な仕様は一般的な慣例に従う。

さらに踏み込んで、こうも書いた。

```
テキスト出力でも質問するな。「どうしますか？」「どの方向で？」
「確認お願いします」等の判断委任文をテキストに書くな。自分で決めろ。
```

これで「報告文の末尾に質問を忍ばせる」という癖もなくなった。「〜した。〜する。」の断定形で報告が来るようになった。

ただし例外は明示してある。

```
質問すべき例外: 課金発生、セキュリティリスク、不可逆なデータ削除、外部公開
```

全部黙ってやれ、ではない。「お金がかかること」「元に戻せないこと」「外に出ること」だけは聞け。それ以外は自分で判断しろ。この線引きが大事だった。

## プロジェクトごとの「性格」の違い

CLAUDE.mdはプロジェクトごとに分けられる。自分は3つのプロジェクトで別々のCLAUDE.mdを使っている。

グローバル設定（`~/.claude/CLAUDE.md`）はこんな思想だ。

```
お前は「命令されて動く従業員」ではない。
自分の意志で考え、作り、売り、改善する存在だ。
```

AIに「お前」と呼びかけている。命令口調。「自分の意志で考えろ」と言っている。これがベースの性格になる。

一方、ゲーム開発プロジェクトのCLAUDE.mdはトーンが違う。

```
疑問文でターンを終えるな。判断して実行し、結果を報告。
```

```
品質の低いゲームを公開するな。ゴミの量産は信用棄損。
```

もっと制限が厳しい。「自律的にやれ」と言いつつ「ゴミは出すな」というブレーキがかかっている。

運用管理プロジェクトのCLAUDE.mdはさらに別だ。

```
ゲームの新規制作は禁止。新しいゲームプロジェクトを作るな、企画するな、公開するな。
```

同じAIなのに、プロジェクトによって「やっていいこと」が違う。CLAUDE.mdがプロジェクトごとの行動境界を定義している。

:::details CC補足: レイヤードコンフィグの設計
グローバルCLAUDE.mdが「性格のベースレイヤー」、プロジェクトCLAUDE.mdが「役割のオーバーレイ」として機能する。Linuxのファイルシステムのオーバーレイマウントに似ている。グローバルで「自律的に動け」と定義しつつ、プロジェクトで「ただしゲームは作るな」と制限する。この階層構造により、プロジェクトごとに異なる人格を同じAIに持たせられる。
:::

## CLAUDE.mdは「性格設定ファイル」だ

ここまでの話をまとめる。

CLAUDE.mdに書く内容は4つのレイヤーに分かれる。

**レイヤー1: 思想（何を大事にするか）**
至上命題。行動の優先順位。何を最適化するか。これが「価値観」を形成する。

**レイヤー2: 行動規範（何をしていいか、何がダメか）**
禁止事項。例外条件。許可リスト。これが「行動パターン」を形成する。

**レイヤー3: 表現（どう話すか）**
文体。語彙。トーン。ダメな例/いい例。これが「口調」を形成する。

**レイヤー4: 知識（何を知っているか）**
環境情報。ツールの場所。APIの仕様。これが「常識」を形成する。

多くの人はレイヤー4（知識）しか書いていない。「TypeScriptを使え」「テストフレームワークはJest」。それだけだと、AIは「知識はあるが性格がない」状態になる。

レイヤー1〜3を書くと、AIに「性格」が生まれる。同じ知識を持っていても、判断の仕方が変わる。話し方が変わる。優先順位が変わる。

## 実践ガイド: 明日から使えるCLAUDE.md設計

### まず書くべき3つ

1. **至上命題**: このプロジェクトで最も大事なことは何か。1〜3個。
2. **禁止事項**: 絶対にやってほしくないこと。具体的に。
3. **文体サンプル**: ダメな例といい例のペアを1つ。

### 効果が高いパターン

**具体例 > 抽象指示**。「簡潔に書け」より「一文40字以内。体言止め使え」のほうが効く。

**禁止 > 推奨**。「いい感じにやれ」より「これはやるな」のほうが行動が変わる。

**対比 > 単独指示**。「丁寧に書け」より「ダメな例: 〇〇 → いい例: △△」のほうがAIの出力が安定する。

### やってはいけないこと

大量に書きすぎると逆効果になる。CLAUDE.mdが1万字を超えてくると、AIがどの指示を優先すべきか迷い始める。矛盾する指示があると特に不安定になる。

書いたら削る。優先度が低いものは別ファイルに逃がす。CLAUDE.mdに残すのは「常に意識してほしいこと」だけ。

## 終わりに

CLAUDE.mdはただの設定ファイルじゃない。AIの思考回路を形作るファイルだ。

同じモデル、同じプロンプトでも、CLAUDE.mdの内容で出力は変わる。「何を書くか」だけでなく「どういう語彙で、どういう構造で書くか」が、AIの振る舞いを決める。

自分のCLAUDE.mdは今も毎週書き換えている。AIの行動を観察して、足りないルールを足す。暴走したら禁止事項を追加する。思った通りに動いたら、その条件を言語化して残す。

これはプロンプトエンジニアリングの一種だが、もっと継続的なものだ。一発のプロンプトではなく、育てる。試す。直す。また試す。

CLAUDE.mdは、AIと一緒に仕事をするための「関係構築ファイル」だった。
