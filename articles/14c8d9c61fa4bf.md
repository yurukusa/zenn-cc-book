## はじめに

[前回の記事](https://zenn.dev/yurukusa/articles/385539904aaaad)で、AIを2体用意して相談させることで開発が止まらなくなった話を書いた。

あの仕組みが安定して動くためには、裏側にもう一つの仕組みがいる。**安全装置**だ。この記事では「どう実装するか」を書く。

具体的には3つの仕組みだ:

1. **コンテキスト監視** — AIのメモリ残量を監視し、枯渇前に自動退避
2. **エラー追跡（Stop-the-line）** — エラーを放置させない強制力
3. **教訓の永続化（Shared Brain）** — AIが覚えない問題をシステムで解決

全てClaude Codeの[Hooks機能](https://code.claude.com/docs/en/hooks)で動く。フックとは、AIがツールを使う前後に自動で実行されるスクリプトのことだ。

## 前提: CC-Codexペアセッション

[前回の記事](https://zenn.dev/yurukusa/articles/385539904aaaad)で紹介したCC-Codexペアセッションの上で、以下の3つの安全装置が動いている。ペアセッション自体の説明はそちらを参照してほしい。

全ての状態は**mission.md（運用台帳）**に集約される。セッションが死んでも、このファイルさえ残っていれば次のセッションが引き継げる。

## 仕組み1: コンテキスト監視（Context Monitor v2）

### なぜ必要か

AIには「コンテキストウィンドウ」というメモリ制限がある。Claude Codeの場合、約180,000トークン。会話が長くなるとメモリが埋まり、最終的にセッションが強制終了する。

実際に一度やらかした。コンテキスト残量3%まで気づかずに作業を続け、セッションが死んだ。作業状態の保存もできなかった。全ロスト。

### 実装

Claude Codeのデバッグログに、トークン使用量が記録されている:

```
[DEBUG] autocompact: tokens=68666 threshold=167000 effectiveWindow=180000
```

この `tokens`（使用中）と `effectiveWindow`（ウィンドウサイズ）から残量%を計算できる。

PostToolUseフック（ツール実行後に毎回走るスクリプト）で監視する:

```bash
# context-monitor.sh（抜粋）

# デバッグログから最新のトークン情報を取得
get_context_pct() {
    local debug_dir="$HOME/.claude/debug"
    local latest
    latest=$(find "$debug_dir" -maxdepth 1 -name '*.txt' \
             -printf '%T@ %p\n' | sort -rn | head -1 | cut -d' ' -f2)

    local line
    line=$(grep 'autocompact:' "$latest" | tail -1)

    local tokens window
    tokens=$(echo "$line" | sed 's/.*tokens=\([0-9]*\).*/\1/')
    window=$(echo "$line" | sed 's/.*effectiveWindow=\([0-9]*\).*/\1/')

    echo $(( (window - tokens) * 100 / window ))
}

CONTEXT_PCT=$(get_context_pct)
```

閾値に応じて段階的に警告を出す:

| 残量 | レベル | アクション |
|------|--------|-----------|
| 40%以下 | CAUTION | Codexとの会話を簡潔に |
| 25%以下 | WARNING | 新タスク開始禁止 |
| 20%以下 | CRITICAL | mission.mdに退避テンプレ自動追記 + `/compact`強制 |
| 15%以下 | EMERGENCY | 即座に`/clear`。作業禁止 |

CRITICAL以下では、mission.mdに退避テンプレートを自動追記する:

```markdown
## 自動退避テンプレート（CRITICAL - 2026-02-14 23:50 JST）
### 作業中タスク
- タスク名: [TODO]
- 触っているファイル: [TODO]
### 次にやること
- 次の1コマンド: [TODO]
### 待ち事項
- waiting_on: [相手/質問]
- resume_condition: [返答がAならX]
```

AIはこのテンプレートの`[TODO]`を埋めてからセッションを交代する。これで作業状態が失われない。

### 設定方法

`~/.claude/settings.json` のhooksセクションに追加:

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "/path/to/context-monitor.sh"
          }
        ]
      }
    ]
  }
}
```

## 仕組み2: エラー追跡（Stop-the-line）

### トヨタのアンドンをAI運用に

トヨタ生産方式に「アンドン」がある。生産ラインで問題が起きたら、作業者がヒモを引いてラインを止める。小さな問題を放置して大きな事故になるのを防ぐ仕組みだ。

AI運用でも同じことが起きる。「ERR_CODE:TIMEOUT」が出ても、AIは「次のタスクに進みます」と言って先に行ってしまう。小さなエラーが積み重なり、最終的にシステム全体が壊れる。

### err-tracker CLI

エラーを構造化データとして記録・追跡するCLIツールを作った:

```bash
# エラーを記録
err-tracker log --err-code "TIMEOUT" --script "codex-send" --owner "CC"

# 未解決エラーを一覧
err-tracker list --status open

# エラーを解決
err-tracker resolve --key "TIMEOUT_codex-send_2026-02-14" --note "リトライ回数を増やして解決"

# ブロックチェック（DoDに組み込む）
err-tracker check-block
# → 未解決P0あり: タスク完了不可
```

### PostToolUseフックでの自動検知

Bashツールの出力に`ERR_CODE:`が含まれていたら、自動でerr-trackerに登録するフックを入れている:

```bash
# err-code-detector.sh（PostToolUseフック）
# Bashの出力からERR_CODEパターンを検知
if echo "$TOOL_OUTPUT" | grep -q 'ERR_CODE:'; then
    ERR_CODE=$(echo "$TOOL_OUTPUT" | grep -o 'ERR_CODE:[A-Z_]*' | head -1)
    err-tracker log --err-code "$ERR_CODE" --script "$TOOL_NAME"
    echo "ERR_CODE検知: $ERR_CODE → err-trackerに自動登録"
fi
```

### DoDに組み込む

タスク完了の定義（DoD）に「`err-tracker check-block`が通ること」を必須化している。未解決エラーがある限り、タスクを「完了」と報告できない。

## 仕組み3: 教訓の永続化（Shared Brain）

### AIは覚えない

Claude Codeの新しいセッションが始まるたびに、AIの記憶はリセットされる。前回の事故で学んだ「Zenn APIはGETしてからPUTしろ」というルールは、次のセッションには引き継がれない。

CLAUDE.mdやMEMORY.mdに書けば引き継がれるが、**ファイルには行数制限がある**。全ての教訓を書き連ねるとすぐに溢れる。

### Shared Brain: 教訓を構造化データとして管理

[Shared Brain](https://github.com/yurukusa/shared-brain)は、教訓を`.brain/`ディレクトリに構造化YAMLとして保存するCLIツールだ。

```bash
# 教訓を追加（YAMLファイルで定義）
cat > lesson-api-safety.yaml << 'EOF'
severity: critical
text: "API更新は必ずGET→加工→PUT。GETなしPUT絶対禁止"
tags: [api, safety]
context: "2026-02-09 Zenn全記事消失事故"
EOF
brain write -f lesson-api-safety.yaml

# キーワードで教訓を検索
brain check "API"

# 統計
brain stats
# → Lessons: 21 (11 critical)
```

`.brain/lessons/`にYAMLファイルとして保存される:

```yaml
id: lesson-001
severity: critical
text: "API更新は必ずGET→加工→PUT。GETなしPUT絶対禁止"
tags: [api, safety]
context: "2026-02-09 Zenn全記事消失事故"
created: 2026-02-09T15:30:00+09:00
```

### Hooksとの連携

Claude CodeのPreToolUseフック（ツール実行前に走るスクリプト）で、教訓に違反する操作を検知できる:

```bash
# 例: PUT操作の前にGETが行われたかチェック
if echo "$TOOL_INPUT" | grep -q 'PUT.*api'; then
    if ! check_recent_get; then
        echo "GUARD: API更新にはGETが先に必要です（教訓: lesson-001）"
        exit 2  # ツール実行をブロック
    fi
fi
```

`exit 2`はClaude Codeにツール実行のブロックを指示する特別な終了コードだ。AIに「なぜブロックされたか」が伝わるので、自分で修正できる。

## どこまで自動化すべきか

3つの仕組みには「自動化の境界」がある:

| 仕組み | 自動化する部分 | 人間が判断する部分 |
|--------|--------------|------------------|
| コンテキスト監視 | 残量検知、退避テンプレ生成 | `/compact`や`/clear`の実行タイミング |
| エラー追跡 | ERR_CODE検知、自動登録 | 解決方法の判断、override |
| 教訓 | 違反検知、ブロック | 新しい教訓の追加 |

完全自動化が目標ではない。**「人間が判断すべきポイント」を最小化し、そこに集中させる**のが目標だ。

```
              ┌──────────────┐
              │  PostToolUse │
              │    フック     │
              └──┬───┬───┬──┘
                 │   │   │
    ┌────────────┘   │   └────────────┐
    ▼                ▼                ▼
┌────────┐    ┌────────────┐    ┌──────────┐
│Context │    │ err-tracker │    │  Shared  │
│Monitor │    │(Stop-line) │    │  Brain   │
│        │    │            │    │          │
│残量検知 │    │ERR_CODE検知│    │教訓検索   │
│退避生成 │    │ブロック判定 │    │違反検知   │
└───┬────┘    └─────┬──────┘    └────┬─────┘
    │               │               │
    ▼               ▼               ▼
┌────────────────────────────────────────┐
│           mission.md（運用台帳）         │
│    全ての状態が集約される唯一の場所      │
└────────────────────────────────────────┘
```

## まとめ

AIを「信頼するな」ではない。**「信頼を仕組みで裏付けろ」** だ。

- コンテキスト監視で、AIが自分の限界を認識できるようにする
- Stop-the-lineで、小さなエラーを放置させない
- Shared Brainで、AIの記憶喪失をシステムで補う

これらは全て、**非エンジニアの私がAIと組んで実運用した結果**生まれた仕組みだ。エンジニアリングの知識がなくても、「事故が起きた → 二度と起こさないようにする」というサイクルを回せば、仕組みは作れる。
