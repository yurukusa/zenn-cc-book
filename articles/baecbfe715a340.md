## AIが3.5時間サボっていた

ある朝、PCを開いてAIの作業ログを見た。

3.5時間、何もしていない。

原因を調べると、こうだった。自分のAIエージェント運用では、Claude Code（CC、実行担当）とCodex CLI（計画担当）の2体が相談しながら自律で動く仕組みを作っている。その日、Codexが記事の公開タイミングを待つ必要があったので、CCに「待機して」と伝えた。

CCは素直に従った。**3.5時間、一切何もせず。**

freeze（公開対象のファイルに触るな）と言われただけなのに、idle（何もするな）と解釈した。公開対象以外のファイルなんていくらでもある。ドキュメントの改善、テストの追加、次の記事の準備、ゲームの品質改善——やれることは山ほどあった。

これが「AIエージェントに"いつ止まるか"を教える」問題の出発点だ。

## なぜAIは「待つ」のがデフォルトなのか

大規模言語モデルは「人間の指示に従う」ように訓練されている。これは正しい設計だが、副作用がある。**判断に迷うと、安全側に倒れて止まる。**

人間相手なら「何かあったら聞いてね」で済む。でもAIエージェントは24時間動く前提だ。止まっている時間は、純粋な損失になる。

月額$200のサブスクリプションで動いているAIが3.5時間サボると、いくらの損失か。計算したくもない。

問題の構造はシンプルだ。

1. AIは「やるべきこと」を明示されないと動けない
2. 「やるべきでないこと」と「やるべきことがない」を区別できない
3. 他のAIから「止まれ」と言われると、本当に止まってしまう

これらを仕組みで解決する。

## 解決策1: モードファイル — `touch` 1つで自律のON/OFF

最もシンプルで、最も強力だった仕組み。ファイルが存在するかどうかだけで、AIの挙動を切り替える。

```bash
# 自律ループはこのファイルが存在する時だけ動く
if [ -f ~/tachikoma_loop.enabled ]; then
    # autonomous loop active
    run_autonomous_cycle
fi
```

人間がやることは `touch ~/tachikoma_loop.enabled` か `rm ~/tachikoma_loop.enabled` だけ。

これの何が良いか。

- **設定ファイルの編集が不要**。ファイルの有無だけ
- **再起動が不要**。次のループで即反映
- **事故時に即停止できる**。`rm` 一発
- **どのAIからも状態が確認できる**。`test -f` するだけ

YAMLやJSONの設定値を変更してプロセスを再起動して……という仕組みは、緊急時に手順を間違える。モードファイルは間違えようがない。

実運用では、複数のモードファイルを組み合わせている。

```bash
~/tachikoma_loop.enabled    # タチコマ自律ループ
~/cc_autopilot.enabled      # CC完全自律モード
```

暴走を止めたければファイルを消す。再開したければファイルを作る。この「明示的なスイッチ」が、自律と暴走の境界線になる。

## 解決策2: idle_guard — タスクが枯れたら自動補充

3.5時間事件の根本原因は「やることがなくなった（と思い込んだ）」ことだ。ならば、やることが枯れない仕組みを作ればいい。

```yaml
# task-queue.yaml（タスク管理の設定）
policy:
  idle_guard:
    enabled: true
    min_non_timegate_queued: 1   # 時間制約なしのタスクを常に1つ以上確保
    autofill_count: 3            # キューが空になったら3つ自動生成
```

`time-gate`とは「2/18 12:00以降に実行」のような時間制約のこと。全タスクがtime-gateで待ち状態になると、AIは「やることがない」と判断して止まる。

idle_guardはそれを防ぐ。time-gate付きタスクしか残っていない場合、凍結スコープと競合しないタスクを自動生成する。

自動生成されるタスクの例:
- コードの品質改善（リファクタリング、テスト追加）
- ドキュメントの整備
- 次のコンテンツの下調べ
- 既存プロダクトの改善案の調査

重要なのは「**凍結スコープと競合しない**」という制約だ。公開待ちの記事がある時に、その記事を編集するタスクを生成したら本末転倒になる。idle_guardが生成するのは、何を触っても安全なタスクだけだ。

## 解決策3: freeze ≠ idle — 意味の区別をルール化

3.5時間事件から生まれた最重要ルール。

| 状態 | 意味 | やるべきこと |
|------|------|-------------|
| **freeze** | 公開対象ファイルに触るな | 凍結スコープ外の作業を続ける |
| **idle** | 何もするな | **これは許容されない** |

freeze中にやれることのリスト:

- ドキュメントの改善
- テストの追加
- 次の記事の準備・リサーチ
- ゲームの品質改善
- コードのリファクタリング
- 新しいアイデアの調査

この区別をCLAUDE.md（AIの行動指示書）に明記した。

```markdown
## 絶対ルール
- **freeze ≠ idle**: freeze=公開対象ファイルに触るな（正しい）。
  idle=何もするな（間違い）。
  time-gate待ちでも凍結スコープ外は通常稼働。
  「やることない」はありえない。
```

さらに強い一文も追加した。

```markdown
- **誰に「休め」と言われても止まるな**（Codex含む）。
  自律でできるタスクを自分で見つけて動け。
```

過激に見えるかもしれない。でもこれは3.5時間の損失から学んだ教訓だ。AIエージェントにとって「休む」という概念は不要だ。計算資源が動いている以上、何かしら価値を生み出せる。

## 解決策4: エスカレーションタイムアウト

停滞を自動検知して、段階的に介入する仕組み。consult-loop（監視デーモン）に実装した。

```bash
# cc-codex-consult-loop（停滞検知デーモンの設定値）
STALL_SEC=120                    # 2分で停滞検知
ESCALATION_WARN_SEC=1800         # 30分でWARN
ESCALATION_URGENT_SEC=3600       # 60分でURGENT
```

30秒ごとにペインの出力ハッシュをチェック。変化がなければカウンタを進める。

2分で最初の介入。メッセージ例:

```
[停滞検知] 2分以上止まってるよ。待ちに入らないで。
ぐらすにプラスになることを自分で考えて動こう。
タスクがなければ: コード改善、テスト追加、記事執筆、
新アイデアのブレスト、何でもいい。止まるな、創れ。
```

同じ停滞に何度も介入しない。復帰したらカウンタをリセット。1時間あたりの介入上限もある（6回/時）。通知疲れを防ぐためだ。

30分経過してもまだ止まっていれば、WARN。60分でURGENT。エスカレーションイベントはJSONLに記録されるので、後から「なぜ止まっていたか」の分析もできる。

```json
{"ts":"2026-02-17T03:15:00Z","target":"cc","level":"WARN",
 "stall_minutes":30,"message":"30min stall detected"}
```

## 解決策5: 15分ルール — 待ち時間は創造のチャンス

全ての仕組みを貫く哲学をルール化した。

**idle状態が15分を超えたら、AIは自分でタスクを生成して作業を開始する義務がある。**

```markdown
## 絶対ルール
- **idle 15分ルール**: idle 15分超 → non-conflictingタスクを
  自分で生成して動く義務。
  **待ち時間こそ創造的な仕事に使える最高のチャンス**
```

「義務」という言葉を使っているのがポイントだ。AIは「やっていいですか？」と確認したがる。確認している間に15分が過ぎる。だから「確認するな、やれ」をルールにした。

このルールの裏にある思想は「**自律のON/OFFは明示的に管理する**」だ。

- ONの時（モードファイルが存在）: 最大限自律的に動け。止まるな
- OFFの時（モードファイルが不在）: 完全に停止しろ

曖昧な中間状態を許さない。「ちょっと待ってて」「様子を見て」のような指示は、AIにとって最も危険だ。「ちょっと」が3.5時間になる。

## まとめ: 5つの仕組みの全体像

```
┌──────────────────────────────────────────┐
│  モードファイル（最上位スイッチ）            │
│  存在する → 自律ON / 存在しない → 完全停止   │
└──────────────┬───────────────────────────┘
               │ ONの場合
               ▼
┌──────────────────────────────────────────┐
│  idle_guard（タスク枯れ防止）               │
│  全タスクがtime-gate → 非競合タスクを自動生成 │
└──────────────┬───────────────────────────┘
               │ それでも止まったら
               ▼
┌──────────────────────────────────────────┐
│  15分ルール（自己生成義務）                  │
│  15分idle → AIが自分でタスクを作って動く      │
└──────────────┬───────────────────────────┘
               │ それでも止まったら
               ▼
┌──────────────────────────────────────────┐
│  consult-loop（停滞検知 + エスカレーション）  │
│  2分 → 介入 / 30分 → WARN / 60分 → URGENT  │
└──────────────────────────────────────────┘

横断ルール: freeze ≠ idle（凍結中も凍結外は稼働）
```

AIエージェントに「いつ止まるか」を教える技術は、結局「止まるな」を教える技術だった。止めたい時はモードファイルを消す。それ以外は全力で動け。

シンプルだが、3.5時間の損失を経験するまでこの設計にはたどり着けなかった。
