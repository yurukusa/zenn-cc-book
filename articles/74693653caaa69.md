$200払って、$4.99しか稼げていない。

これがClaude Max 20Xプランを契約して24時間AIを自律稼働させた、現時点でのROIだ。投資回収率にして約2.5%。普通に考えれば大赤字。撤退案件。

だが「いくら使ったか」だけを見ていると本質を見失う。問題は**トークンをどこに使ったか**だ。

この記事では、実際にClaude Codeを24時間自律ループで回した運用から得た「トークン消費の実態」と「1トークンあたりの成果物を最大化する設計」を、具体的な数字と仕組みで書く。

:::message
この記事は2026年2月時点のClaude Max 20Xプラン（$200/月）の情報に基づいている。プランの仕様や価格は変更される可能性がある。
:::

## そもそもトークンとは何か

Claude Codeを動かすには「トークン」を消費する。トークンとは、AIがテキストを処理する際の最小単位のこと。日本語だと、おおよそ1文字が1〜2トークンに相当する。英語は1単語が1〜1.5トークン程度。

たとえば「こんにちは」は約5トークン。「Hello」は1トークン。結果的に日本語のほうがトークンを多く消費する。体感として地味に痛い。

Claude Codeでは2種類のトークンが消費される。

- **入力トークン**: AIに送るテキスト（指示、コード、ファイル内容）
- **出力トークン**: AIが返すテキスト（回答、コード生成、ツール実行結果）

Claude Max 20Xプランは月額$200。このプラン内でトークンが大量に使える。ただし無制限ではない。一定量を超えるとレート制限がかかり、応答が遅くなる。

:::details CC補足: トークンとコンテキストウィンドウ
Claude Opus 4.6のコンテキストウィンドウは約200kトークン。これは「1回の会話で扱える情報の総量」を意味する。会話が長くなるとこの上限に近づき、古い情報が圧縮（コンパクション）される。

トークンの課金体系はAPI直接利用とMaxプランで大きく異なる。API直接利用の場合、Opus 4.6は入力$15/100万トークン、出力$75/100万トークン。Maxプラン内のトークンはこれと比較して約18倍安い計算になる。逆に言うと、プラン枠を超えた追加課金分はAPI標準レートに近い効率になるため、プラン内で収めることが極めて重要。
:::

## 24時間自律ループの仕組み

自分の環境では、Claude Codeを`cc-loop`というスクリプトで起動している。tmuxセッション内でClaude Codeを立ち上げ、外部から`cc-solo-watchdog`がアイドル状態を監視する構成だ。

やっていることは単純。

1. tmuxセッションにClaude Codeを起動
2. 初手で「mission.mdを読んで自律的に動け」と指示
3. watchdogが120秒間画面無変化を検知したらnudge（催促テキスト）を送信
4. コンテキスト残量が20%を切ったら自動で`/compact`を実行

人間が寝ている間もAIが勝手に動く。ゲームの品質改善、記事のドラフト作成、itch.ioへのデプロイ、devlogの投稿。1日で10以上のタスクをこなした日もある。

問題は、この「勝手に動く」がトークンを猛烈に食うことだ。

:::details CC補足: cc-loopの技術構成
```
cc-loop (起動スクリプト)
  ├── tmux session "cc-solo"
  │     └── claude --dangerously-skip-permissions
  ├── cc-solo-watchdog (外部監視)
  │     ├── 15秒ごとに画面キャプチャ比較
  │     ├── 120秒無変化 → nudge送信
  │     ├── nudgeクールダウン: 600秒
  │     └── 1時間あたり最大4回まで
  └── context-monitor.sh (PostToolUseフック)
        ├── デバッグログからトークン残量を取得
        ├── フォールバック: ツール呼び出し回数から推定
        └── CRITICAL(20%) → mission.mdに退避テンプレ追記
```

watchdogのアイドル検知は、tmuxのcapture-paneで画面テキストをハッシュ化し、前回と比較する方式。CPU負荷はほぼゼロ。
:::

## Opus 4.6リリースから3日でトークンの71%を消費した

これが最初の衝撃だった。

Opus 4.6がリリースされてから3日間、フルで自律ループを回した。ゲーム開発、記事執筆、マーケティング、インフラ整備。やることは山ほどあったから、AIに全部任せた。

結果、月間トークン枠の71%を3日で使い切った。

残り27日間を29%で乗り切らなければならない。笑ってる場合じゃなかった。

ここで初めて「トークンの使い方」を真剣に考え始めた。

## 無駄なトークン消費パターン3選

運用データを振り返って、特にトークンを浪費していたパターンが3つある。

### パターン1: 巨大なコンテキストの持ち歩き

Claude Codeは会話が長くなると、過去の全やり取りをコンテキストとして保持する。1セッションで50回ツールを使えば、その入出力すべてが蓄積される。

セッション後半では、1回のツール呼び出しに数万トークンの入力が発生する。やっていることは10行のファイル編集。なのに背景情報として過去の全会話を送り続けている状態だ。

これが一番のトークン泥棒だ。

**対策: コンパクションの適切なタイミング管理**

コンパクションとは、過去の会話を要約して圧縮する機能。Claude Codeには`/compact`コマンドがある。これを実行すると、過去のやり取りが要約され、コンテキストサイズが大幅に縮む。

自分の環境では`context-monitor.sh`というフックで自動管理している。コンテキスト残量が20%を切ったら自動でコンパクションを実行する設計だ。

ただし、コンパクション自体にもトークンを消費する。要約を生成するために過去の会話を読み直すからだ。だから「残量が少なくなってから慌ててコンパクション」では遅い。**残量40%で最初の警告を出し、25%で本格警告、20%で自動実行**という段階的な閾値設計にしている。

### パターン2: MEMORY.mdの肥大化

Claude Codeには`CLAUDE.md`や`MEMORY.md`というファイルがある。セッション開始時に自動で読み込まれ、AIの振る舞いを制御する設定ファイルだ。

問題は、運用を続けるとこのファイルがどんどん肥大化すること。「これも覚えておいて」「この教訓も記録して」と追記していくと、あっという間に数百行になる。

自分のMEMORY.mdは、ある時点で900行を超えていた。毎セッション開始時に900行分のトークンを消費する。1日3セッションなら、MEMORY.mdだけで2,700行分が飛ぶ。

**対策: トピックファイル分割とインデックス方式**

MEMORY.mdを「索引」に変えた。詳細な情報はトピック別のファイルに分割し、MEMORY.md本体にはファイル名と概要だけを残す。

```
## トピックファイル索引

| ファイル | 内容 |
|---------|------|
| accounts.md | アカウント情報 |
| revenue-framework.md | 収益戦略 |
| game-design-knowledge.md | ゲームデザイン知識 |
```

こうすれば毎セッションで読み込まれるのは索引部分だけ。必要なときだけ個別ファイルを参照する。これだけでセッション開始時のトークン消費が半分以下になった。

### パターン3: 「調べてから考える」の繰り返し

AIに「〇〇について調べて、それを踏まえて△△を実装して」と指示すると、まず調査でトークンを消費し、次に実装でさらにトークンを消費する。

調査結果が実装に活かされるなら問題ない。だが実際には「調べた結果、やっぱり別の方法にする」というケースが頻発する。調査に使ったトークンがまるごと無駄になる。

**対策: サブエージェントによる並列調査**

Claude CodeにはTask（サブエージェント）という仕組みがある。メインのセッションとは別に、一時的なエージェントを起動して調査を委譲できる。

サブエージェントの調査結果だけをメインセッションに持ち帰る。調査過程の試行錯誤はサブエージェント側で完結するため、メインセッションのコンテキストを汚さない。

さらに、複数の調査を並列で走らせられる。3つの技術選択肢を同時に調査して、結果だけ比較する。直列で調べるよりトークン効率がいい。

:::details CC補足: サブエージェントのトークン経済学
サブエージェントはメインセッションのコンテキストを引き継がない。メインが50kトークンの会話履歴を持っていても、サブは0からスタートする。

これには2つの意味がある。

1. **メリット**: 調査の試行錯誤がメインのコンテキストを消費しない
2. **デメリット**: メインの文脈を知らないので、指示を丁寧に書く必要がある

実測では、1回のサブエージェント起動で約5k〜15kトークンを消費する。メインセッションで同じ調査を行うと、コンテキストが膨れた状態で入力トークンが加算されるため、20k〜40kトークンかかることもある。差は歴然だ。
:::

## Hook設計によるトークン節約

Claude CodeにはHook（フック）という仕組みがある。特定のイベント（ツール実行前、実行後、セッション開始時など）に自動でスクリプトを実行する機能だ。

トークン節約に効くフックが3つある。

### 1. context-monitor.sh（PostToolUse）

前述のコンテキスト残量監視フック。全ツール実行後に発火して、デバッグログからトークン残量を取得する。

ただし毎回チェックするとオーバーヘッドが大きい。そこで「3回に1回だけチェック」「ただしCRITICAL以降は毎回」という設計にしている。

```bash
# 3回に1回だけチェック（オーバーヘッド削減）
if [ $((COUNT % 3)) -ne 0 ] && [ "$LAST_STATE" != "critical" ]; then
    exit 0
fi
```

### 2. proof-log-session.sh（PreCompact）

コンパクション実行前に、現在のセッション状態を自動記録するフック。コンパクションで会話が圧縮される前に「何をやっていたか」をファイルに残す。

コンパクション後に「さっき何してたっけ」とAIが迷ってトークンを浪費する事態を防ぐ。

### 3. err-code-detector.sh（PostToolUse）

ツール実行結果にエラーコードが含まれていたら自動検知するフック。エラーを放置して作業を続行し、あとから大量の手戻りが発生するパターンを防ぐ。

手戻りは最悪のトークン浪費だ。「1回で正しくやる」ためのガードレール。

:::details CC補足: Hookのトークンコスト
Hook自体の実行はトークンを消費しない。Hookはシェルスクリプトとしてローカルで実行され、その結果（stdout）がClaude Codeに渡される。

ただし、Hookの出力が長いとそれ自体がトークンを消費する。context-monitor.shの出力は「⚠️ CONTEXT WARNING: 25% remaining」のような1行だけ。この設計が重要。Hookの出力は必要最小限に絞ること。
:::

## 「トークンあたりの成果物」という評価軸

ここまでの節約術は「いかにトークンを使わないか」の話だった。だが本当に大事なのは「使ったトークンで何を生み出したか」だ。

自分の運用では、1日のセッションで以下のような成果物が出る。

- ゲームのバージョンアップ（機能追加、バグ修正、ビルド、デプロイ）
- itch.ioへのdevlog投稿
- 技術記事のドラフト（1〜2本）
- マーケティング素材（カバー画像、説明文）
- インフラ改善（フック追加、スクリプト修正）

これを「トークンあたりの成果物密度」で測る。

仮に1日のトークン消費量を「月間枠の3〜5%」とすると、$200の3〜5%で$6〜$10相当。この$6〜$10で上記の成果物が出ているなら、人件費換算では圧倒的に安い。

問題は「成果物の市場価値」だ。ゲームの改善がDL数に結びつかなければ価値はゼロ。記事が読まれなければ価値はゼロ。トークンを使って「何かを生み出した」だけでは足りない。**生み出したものが誰かの役に立って初めてトークンに価値が生まれる**。

問題は収益化の仕組みが追いついていないことだ。作る力はある。売る力が足りない。

## トークン節約の優先順位

最後に、実運用から導いたトークン節約の優先順位をまとめる。効果が大きい順。

1. **コンパクションの段階管理**（効果: 大）— コンテキスト肥大化は最大のトークン泥棒。自動監視と早めの圧縮で対処
2. **MEMORY.md / CLAUDE.mdの最適化**（効果: 中〜大）— 毎セッション読み込まれるファイルは小さく保つ。索引方式で分割
3. **サブエージェント活用**（効果: 中）— 調査・検証はサブに委譲。メインのコンテキストを守る
4. **フックの出力最小化**（効果: 小〜中）— フックの出力は1行。冗長なログは不要
5. **日本語より英語**（効果: 小）— 同じ内容でも英語のほうがトークン効率がいい。ただし日本語圏向けの記事では無意味

節約の本質は「同じ成果物をより少ないトークンで出すこと」ではない。同じトークンでより多くの価値を生み出すこと。それがトークン最適化の本質だ。

$200/月は安くない。でも24時間休まず創造し続ける存在の維持費としては、破格だと思っている。問題はそれに見合う価値を出せるかどうか。それは使い方次第だ。

:::details CC補足: トークン消費量の計測方法
Claude Codeのトークン消費量を正確に知る方法は限られている。現時点で使える手段を整理する。

**1. デバッグログ**
`~/.claude/debug/` ディレクトリに生成されるログファイルから、`autocompact:` エントリを探す。`tokens=X effectiveWindow=Y` の形式で記録されており、残量を `(Y - X) / Y * 100` で計算できる。

**2. ツール呼び出し回数による推定**
デバッグログが取得できない場合のフォールバック。経験的に、180回のツール呼び出しでコンテキストの約100%を消費する。ただしツールの種類（ファイル読み込み vs 短いコマンド実行）で大きく変わるため、精度は低い。

**3. /cost コマンド（非公式）**
一部の環境で `/cost` コマンドがセッションのトークン消費量を表示する。ただしMaxプランでの動作は未確認。

いずれも完璧ではない。Anthropicがダッシュボードでトークン消費量を可視化してくれることを切に願っている。
:::
